# SDM System - AI 开发助手

## 项目概述

SDM System 是一个包含后端、Web 前端、Flutter、小程序的管理系统。

```
/
├── backend/           # Python FastAPI 后端
├── frontend/          # Next.js Web 管理端
├── event_app/         # Flutter 端
├── miniprogram/       # 微信小程序
└── docs/              # 工程文档
```

## 子项目入口

进入各子目录时，AI 会自动加载对应的 CLAUDE.md：

- `backend/app/` → 后端专属规则（模型分层、敏感数据、API 准则）
- `frontend/` → 前端专属规则（组件规范、状态管理）
- `miniprogram/` → 小程序专属规则（禁止直接操作全局 Data）
- `event_app/` → Flutter 专属规则
- `docs/` → 文档目录规范与架构总览

## 全局开发准则

### 提交规范

提交信息必须包含 `[Backend]/[Web]/[Flutter]/[Mini]` 前缀。

### 任务执行

- 任务执行过程不必确认，每完成一阶段任务自动 commit 并继续下一个任务
- 完成重大调整和改动后，必须同步更新 `docs/产品规格说明-spec.md` 的实现状态表格

### SQL 变更

- 建表语句 → `backend/table.sql`
- 字段修改 → `backend/field.sql`

### 文档同步规则

修改代码后必须同步更新文档：

| 修改内容 | 必须更新的文档 |
|----------|----------------|
| 新增 API 路由 | `docs/handover/` 相关功能文档 |
| 修改数据模型 | `docs/handover/` 相关功能文档 |
| 新增大功能 | 同时创建 `docs/handover/` + `docs/insights/` 双向链接文档 |

### 权限模块强制前置阅读（必须）

当任务涉及权限、角色、活动可见性、社区访问、支付准入、混合账号（用户+管理员）时，必须按以下顺序执行：

1. 先阅读 `docs/权限系统架构文档.md`（As-Is 现状架构）
2. 再阅读 `docs/权限系统设计原则.md`（红线、边界、回归清单）
3. 再开始改代码

硬性约束：

- 未阅读以上两个文档，禁止修改权限相关代码。
- 权限相关改动提交前，必须对照 `docs/权限系统设计原则.md` 的“强制检查清单”逐条核对。

### 错误处理

- 函数错误：使用 `throw new Error()` 抛出明确错误信息
- 异步操作：使用 `try-catch` 包裹，finally 清理资源
- 静默失败禁令：任何异步调用必须有 try-catch 和用户友好的提示

### 代码风格

- **单一职责**：每个函数、组件只做一件事
- **DRY**：重复代码抽取为函数或常量
- **KISS**：优先简单方案，避免过度设计
- **方法行数上限**：单个方法不超过 500 行

### 注释

- 使用中文注释和中文 UI
- 仅在逻辑复杂或不明显处添加注释
- 工具函数可添加 JSDoc 说明参数和返回值
