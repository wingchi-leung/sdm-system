# SDM 系统 — 工程现状与功能说明

本文档描述当前整个工程的架构、技术栈以及已实现的功能。

---

## 一、工程概览

SDM 系统是一套**活动报名与签到**解决方案，由三部分组成：

| 模块 | 技术栈 | 角色 |
|------|--------|------|
| **backend** | FastAPI + MySQL + SQLAlchemy | 统一 API 服务 |
| **frontend** | React (Create React App) | 管理端 Web：签到、创建活动、用户管理、统计 |
| **event_app** | Flutter | C 端 App：活动列表、详情、报名（Web / iOS / Android） |
| **miniprogram** | 微信小程序（原生） | C 端小程序：活动列表、详情、报名、登录/注册、我的、管理员发布活动 |

- **管理端**（frontend）：管理员登录后发布活动、现场签到、查看统计、管理用户。
- **C 端**（event_app）：普通用户浏览活动、查看详情、填写报名信息；管理员还可在 App 内发布活动。

---

## 二、后端（backend）

### 2.1 技术栈

- **Python 3.10+**，**FastAPI**，**uvicorn**
- **MySQL 8**，**SQLAlchemy 2.x**，**PyMySQL**（含 `cryptography` 以支持 MySQL 8 认证）
- **JWT**（python-jose）、**bcrypt**（passlib）：管理员鉴权
- 依赖管理：**uv**（推荐）/ pip，以 `pyproject.toml` 为准

### 2.2 数据模型（核心表）

| 表名 | 说明 |
|------|------|
| **activity** | 活动：名称、开始/结束时间、状态（1 未开始 / 2 进行中 / 3 已结束）、标签等 |
| **activity_participants** | 活动参与人：关联活动、姓名、手机、证件号，可选关联 user_id |
| **checkin_records** | 签到记录：活动、姓名、证件号、手机、签到时间、是否参与、备注等 |
| **user** | 用户档案：姓名、证件号、手机（唯一索引）、password_hash、性别；拉黑字段 isblock、block_reason；支持注册登录 |
| **admin_user** | 管理员：username、password_hash，用于登录鉴权；可选 user_id 关联 user |

### 2.3 API 路由（/api/v1）

| 模块 | 路径前缀 | 主要接口与说明 |
|------|----------|----------------|
| **auth** | `/auth` | `POST /login`：管理员登录，返回 JWT；`POST /user-login`：普通用户登录（手机+密码），返回 JWT（含 user_id、user_name） |
| **users** | `/users` | `POST /register`：用户注册（姓名+手机+密码）；`GET /me`：获取当前用户信息（需 user token）；`GET /` 用户列表；`GET /{user_id}` 用户详情；`POST /create` 创建用户 |
| **activities** | `/activities` | `GET /` 活动列表（支持 status）；`GET /unstarted/` 未开始活动；`POST /` 创建活动（**需管理员 token**）；`PUT /{id}/status` 更新状态；`GET /{id}/checkins/` 某活动签到列表；`GET /{id}/statistics/` 某活动统计 |
| **participants** | `/participants` | `POST /` 报名（C 端提交，含黑名单拦截）；`GET /{activity_id}/` 某活动参与人列表（分页） |
| **checkins** | `/checkins` | `POST /` 签到（校验已报名、防重复签到） |

### 2.4 已实现功能摘要

- **双角色 JWT 鉴权**：管理员（admin_user 表）和普通用户（user 表）分别登录，JWT 中包含 role 字段区分
- **普通用户注册/登录**：手机号 + 密码注册，手机号 + 密码登录；登录时检查黑名单状态
- **统一鉴权中间件**：`get_current_admin`（仅管理员）、`get_current_user`（任意角色）、`get_current_user_optional`（可选登录）
- 活动 CRUD、状态流转、未开始活动查询
- 活动参与人报名（含重复证件号校验 + **黑名单拦截**：按手机号查用户 isblock）
- 签到：校验是否已报名、是否已签到，写入 `checkin_records`
- 活动维度：签到列表、统计数据
- 用户管理：列表、创建、个人信息查询（`GET /users/me`）
- 环境配置：`.env` 配置 MySQL、JWT_SECRET 等；生产需改 JWT_SECRET

详细鉴权与首次建表、插入管理员步骤见：`backend/docs/鉴权说明.md`。

## 三、管理端前端（frontend）

### 3.1 技术栈

- **React**（Create React App）
- **React Router**、**lucide-react**、自定义 UI 组件（button、card、dialog、table、toast 等）
- 通过 `src/config/api.ts` 配置 `REACT_APP_API_URL`，默认 `http://localhost:8000/api/v1`

### 3.2 路由与页面

| 路径 | 页面/组件 | 功能简述 |
|------|------------|----------|
| `/` | 重定向到 `/signin` | — |
| `/signin` | SignInPage | 活动签到：选择活动、参与人列表后，支持手动签到 / 列表签到 / 签名模式（签名保存逻辑部分注释） |
| `/statistics` | Statistics → RecordsDisplay | 签到数据统计：展示签到记录（当前请求 `http://localhost:8000/records`，若后端未实现该接口则需对接活动/签到 API） |
| `/createactivity` | CreateActivity | 创建活动：活动名、时间、标签；参与人可手动添加或从「用户管理」导入；提交调用创建活动 API（需带管理员 token） |
| `/users` | UserManagement | 用户管理：用户列表、新增/编辑用户（姓名、证件号、电话、性别等） |

### 3.3 已实现功能摘要

- 统一布局：侧边栏（活动签到、日志统计、创建活动、用户管理）
- 活动签到：选活动、拉参与人、多种签到方式（含签名组件）
- 创建活动：表单 + 从用户库勾选导入参与人
- 用户管理：基于 user 表的增删改查
- 统计页：签到记录展示（依赖后端 `/records` 或需改为活动/签到相关 API）

**说明**：创建活动、需鉴权的接口需在请求头携带 `Authorization: Bearer <token>`；若当前前端未全局接入登录与 token，需在调用处补充。

## 四、C 端 App（event_app）

### 4.1 技术栈

- **Flutter**，支持 **Web、iOS、Android**
- 通过 `lib/services/api_service.dart` 配置 `baseUrl`（如本机 `http://localhost:8000/api/v1`，真机需改为电脑局域网 IP）

### 4.2 页面与流程

| 页面/文件 | 功能 |
|-----------|------|
| **MainShell** | 底部导航：「活动」和「我的」两个 Tab |
| **ActivityTabScreen** | 活动 Tab：展示可报名活动（未开始 + 进行中），下拉刷新 |
| **ActivityListScreen** | 全部活动列表（管理员从「我的」进入），支持发布/刷新/登录 |
| **ActivityDetailScreen** | 活动详情：状态、时间、标签；未结束活动可进入报名 |
| **RegisterScreen** | 活动报名表单：姓名、手机、证件号（选填），自动携带登录 token |
| **CreateActivityScreen** | 管理员在 App 内发布活动（需管理员 token） |
| **LoginScreen** | 统一登录页：默认普通用户（手机+密码），可切换管理员（用户名+密码）；可跳转注册 |
| **UserRegisterScreen** | 用户注册：姓名、手机、密码必填，邮箱选填；注册成功自动登录 |

### 4.3 已实现功能摘要

- 活动列表、详情、报名全流程
- **双角色登录**：普通用户（手机+密码）和管理员（用户名+密码）统一在同一登录页切换
- **用户注册**：注册成功后自动登录
- **AuthService 统一状态管理**：持久化 token、role、userId、userName；支持 isLoggedIn / isAdmin / isUser
- **「我的」页面三态显示**：未登录（引导登录/注册）→ 普通用户（个人信息展示）→ 管理员（发布/查看活动）
- 管理员在 App 内发布活动
- 详见 `event_app/README.md`

## 五、C 端小程序（miniprogram）

### 5.1 技术栈

- **微信小程序**（原生 WXML/WXSS/JS）
- 通过 `utils/api.js` 配置 `baseUrl`，与 backend `/api/v1` 对接；真机需配置 request 合法域名

### 5.2 页面与流程

| 页面 | 功能 |
|------|------|
| **index**（Tab） | 活动列表：可报名活动，下拉刷新；管理员可点击「发布活动」 |
| **activity-detail** | 活动详情：状态、时间、标签；未结束可进入报名 |
| **register** | 活动报名：姓名、手机、证件号（选填），可带登录 token |
| **login** | 统一登录：普通用户（手机+密码）/ 管理员（用户名+密码），可跳转注册 |
| **user-register** | 用户注册：姓名、手机、密码必填；注册成功自动登录 |
| **mine**（Tab） | 我的：未登录引导登录；普通用户展示个人信息；管理员可发布活动、查看活动、退出 |
| **create-activity** | 管理员发布活动：活动名、标签、开始时间 |

### 5.3 已实现功能摘要

- 与 Flutter event_app 对齐：活动列表/详情/报名、双角色登录、用户注册、我的三态、管理员发布活动
- 鉴权与 API 与后端一致：JWT 存本地，报名/创建活动等接口按需带 `Authorization`
- 详见 `miniprogram/README.md`

## 七、建议的后续改进（可选）

1. **frontend 鉴权**：若创建活动等接口需管理员 token，建议增加登录页并在 api 层统一附带 `Authorization`。
2. **统计页接口**：将 RecordsDisplay 改为调用后端已有接口（如按活动查签到 `GET /activities/{id}/checkins/` 或统一签到列表），或在后端实现 `/records` 并保持前端不变。
3. **签名签到**：SignInPage 中签名保存逻辑若需落库，可在后端增加签名存储字段或接口，并与前端解注释代码对齐。
4. **生产部署**：后端设置强 JWT_SECRET、配置 CORS 与 HTTPS；前端/App 配置生产 API 地址。
