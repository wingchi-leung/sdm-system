# 权限系统设计原则（AI 执行版）

> 本文档是“改权限代码的硬约束”。  
> 先读 `权限系统架构文档.md`（As-Is），再按本文执行。

---

## 1. 不可违反的红线（违反即高风险）

1. **禁止用 `role_id` 推断全局权限**  
只能用 `UserRole.scope_type is NULL` 判定全局。

2. **禁止在用户能力接口里写 `if ctx.role == "admin": reject`**  
报名、支付、订单、评论必须兼容混合账号（管理员+用户）。

3. **禁止前端覆盖后端权限结果**  
前端可以隐藏按钮，不能再做一套“二次可见性过滤”去删后端已返回数据。

4. **禁止把加密属性当 SQL 列直接 `label/group_by`**  
例如 `User.name` 是 property，不能 `User.name.label(...)`。应查询 `User` 实体后在 Python 层取 `user.name`。

5. **禁止在未定义回归清单前合并权限改动**  
权限改动必须附带测试或最小复现验证结果。

---

## 2. 判定顺序（必须按顺序思考）

### 2.1 身份层（Authentication）

1. 是否登录（`user_id/tenant_id` 是否有效）
2. 是否有管理员能力（`ctx.has_any_role(db)`）
3. 是否需要用户视角（如 `as_user_view=1`）

### 2.2 授权层（Authorization）

1. 管理能力判定：RBAC + scope_type
2. 用户能力判定：是否已报名 / 是否在 user_activity_types / 是否公开活动
3. 混合账号：管理能力和用户能力并存，不互斥

### 2.3 输出层（Response）

1. 后端给出最终可见/可操作结果
2. 前端只展示，不改写权限结论

---

## 3. 关键边界案例（每次改动都要想到）

1. **账号同时是活动管理员 + 报名用户**  
应能：看活动、报名/支付、看订单、评论、做管理动作。

2. **管理员用户切到用户视角（as_user_view）**  
活动列表应按用户分配过滤，不按管理员全局放开。

3. **用户刚被授权/收权**  
下一次请求立即生效（权限来自 DB 实时查询）。

4. **非公开活动**  
未报名且无管理权限不可见；已报名或有活动级管理权限可见。

5. **Web 与小程序接口隔离**  
Web 明文用户列表必须走 `/users/admin/all-web`，不能改坏小程序兼容链路。

---

## 4. 你必须避免的历史回归（真实出过）

1. 接口返回结构变更但前端仍按旧结构解析，导致“分配成功但不生效”。
2. 小程序活动请求漏带 token，后端按游客处理，只显示公开活动。
3. 前端在列表页二次过滤活动，把后端已放行数据误删。
4. 管理员登录清空 user_id/user_name，导致“我的活动/订单”不可用。
5. 后端用 `ctx.role == "admin"` 直接禁掉支付/评论。
6. 社区查询中把 `User.name` 当 SQL 列查询，运行时报错。

## 5. AI 修改权限代码的强制检查清单

### 5.1 代码检查

- [ ] 没有新增 `ctx.role == "admin"` 的硬拦截用户能力代码
- [ ] `scope_type` 判定仍是唯一范围来源
- [ ] 没有在前端新增二次权限过滤覆盖后端结果
- [ ] 任何涉及 `User.name/phone` 查询都未直接 SQL `label/group_by` property

### 5.2 行为检查

- [ ] 混合账号可看活动列表（含非公开且已分配类型）
- [ ] 混合账号可报名/支付并生成订单
- [ ] 混合账号可查看我的活动/我的订单
- [ ] 混合账号在已报名活动可评论社区
- [ ] 无权限且未报名账号访问社区被正确拒绝

### 5.3 回归检查（至少一条后端测试 + 一条端侧验证）

- [ ] 后端新增/更新测试通过
- [ ] 小程序关键路径手动验证至少 1 轮（登录 → 活动 → 报名/订单/社区）
