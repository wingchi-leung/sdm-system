## 项目背景

用户对小程序社区模块进行 UI 大改,目标:**视觉 + 交互 + 新功能**(最大力度)。

---

## 用户已明确的功能需求 (P0)

### 需求 1: 瀑布流信息流 + 原位展开

**当前痛点**: 帖子流是常规单列列表,内容/评论查看都要跳页 `community-post-detail`,体验割裂、回流成本高。

**用户要求**:
- 社区内容用 **瀑布流** 形式展现(两列错落卡片,卡片高度按内容驱动)
- 内容过长时加「**查看全部**」按钮 → **不跳详情页**,在原位就地展开/折叠
- 评论也 **直接展示在社区页面**(原位展开评论列表,而不是跳到 detail 页)

**讨论点 / 需要决策**:
- `community-post-detail` 详情页是否保留?
  - 倾向:保留作为「分享链接 / 系统消息深链」的落点,但主流交互不再依赖它
  - 架构师给最终建议
- "瀑布流" 指视觉布局(两列错落)还是仅指"无限滚动"?
  - 优先按 **两列错落瀑布流** 规划;前端如有性能/小程序框架限制再退到单列长流
- 评论原位展开的交互细节:
  - 点击「评论」按钮就地展开?展开后再分页加载?
  - 评论输入框跟随展开 or 底部固定?
  - 前端 + 设计师对齐方案

---

### 需求 2: 富文本编辑器 + 多选图片

**当前痛点**: 发布页是"图片块 / 文字块"分离的块编辑器(参见 `miniprogram/pages/community-post-create`),反复切换块体验差;一篇图文要先加文字、再加图片、再加文字,操作繁琐。

**用户要求**:
- 一个**富文本编辑器**搞定所有图文混排(类似 Word/印象笔记/微信公众号编辑器的体验)
- 支持**多选图片上传**(一次性选多张图片,自动插入编辑器)

**讨论点 / 需要决策**:
- 小程序富文本编辑器选型:
  - 选项 A: 微信原生 `<editor>` 组件(原生支持图文混排,但定制能力一般)
  - 选项 B: 自研(textarea + 内嵌图片标记,定制度高但工作量大)
  - 选项 C: 第三方组件
  - **前端 agent 给推荐 + 对比表**
- 内容存储格式(影响后端 schema 与前后端解析):
  - HTML / Markdown / 自定义 JSON blocks
  - **前端 + 后端对齐**
- 后端 `content` 字段现状(text/longtext)是否够用?是否需要改为结构化字段?
  - 后端 agent 评估
- 多图选择上传:
  - `wx.chooseMedia({ count: 9, mediaType: ['image'] })` → 顺次走现有 `uploads` endpoint
  - 上传中的占位、失败重试、并行/串行策略
  - 前端 agent 给方案

---

### 需求 3: 创建频道支持上传头像

**当前痛点**: 当前创建频道用 `wx.showModal({ editable: true })` 只支持输入名称(`miniprogram/pages/community/index.js:62-89`),后端虽然支持 `avatar_url` 和 `description` 字段(`backend/app/models/community.py:135-164`),但前端完全没用上。

**用户要求**:
- 创建频道时**可以上传图片作为频道头像**
- **功能先加,UI 等设计稿**(本阶段做技术骨架)

**实现要点**:
- 把当前 `wx.showModal` 改造为**独立页面**(推荐 `pages/community-channel-create/community-channel-create`)
  - 字段:名称(必填) + 描述(选填) + 头像(选填,推荐填)
- 复用现有 `uploads` endpoint 上传头像图片
- 提交时把上传后的 URL 写入 `CommunityChannelCreate.avatar_url`
- 后端无需改动(字段已存在)

## 重要原则

### 1. 以源码为准

摸底报告(`社区UI改造-00-现状摸底.md` §5 第 6 条)指出 spec 与落地存在差异:

- spec 写 `community_channel_post` / `community_channel_comment` / `community_notification` 三张表
- 落地复用了 `community_post` / `community_comment`,且 `community_post` **缺少 `channel_id` 字段**

**处理原则**:
- **按现有源码实现的方向走** —— `community_post` 这张老表就是"频道动态"的载体
- 但 **缺失的 `channel_id` 字段建议补上**(架构师 + 后端评估),否则 spec 与代码语义会继续漂移
- 后续 spec 文档要被更新以匹配源码,**而不是反过来**

### 2. 不依赖 spec 里的未实现接口

spec §3 提到但代码未实现的接口(`PUT/DELETE /channels/{id}`、置顶、官方标签、转让管理员等)**默认不纳入本期**,除非架构师明确论证需要。

### 3. 数据库变更规则(CLAUDE.md 硬性要求)

- 新建表 → `backend/sql/table.sql`(append)
- 字段修改 → `backend/sql/field.sql`(用 ALTER)
- **禁止**通过 `backend/app/db_migrations.py` 补新增字段
- 已经 `CREATE TABLE` 的表不要改原 DDL,只做 ALTER

### 4. 权限模块改动前置阅读(CLAUDE.md 硬性要求)

如本次改动涉及:权限、角色、频道可见性、社区访问、混合账号 —— 必须先读:
- `docs/handover/权限系统架构文档.md`
- `docs/handover/权限系统设计原则.md`
