## 2. 用户需求 1 — 瀑布流 + 原位展开(本次重点)

### 2.1 方案对比

#### A. 单 `scroll-view` + 2 列 `flex` 容器(前端拆列)

- **实现**:
  - `scroll-view` 内放一个 `column-flex` 容器(2 列 `flex: 1`)。
  - 数据层维护 `leftColumn` / `rightColumn` 两个数组,新数据到来时**比较两列当前累计高度**(`heights[i] += cardHeight[i]`),矮列优先 push。
  - 卡片高度**预估**:`估算高度 = 标题行数(按 `length/14`)+ 摘要截断行(2 行) + 图片比例 + meta + padding`;真实高度在 `bindload` 后回填。
- **优点**: 微信原生、零依赖、滚动连贯性好;`column-count` 兼容。
- **缺点**:
  - 高度预估有误差,首屏会"抖"一帧(可加 `.skeleton` 兜底)。
  - 短列优先 → 列长差异在 1 列宽内,视觉"错落感"略弱;设计师若要更错落可改为"加权随机"。
  - 评论原位展开时,卡片高度变化只影响**本列下方**,但因为是 **setData 重排** 整列,会出现整列位移 → 必须用**懒挂载 + transform 过渡**。
- **结论**: **本期推荐**。

#### B. 第三方 / 自研 layout 算法

- **实现**:
  - 第三方如 `miniprogram-waterfall`(社区方案,2019 后基本不维护)、`@vant/weapp` 无原生瀑布流组件。
  - 自研:在 A 基础上加"按图片宽高比精确分列"。
- **优点**: 错落感更强。
- **缺点**: 第三方维护差、风险高;自研 1 人/天起,且与本项目"无第三方依赖"基调冲突。
- **结论**: **不推荐**(本期不接受新依赖,设计师需在设计稿中说明是否真的需要 2 列错落)。

### 2.2 瀑布流数据拆分算法(推荐 A 的细化)

```
input: items[] (按时间倒序)
output: leftColumn[], rightColumn[]

heights = [0, 0]                // 估算两列累计高度
for item in items:
  // 估算高度
  titleLines = ceil(item.title.length / 14)
  summaryLines = min(3, ceil((item.content_summary || '').length / 14))
  hasImage = item.images && item.images[0]   // 仅展示第一张封面
  imageHeight = hasImage ? cardWidth * (imageAspect || 0.66) : 0
  metaHeight = 70
  padding = 48
  est = titleLines * 44 + summaryLines * 36 + imageHeight + metaHeight + padding

  // 矮列优先;并列等高时,交错推入保证视觉均衡
  if heights[0] <= heights[1]:
    leftColumn.push(item)
    heights[0] += est
  else:
    rightColumn.push(item)
    heights[1] += est
```

- 性能:
  - 100 条以内纯 JS 计算 < 1ms,无压力。
  - 真实场景拉取一次 20 条,后续滚动追加再算。
- 图片高度校正: `image` `bindload` 时 `e.detail.width/height` → 计算真实 aspect → 写回 `data.imageAspect` → 用 `mode="widthFix"` 渲染,**避免 setData 触发整列 re-layout**。

### 2.3 "查看全部" 原位展开

#### 2.3.1 DOM 挂载策略

| 策略 | 描述 | 适用 |
|------|------|------|
| **懒挂载**(推荐) | 卡片始终只渲染**摘要态**;`expanded` 标记位为 true 时**才把全文 blocks 渲染到 DOM** | 99% 场景 |
| 全量挂载 | 一开始就渲染全文,纯靠 `wx:if` 切换 CSS `max-height` 过渡 | 短文 + 性能无关紧要的 demo |

**推荐懒挂载 + 高度动画**:

- `card.expanded: boolean` 标记;
- 卡片结构:
  ```
  <view class="masonry-card" style="height: {{card.displayHeight}}rpx;">
    <view class="masonry-card-summary">…</view>
    <view wx:if="{{card.expanded}}" class="masonry-card-full">…</view>
  </view>
  ```
- 展开时:先 `setData({ expanded: true })` → 用 `wx.createSelectorQuery()` 测真实高度 → 同步写回 `displayHeight`(过渡用 `transition: height 240ms ease`);
- 收起时:先记下当前 height,直接 `setData({ expanded: false })` → 同时把 height 切回摘要态 height(同 transition 240ms)。
- **避免抖动关键**: 同一列相邻卡片只受**自己**高度变化影响,不要在 `setData` 里对**整列**做 transform 调整 → 高度过渡与"未展开卡片"互不干扰。

#### 2.3.2 高度动画细节

- 用 `transition: height 240ms cubic-bezier(0.2, 0.8, 0.2, 1)` 统一节奏。
- 全文 blocks 解析沿用现有 `utils/community-content.js::parsePostContent`,渲染复用 `image-grid` + `<text>`。
- 折叠判断阈值:**全文高度 - 摘要高度 ≤ 80rpx** 时不显示"查看全部"按钮(直接全文)。

### 2.4 评论原位展开

#### 2.4.1 三种交互对比

| 方案 | 优点 | 缺点 | 推荐 |
|------|------|------|------|
| **A. 卡片下方原位插入**(本方案) | 一次点击、上下文不丢、回流成本 0 | 长评论列表会撑高卡片,可能影响瀑布流错落 | **推荐** |
| B. Modal/半屏 picker | 不影响主列表 | 上下文断裂,要二次返回 | 不推荐 |
| C. 跳 detail 页 | 沿用现有 | 与需求 1 主线相悖 | 弃用 |

**推荐 A**。

#### 2.4.2 评论分页 + 输入框位置

- **列表**: 默认展开**前 3 条** + "查看更多评论(N)";点击再拉取 `skip=3, limit=20` 追加。
- **输入框**: **跟随卡片**(展开评论时,卡片底部追加一个 `comment-composer`)。
  - 优点: 评论上下文在卡片内自洽,无全局浮动;若卡片收起则 `comment-composer` 随卡片消失,避免 "评论了别处帖子" 的混淆。
  - 缺点: 同一时刻只允许一个卡片展开评论(`data.activeCommentPostId`),与"单卡片"互斥。
- 输入框形态: 跟现有 `community-post-detail.wxml:61-87` 的 `comment-bar` 一致(图片多选 + textarea + 发送按钮),只是从 fixed 改成 block,跟随在评论列表下方。

#### 2.4.3 性能 & 体验

- 评论懒加载: 用 `IntersectionObserver`(基础库 2.x)监听"查看更多评论"按钮的曝光,曝光后预加载下一页。
- 乐观更新: 评论提交成功后**立刻**插入到本地列表(预占位),失败再回滚 + toast。
- 评论图片预览沿用 `wx.previewImage`。

### 2.5 `community-post-detail` 详情页处置

**结论(根据架构师 agent 01-架构蓝图预期方向)**: 详情页**保留为深链落点**,但交互降级。

- **保留原因**:
  - 系统消息/分享卡片/扫码/外部链接 → 必须有一个"无上下文"的详情页可去(`/pages/community-post-detail?channelId=&id=`)。
  - 站内信 `onOpenNotification`(`community-notifications.js:101-106`)目前是跳 `community-post-list`,新方案里若有"深链到某条动态"仍要能落到 detail。
- **改造方向**:
  - 顶部加"返回频道流"按钮(显式返回),而不是"返回上一页"。
  - 移除独立的"评论区"组件,改用**与瀑布流内一致的 `comment-composer` + 评论列表**,**视觉/逻辑同源**。
  - 数据加载复用瀑布流"卡片数据 + 评论数据"两个 API。
- **数据流**: `onLoad` 走与瀑布流卡片完全相同的 fetch 逻辑(抽 `utils/community-post-loader.js` 公共),减少双轨维护。

### 2.6 列表性能(瀑布流 + 评论)

| 维度 | 策略 |
|------|------|
| **节流加载** | `onReachBottom` 触发分页,`skip` 步长 20;`loadingMore` 标志位防重复触发 |
| **图片懒加载** | `<image lazy-load="{{true}}">` + `mode="widthFix"`(高度自适应,减少 layout) |

| **下拉刷新** | `enablePullDownRefresh: true`;刷新时**重置** columns + heights,避免残影 |
| **setData 瘦身** | 列表只存必要字段(剔除后端冗余),评论只存 `id, user_name, content, images, create_time_display, is_unread` |

## 3. 用户需求 2 — 富文本编辑器(本次重点)

### 3.1 方案对比表

| 方案 | 实现难度 | 图文混排 | 多选图片 | 定制度 | 维护性 | 推荐度 |
|------|---------|---------|---------|--------|--------|--------|
| **微信原生 `<editor>`** | 低(内置,直接用) | 支持(图片/文字) | 支持(但样式/插入位置受控) | 中(只暴露 toolbar 部分 hook) | 高(官方维护) | **⭐⭐⭐⭐** |
| **自研(`textarea` + 块)** | 高(1 人/3 天起) | 完全可控 | 完全可控 | 高 | 自己维护 | ⭐⭐ |
| **第三方**(Vant/Taro UI Editor) | 中 | 支持 | 支持 | 中(主题受限于库) | 中(库升级风险) | ⭐⭐(本项目无依赖传统) |

**推荐:微信原生 `<editor>`**。

理由:

1. 富文本场景是**长文 + 偶尔插图**,与"图文混排自由排版"诉求高度匹配,原生 `<editor>` 已经在微信体系内验证;
2. 维护成本最低(微信官方长期支持),不需要引入新依赖,符合"以源码为准 + 最小变更"基调;
3. 自研 textarea+块的工作量(光标/选区/图片插入位置/撤回 redo)远超本期时间预算;
4. 第三方组件维护节奏慢、定制性差、容易引入未审计代码;
5. 现有 `utils/community-content.js` 的 JSON blocks 协议**继续作为存储格式**—— `<editor>` 输出 HTML,前端在 `onInput` 时把 HTML 转回 JSON blocks(见 §3.4);后端 `content` 字段保持 `text/longtext` 不动(后端 agent 评估)。

### 3.2 实现方案

#### 3.2.1 编辑器初始化

```html
<editor
  id="post-editor"
  class="editor"
  placeholder="写点什么吧..."
  bindready="onEditorReady"
  bindinput="onEditorInput"
  bindstatuschange="onStatusChange"
/>
```

- `editor` 组件用 `<form>` 包裹 + `<button form-type="submit">` 提交,保证键盘"完成"键同步;
- 工具栏使用 `<editor>` 内置 toolbar,通过 `id="toolbar"` 关联;
- 高度:`min-height: 480rpx`,根据内容自动撑高。

#### 3.2.2 多图选择 + 插入

- 工具栏"图片"按钮 → 触发 `wx.chooseMedia({ count: 9, mediaType: ['image'], sizeType: ['compressed'] })`。
- **并发上传策略**:
  - 最多 3 并发(经验值,平衡速度与 wx.uploadFile 队列限制);
  - 队列实现:用 `Promise` + 简单 `concurrencyLimit` 函数(工具函数 `utils/concurrency.js`,与现有 `utils/api.js` 同级);
  - 失败重试:每张图最多 2 次自动重试,失败后弹"重试/取消"。
- **占位插入**:
  - 用户选完图后**立即**在光标位置插入一个 `<img src="local://{{tmpFilePath}}" data-tmp="true" />` 占位;
  - 上传成功后,把 `src` 替换为后端返回的 `url`,并打 `data-uploaded="true"`;
  - 失败时移除占位 + 弹 toast。
- **上传 endpoint**:`backend/app/api/v1/endpoints/uploads.py:194-223` `POST /api/v1/uploads/community-image`(已存在,直接复用)。

#### 3.2.3 编辑器 → JSON blocks 序列化

- 监听 `bindinput`,在 debounce(200ms)后调用 `utils/community-content.js::parsePostContent(html)` —— **升级该函数,支持 HTML 输入**:
  - 简化方案:用轻量 HTML 解析(自实现 mini 解析器:split 标签 → 区分 `<img>` / `<p>` / 纯文本),不要引外部 markdown/html 库;
  - 转换规则:
    - `<p>...</p>` → `{ type: 'text', text: '...' }`
    - `<img src="..." data-uploaded="true">` → `{ type: 'image', url: '...' }`
    - 多个相邻 `<img>` 合并为一个 `{ type: 'images', images: [...] }`
- 提交时:把当前 HTML 直接放入 `content`(主存),JSON blocks 放入 `images`(平铺所有 URL,沿用现有 `CommunityChannelPostCreate.images` 字段)。
- **存储格式**: **HTML(主存)+ JSON blocks(辅助)**,与后端 `04-后端接口方案.md` 约定保持一致(本方案推荐 HTML,JSON blocks 仅作前端可读缓存,后端不必解析)。

#### 3.2.4 草稿保存

- **触发**: `onInput` debounce 500ms;
- **存储**: `wx.setStorageSync('community_post_draft_<channelId>', { title, html, blocks, updatedAt })`;
- **恢复**: 页面 `onShow` 时检查 draft 时间戳,若 < 24h 则提示"恢复草稿"弹窗;
- **清理**: 发布成功 / 主动丢弃时清空对应 key;
- **冲突**: 仅保留最新一份,不区分"编辑"vs"新建"(本期范围内仅新建)。

### 3.3 兼容性 & 体验细节

- `<editor>` 在 iOS / Android 真机行为差异大:
  - 输入法弹起时需要 `adjust-position="{{true}}"` + 自定义键盘工具栏(在 `bindfocus` 时把工具栏 translateY)。
- 空态: placeholder 文案"写点什么吧…(最多 10000 字)"。
- 字数限制: 10000(沿用后端约束);`<editor>` 内置 `maxlength`。
- 安全: 提交前过滤 `<script>` / `on*` 事件属性(简易黑名单即可,因为内容只在本应用渲染)。

## 6. 状态管理与数据流

### 6.1 现有 `utils/api.js` 评估

- 优点:
  - 单一文件聚合所有 API,函数命名一致(动词 + 资源);
  - `requestWithBody` 抽出共用 Promise 封装;
  - `getImageUrl` 处理相对/绝对路径,集中处理;
  - `ApiError` 类 + `normalizeApiErrorMessage` 中文友好提示。
- 缺点:
  - **重复模板**: 每个 GET/POST 函数都是一段几乎一样的 `new Promise` 模板,约 40+ 处;
  - **未拦截 401**: 没有统一处理 token 过期跳转登录;
  - **未去重并发**: 同一资源高频请求会重复打后端;
  - **未暴露订阅**: 没有 EventBus,跨页面通信靠 `wx.setStorage` 同步。
- 结论: **结构可用,本期内不重写**;但:
  - 在 `api.js` 头部加**轻量增强**:`apiGet(url, opts) / apiPost(url, data, opts)`,后续新增 API 走通用方法(本期不强制改造老函数);
  - 加 `attach401Handler(handler)`,在 `app.js` 启动时挂上"401 → 跳登录"。

### 6.2 是否需要引入轻量 store

| 维度 | 不引入 | 引入(参考 `utils/store/` 已有说法) |
|------|--------|------------------------------------|
| 跨页共享数据 | 走 `wx.getStorage` | 响应式 + 内存缓存 |
| 频道列表缓存 | `community/index` 重拉 | 内存 + 持久化 |
| 未读数 | 各页自己 `getUnreadCount` | 全局唯一来源,事件驱动刷新 |
| 复杂度 | 低 | 中(要新写 store) |

**结论**: **本期不引入新 store**。理由:
- `miniprogram/utils/` 下**目前无 `store/` 目录**(已确认),与 `miniprogram/CLAUDE.md` 中"状态管理工具"的说法有出入;若引入需先搭建基础设施;
- 跨页通信需求不高(主要是"未读数");改用 `wx.$emit` / `wx.$on`(App 实例的事件总线)即可,零成本;
- 频道列表缓存用 `wx.setStorageSync` 同步即可,刷新策略走 `onShow`。

### 6.3 缓存策略(本期)

| 数据 | 缓存位置 | 过期 | 失效时机 |
|------|----------|------|----------|
| 频道列表 | `wx.setStorageSync('community_channels', { ts, items })` | 5 分钟 | 发布/创建/邀请后清缓存 |
| 频道详情(单频道) | `wx.setStorageSync('community_channel_<id>', ...)` | 5 分钟 | 同上 |
| 未读数 | 内存(`app.globalData.unreadCount`) | 实时 | 收到新通知/已读时更新 |
| 草稿 | `wx.setStorageSync('community_post_draft_<channelId>', ...)` | 24 小时 | 发布成功/主动丢弃 |
| 频道头像临时路径 | 不缓存(下次发布重新选) | - | - |

### 6.4 跨页通信

- **新动态发布成功**(`post-create`): `wx.navigateBack` + `getCurrentPages()[getCurrentPages().length-2].onRefresh?.()`,或通过 `app.globalData` 暴露 `eventBus`:
  ```js
  // app.js
  onLaunch() {
    this.globalData.bus = {
      on: (event, fn) => { this._bus = this._bus || {}; (this._bus[event] = this._bus[event] || []).push(fn); },
      emit: (event, payload) => { (this._bus?.[event] || []).forEach(fn => fn(payload)); },
    };
  }
  ```
- **未读数变化**(notifications 全部已读 / 收到新通知): 同上 eventBus。

> 备注:这是**绕过"禁止直接操作全局 Data"** 的合规方式—— `app.globalData.bus` 本身是工具,不是业务数据。`auth.js` 仍然走 storage,符合铁律。
