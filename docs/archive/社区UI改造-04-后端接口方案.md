## 0. 范围与基线声明

本阶段为「小程序社区模块 UI 大改」项目的**后端接口方案**，**只产出方案文档，不写代码**。
设计稿由用户后续提供，本文档只解决技术骨架与方案决策。

### 重要原则（来自需求基线，沿用执行）

- **以源码为准**：spec 与落地有差异时，按现有源码方向走，后续 spec 文档要被改造以匹配代码。
- **不依赖 spec 未实现的接口**：默认不纳入本期（置顶 / 官方标签 / 转让管理员 / PUT/DELETE /channels/{id} 等）。
- **数据库变更**：新建表写 `backend/sql/table.sql`；字段改动写 `backend/sql/field.sql`；**禁止走 `db_migrations.py`**；已经 `CREATE TABLE` 的表只做 ALTER。
- **权限前置阅读**：本期涉及频道可见性 / 角色 / 混合账号（管理员+成员）已读完 `docs/handover/权限系统架构文档.md` 与 `docs/handover/权限系统设计原则.md`；所有建议改动已按"5. 强制检查清单"自检过（详见 §10）。

## 4. 需求 2 富文本 - 后端 `content` 字段方案

### 4.1 当前 `content` 字段类型

- ORM: `schemas.py:376` `content = Column(Text, nullable=False)`，MySQL 实际为 `TEXT`（最大 64KB / ~21000 汉字）。
- 现状 Pydantic 限制：`CommunityChannelPostCreate.content` `max_length=10000`（约 10000 字符）。
- 落地图片存储：与正文平级，单独 `images: List[str]` 字段（`Text` 存 JSON 数组）。

### 4.2 富文本存储格式三选一

| 方案 | 后端利 | 后端弊 | 与现有系统契合度 |
|------|--------|--------|------------------|
| **HTML** | 简单；前端 `<rich-text>` 或微信原生 `<editor>` 输出的就是 HTML；CRUD 几乎不动 | XSS 风险高（必须做白名单清洗）；难以做客户端无关的二次编辑；图片以 `<img src>` 内联，**难以做图片安全审核**（只能从 HTML 字符串中正则提取 URL） | 与现有 `images` 字段语义有重叠，需要回归 |
| **Markdown** | 干净；渲染器开源成熟；评论展示场景友好 | 前端需要渲染器（marked / towxml / 小程序 markdown 组件）；表格 / 代码块等扩展语法需要约定；图片同样以 `![]()` 内联，审核需正则提取 | 与现状 `content` 字段语义相近（纯文本），但要前后端约定语法子集 |
| **JSON blocks** | 结构化（block 类型 + 内联数据）；图片、视频、表情、@ 都可建模；**图片 URL 在 JSON 中结构化**，审核时精确取 `image` 类型 block 的 URL | schema 工作量大（要定义 `paragraph/image/quote/code/...` 块类型）；CRUD 改动最大；老数据要迁移 | 与现状 `images` 字段语义**最契合**——图片天然在 blocks 中 |

### 4.3 推荐方案：JSON blocks（结构化），老数据用 Markdown 兜底

**主存储：JSON blocks**
- 写入：`content_blocks` 列（`JSON` 类型），schema 草案：
  ```json
  {
    "version": 1,
    "blocks": [
      {"type": "paragraph", "text": "今天参加了…"},
      {"type": "image", "url": "/uploads/community/posts/2026/06/abc.jpg", "width": 1080, "height": 720},
      {"type": "paragraph", "text": "收获满满。"},
      {"type": "mention", "user_id": 42}
    ]
  }
  ```
- 读取：直接 `json.loads` 校验后下发；前端按 block 类型渲染。
- 老数据兼容：**保留 `content` 字段**作为 Markdown 兜底（`Text` 字段保留），新发的帖子 `content` 留空（前端不写）、`content_blocks` 为主。

**为什么选 JSON blocks？**

1. **图片审核友好**——`community_media_moderation_task` 的 `media_url` 字段可以直接从 `image` 块 `url` 提取，不再依赖正则扫 HTML/Markdown。
2. **可扩展性**——后续加 `video` / `code` / `mention` 块不需要改表结构，只需要前端 block renderer 加分支。
3. **避免 XSS**——块结构天然把"展示样式"和"数据"分离，渲染器是白名单（不像 HTML 那样可以写 `<script>`）。
4. **可降级**——若 `content_blocks` 为空但 `content` 非空，渲染端按 Markdown 展示，保证老帖正常。

**理由**（对比 HTML / Markdown）：

- HTML 的 XSS 风险在生产必须做严格的 bleach/sanitize，且 sanitize 后反而失去了富文本的某些展示能力。
- Markdown 看起来"轻"，但前端引入 markdown 渲染器要重新打包，且图片/链接/代码块的样式与 `editor` 组件输出格式存在差异，会出现"编辑看到的"和"发布后看到的"不一致。
- JSON blocks 一次投入大，但**对审核、扩展、未来 5.0 的"@ 某人 / 投票 / 卡片" 都留好了接口**。

### 4.4 schema 增量

| 变更 | 是否必做 | 理由 |
|------|----------|------|
| 新增 `content_format` 字段（`VARCHAR(16) DEFAULT 'text'`） | **必做** | 用于兼容"老数据为 `text`、新数据为 `blocks`"的混合场景；前端拿到这个字段决定渲染策略 |
| 新增 `content_blocks` 字段（`JSON NULL`） | **必做** | 主存储 |
| 保留 `content` 字段为 Markdown 兜底（`Text` 不动） | 必做 | 老数据兼容 |
| `content` 长度上限调整 | **必做**——`MEDIUMTEXT`（最大 16MB） | 10000 字符对 JSON blocks 太小；统一放 `MEDIUMTEXT` |
| 新增 `cover_url` 字段 | 可选 | 用于列表卡片封面；本期可省，瀑布流卡片用第一张 image block 即可 |

### 4.5 内容审核对富文本的影响

| 审核环节 | 现状 | 富文本后改造 |
|----------|------|--------------|
| 文本审核（`msg_sec_check`） | `_resolve_review_status(text=f"{title}\n{content}")` | 拆解 `content_blocks` 内所有 `paragraph` / `quote` 块 `text` 字段，拼接后送审 |
| 图片审核（`media_check_async`） | 从 `body.images` 数组提取 URL | 拆解 `content_blocks` 内 `type == "image"` 块，提取 `url` 列表送审 |
| 头像审核 | 暂无（频道头像新建时无审核） | 复用 `media_check_async`；新增 `_submit_image_tasks(item_type="channel_avatar", item_id=channel.id, ...)` 流程（详见 §6） |
| 审核失败提示 | 现状：返回 500/普通错误 | 建议：审核失败时 `status=0`，前端提示"内容正在审核中，请稍候"；不直接拒绝用户输入 |

**新增** `item_type` 取值：
- `channel_post`（已存在）
- `channel_comment`（已存在）
- `channel_avatar`（**新增**）

### 5.1 频道动态列表接口增加 `top_comments`

- 在 `CommunityChannelPostResponse` Pydantic 模型中**新增** `top_comments: List[CommunityChannelCommentResponse] = []` 字段。
- 在 CRUD `list_channel_posts`（`crud_community_channel.py:518`）中：
  - 当前已经 `outerjoin(CommunityChannelComment, ...)` 拿 `comment_count`。
  - **改造**：把 `func.count` 改为 `func.array_agg`（MySQL 不支持 `array_agg`；用 `GROUP_CONCAT` 拿到 id 列表后再二次查询）或**两次查询**（先拿 post id 列表，再一次性 `IN` 查评论表，按 post_id 分组取前 N 条）。
- **推荐：两次查询**（更通用，MySQL/PG 都可移植）：

  ```python
  # 1) 拿帖子列表（已有）
  post_rows = db.query(CommunityChannelPost, ...).offset(skip).limit(limit).all()
  post_ids = [p.id for p, _, _ in post_rows]

  # 2) 一次性查这些帖子的前 N 条评论
  # MySQL 8.0 窗口函数；老 MySQL 5.7 退化到子查询 + LIMIT
  comments = db.query(CommunityChannelComment, User).join(User, ...).filter(
      CommunityChannelComment.post_id.in_(post_ids),
      CommunityChannelComment.status == 1,
      # 取每帖最早 N 条
  ).order_by(CommunityChannelComment.post_id, CommunityChannelComment.create_time.asc()).all()

  # 3) 在 Python 层按 post_id 分组
  comments_by_post = defaultdict(list)
  for c, u in comments:
      if len(comments_by_post[c.post_id]) < TOP_COMMENT_N:
          comments_by_post[c.post_id].append(serialize(c, u))
  ```

**N 的建议值**：`TOP_COMMENT_N = 2`（默认 2 条，瀑布流两列卡片高度差异可控；若单列流可放 3 条；通过环境变量 `COMMUNITY_TOP_COMMENT_N` 调）。

**是否加在 `CommunityChannelPostResponse` 还是单独接口**：
- **加在 `CommunityChannelPostResponse`**——瀑布流一次请求拿齐数据，省一次往返；列表响应体增大可控（每条评论 ≤ 200 字 + 用户名）。
- 评论分页详情仍走 `/posts/{pid}/comments` 单独接口（保持现状）。

### 5.2 评论分页接口：cursor vs offset

**现状**：评论分页用 `skip/limit`（`community.py:339` 与 `community.py:820`）。

**评估**：
- 频道评论按 `create_time asc` 排序——评论流是**追加为主**（新评论追加到尾部），用户查看时"加载更多"是顺势的；
- 评论总数一般较小（热门帖几百条以下），`OFFSET 200` 性能尚可；
- 瀑布流卡片内嵌评论**不需要** cursor，瀑布流整体才是 cursor/offset 议题。

**决策**：
- **本期评论分页继续用 `skip/limit`**，不切换 cursor；
- 若后续评论量爆炸（>10k/帖），再考虑 cursor（`after_id` 或 `after_create_time`）。
- 列表接口（频道动态、通知、审核队列）也保持 `skip/limit` 现状，瀑布流改用**滚动到底加载下一页**的标准模式。

### 5.3 "查看更多评论"的批量加载策略

前端交互：瀑布流卡片内默认显示 2 条 `top_comments`，卡片底部"查看 N 条评论"按钮点击后**展开内联**：

- 方案 A（**推荐**）：前端点击时**调** `GET /channels/{id}/posts/{pid}/comments?skip=0&limit=20` 拿前 20 条，卡片内切换为完整评论列表。
- 方案 B：列表接口一次性返回全部评论（**不推荐**——响应体不可控）。

**后端建议**：
- `/comments` 接口默认 `limit=20` 比现状 50 略小（瀑布流卡片内嵌不需要 50 条）；
- 仍支持 `skip/limit` 分页参数，前端需要时继续"加载更多"。

### 5.4 频道列表加"最新动态摘要"

为方便频道首页卡片显示**"最近 24h 有人发了 3 条动态"**，可考虑在 `CommunityChannelResponse` 加：

- `last_post_excerpt: Optional[str]` —— 最近一条动态的标题或前 30 字。
- `last_post_create_time: Optional[datetime]` —— 最近动态时间。

**本期是否做**：
- **P2 改造**：CRUD `list_user_channels` 加一次 `LEFT JOIN LATERAL`（MySQL 8.0 支持）或子查询拿 `last_post`。
- 若工作量评估超 0.5 人日，**可推迟到 Phase 2**。

## 6. 需求 3 频道头像 - 后端方案

### 6.1 现状

- `CommunityChannelCreate` (`community.py:135-164`) **已支持** `avatar_url: Optional[str]`。
- `CommunityChannel` ORM (`schemas.py:400`) **已支持** `avatar_url: String(500)`。
- 头像上传走 `POST /api/v1/uploads/community-image` (`uploads.py:194`)，**已就绪**。

### 6.2 上传接口能力回顾

| 维度 | 当前状态 | 是否满足需求 3 |
|------|----------|----------------|
| 接口路径 | `POST /api/v1/uploads/community-image` | OK |
| 权限 | `deps.get_current_user`（任意已登录用户） | OK |
| 文件大小 | 5MB（`MAX_POSTER_SIZE`） | 略大（头像 1MB 足够），可保持现状 |
| 文件类型 | PNG / JPG / JPEG | OK |
| 内容审核 | **未走微信 media_check_async** | **需要补** |
| 压缩 | 无（原图保存） | 可选（头像建议压缩到 512×512 复用 `_optimize_avatar_image`） |

### 6.3 改造建议

**方案选择**：
- 方案 A：复用现有 `/community-image` 接口，**前端拿到 url 后**单独调一次"创建/更新频道"接口把 `avatar_url` 写入；上传时**不走审核**。
- 方案 B：**新增** `POST /api/v1/community/channels/avatar-upload` 专门通道，与 `uploads.py` 的 `community-image` 共用底层存储逻辑，但强制走微信 `media_check_async` 审核、压缩到 512×512（复用 `_optimize_avatar_image`）。
- 方案 C：复用 `/community-image` 但在调用方主动调 `submit_media_check_async`，由前端把 `trace_id` 暂存，频道创建/更新成功时再关联 `media_url` + `trace_id` 入 `community_media_moderation_task` 表。

**推荐方案 B**：
- 单独通道的好处：**mime 检查更严格**（仅 image/jpeg、image/png）、**强制压缩**（节省 CDN 流量）、**强制审核**（不依赖调用方"记得审核"）。
- 缺点：多一个接口，前端接入多一次跳转。
- 落地：复用 `uploads.py:71-109` 的 `_optimize_avatar_image` 工具函数，**新增** 一个 `POST /community/channels/avatar-upload` endpoint。

### 6.4 默认头像生成（可选）

当频道创建者**不上传头像**时，可考虑服务端生成默认头像（基于频道名首字 + hash → 背景色）。

**本期建议**：
- **暂不实现**——前端有占位图（initials）方案即可；
- 若设计师后续要求"统一风格默认头像"，可放 Phase 2，用 Pillow 在上传时一并生成。

### 6.5 头像审核流程

- 频道创建时若提供 `avatar_url`，**强制**走 `media_check_async`：
  1. 写 `community_channel.status=0`（或保留 `1` 但加审核标记）；
  2. 写 `community_media_moderation_task`（`item_type='channel_avatar'`），`item_id=channel.id`；
  3. 等微信回调后 `_evaluate_and_finalize_item` 改 `channel.status=1`（通过）或 `0`（驳回）；
- **问题**：现状 `community_media_moderation_task` 关联的是 `community_channel_post` / `community_channel_comment` / `activity_post` / `activity_comment` 四类，**没有 `community_channel` 自身的 `item_type`**。需要 §8 增加 item_type 扩展支持（不需要新表，复用现有 task 表）。

## 7. 微信内容安全审核流程复盘

### 7.1 当前 `community_media_moderation_task` 流转

1. **创建任务**（同步触发）：
   - `community.py:187-206` `_submit_image_tasks`
   - 调 `submit_media_check_async(image_url)` 拿到 `trace_id`
   - 写 `CommunityMediaModerationTask(item_type, item_id, media_url, trace_id, status='pending' | 'failed_submit', reason)`

2. **微信回调**（异步）：
   - `POST /api/v1/community/moderation/wechat-media-callback` (`community.py:967`)
   - 校验签名（`_verify_wechat_media_callback_signature`，`community.py:244-257`）
   - 解析 XML/JSON payload，取 `trace_id` + `suggest`
   - 调 `crud_community_moderation.update_media_task_result`
   - 调 `_evaluate_and_finalize_item`（`community.py:227-242`）汇总所有图：任一 `risky`/`failed_submit` → 整条 `status=0`；全 `pass` → `status=1`

3. **状态字段取值**：
   - task.status: `pending` / `failed_submit` / `pass` / `risky`（注意：表 DDL 注释写 `pending/approved/rejected`，但实际代码用 `pass/risky`，**注释与实现不一致**）
   - post.status: `1` 正常 / `0` 待审核 / `-1` 驳回（推断，CRUD 接受任意整数）

### 7.2 健康度评估

| 维度 | 评价 | 是否需要修 |
|------|------|------------|
| 签名校验 | 有，`sha1(sorted(token, timestamp, nonce))` 标准微信方式 | OK |
| XML 解析 | `xml.etree.ElementTree` | OK |
| JSON 解析 | 兼容 | OK |
| trace_id 全局查询 | `get_media_task_by_trace_id_global`（不带 tenant_id 过滤） | **可接受**——trace_id 本身唯一 |
| 重试 / 幂等 | 回调重试时直接 `_evaluate_and_finalize_item` 二次计算 | OK |
| 失败 task 的 `media_url` 仍在前端 | 走 `community_media_moderation_task.media_url` 字段 | OK |
| 审核失败的**用户提示** | **缺失**——前端需要轮询或 webhook 才有提示 | **前端缺口**，后端暂无"审核状态查询"接口；可考虑在 `CommunityChannelPostResponse` 已有的 `status` 字段直接展示（0=待审核，-1=已驳回） |
| `item_type` 枚举扩展 | 现状只覆盖 4 类 | **需求 3 需扩展到 5 类**（加 `channel_avatar`） |
| DDL 注释与代码不一致 | 表注释写 `pending/approved/rejected`，代码用 `pending/pass/risky/failed_submit` | **建议同步**（详见 §8） |

### 7.3 富文本拆解审核策略

- **文本**：后端拿到 `content_blocks` 后，遍历 `type == 'paragraph' | 'quote'` 块，累加 `text` 字段，调用 `check_text_security(text_combined)`。
- **图片**：遍历 `type == 'image'` 块，累加 `url` 字段，对每个 URL 调 `submit_media_check_async(url)`，写 `community_media_moderation_task`。
- **@ 提及**（`mention` 块）：`user_id` 字段跳过文本审核；若 mention 块带 `text` 字段也合并到文本审核。
- **视频块**（`video` 块，**未来扩展**）：走 `media_check_async` 同图片流程；本期不实现 `video` 块类型。

### 7.4 头像审核流程

- 创建频道时若 `avatar_url` 非空：
  1. 写 `CommunityChannel`（`status=0`）；
  2. 调 `submit_media_check_async(avatar_url)` 拿 `trace_id`；
  3. 写 `CommunityMediaModerationTask(item_type='channel_avatar', item_id=channel.id, ...)`；
  4. 微信回调到 `wechat-media-callback`：
     - `item_type='channel_avatar'` → **新增** 路由分支，调 `crud_community_channel.update_channel_status` 改 `status=1` 或 `0`（驳回）。
- **现状缺**：`_update_item_status` (`community.py:209-225`) 的 4 个分支**不含 `community_channel`**，需扩展。

### 7.5 审核失败的用户提示

- **现状**：用户在 `community-post-create` 提交后，**后端 status=0 时直接返回 200**（`create_post` 流程见 `community.py:292-334`），前端读不到"被驳回"的信号。
- **建议**：
  - 短期：前端在发布成功后 5 秒内调 `GET /channels/{id}/posts/{pid}` 轮询一次 `status`；若 `status=0` 提示"审核中"，若 `status=-1` 提示"内容未通过审核"。
  - 中期：增加"我发布的待审核"列表接口（管理员视角与用户视角分开），本期**不实现**。

ALTER TABLE `community_media_moderation_task`
  MODIFY COLUMN `status` varchar(32) NOT NULL DEFAULT 'pending' COMMENT '状态：pending/pass/risky/failed_submit';

## 9. 性能与安全

### 9.1 长列表分页策略（cursor vs offset 对照）

| 场景 | 推荐 | 理由 |
|------|------|------|
| 频道动态瀑布流（按 create_time desc 排序） | **offset 即可** | 用户刷新看新内容场景多，`OFFSET 0` 是热点；翻到第 N 页才付出 N*page_size 行扫描成本 |
| 通知列表 | **offset** | 通知不会无限增长（用户每天最多几十条） |
| 审核队列 | **offset** | 管理员场景，并发低 |
| 评论分页（按 create_time asc 排序） | **offset** | 评论总数 < 几千；老 MySQL offset 性能足够 |
| 频道列表（用户已加入的频道） | **offset** | 一般 < 50 个 |

**cursor 分页**的引入时机：
- 当 `OFFSET > 1000` 时考虑；
- 当有"实时增量"（用户在第 1 页看时新帖子源源不断）需要稳定游标时考虑。

**结论**：本期全部维持 `skip/limit`；cursor 留作未来扩展点。

### 9.2 评论嵌套查询性能

- `list_channel_posts` 当前已 `outerjoin(CommunityChannelComment, ...)` + `group_by` 拿 `comment_count`。
- 改造加 `top_comments` 时（§5.1），**不要在主查询里再 join 一次**（会出现 N+1 与笛卡尔积风险）。
- **采用"两次查询"模式**：主查询拿 post 列表，第二次按 `post_id IN (...)` 一次性查评论。

**索引建议**（已有 `idx_ccc_post_id`，可补联合索引）：
```sql
ALTER TABLE `community_channel_comment`
  ADD KEY `idx_ccc_post_status_time` (`post_id`, `status`, `create_time`);
```

### 9.3 频道未读计数性能

- **当前不存在**频道动态未读数；只存在站内信未读（`/notifications/unread-count`）。
- 频道动态未读若要加：
  - 方案 A：维护 `community_channel_member.last_read_post_id`，每次进入频道更新。
  - 方案 B：每次扫表（`COUNT(*)` + 关联）——简单但慢。
  - 方案 C：维护 `community_channel_member.unread_post_count` 列，发帖/读帖时维护。
- **推荐方案 A + C 组合**：
  - 读帖：用户进入频道详情时 `UPDATE community_channel_member SET last_read_post_id=MAX(post_id) WHERE user_id=...`
  - 计数：列表接口计算 `MAX(post_id) - last_read_post_id`（仅当 post 持续递增时有效）
- **本期不做**——需求 1/2/3 均未涉及未读数。

### 9.4 防刷

| 维度 | 现状 | 建议 |
|------|------|------|
| 发帖频率限制 | 无 | **建议加 Redis 限流**：`user_id` 维度，1 分钟 5 条 / 1 小时 30 条 |
| 评论频率限制 | 无 | 同上，1 分钟 10 条 |
| 敏感词 | `check_text_security` 调微信 `msg_sec_check` | 已覆盖；富文本化后注意拆解 `content_blocks` 中的文本 |
| 图片审核兜底 | `media_check_async` 已覆盖 | 已覆盖 |
| 头像审核兜底 | 现状缺 | 见 §6.3 方案 B |
| SQL 注入 | SQLAlchemy ORM 已防御 | OK |
| XSS | 现状 `content` 字段直接出参（前端负责渲染） | **富文本后** `content_blocks` 由前端白名单渲染器渲染，**后端不做 sanitize**（结构化数据天然安全） |

### 9.5 PII 处理（对照 `backend/app/core/pii.py`）

- `community_post` / `community_comment` / `community_channel` / `community_channel_post` / `community_channel_comment` **不存 PII**（除 `author_user_id` / `user_id` 关联到 `user.name`，而 `User.name` 走 `EncryptedContactMixin` 已加密）。
- `data` 字段（通知 JSON）可能存 `inviter_name`（邀请人姓名）——**理论上是 PII**。
  - 现状：明文存 JSON；
  - 建议：通知 `data` 字段内的 `inviter_name` 改成 `inviter_masked_name`（如"张*"）以避免 PII 泄露；
  - **本期不实现**——重构面太大，列为后续 PII 治理项。

## 11. 给其他 Agent 的协作接口

### 11.1 → 前端 agent

- **频道动态列表接口**（`GET /channels/{id}/posts`）**响应体变更**——`items[].top_comments: List<CommentPreview>` 字段新增；CommentPreview 结构复用 `CommunityChannelCommentResponse`。
- **创建/更新动态接口**（`POST /channels/{id}/posts`）**入参/出参变更**——
  - 入参新增 `content_format: 'text' | 'blocks'`、`content_blocks: List<Block>` 字段（与 `content` 二选一必填）。
  - 出参 `post.content_format / post.content_blocks` 同步返回。
  - 老数据**仍可**只传 `content`，**后端默认** `content_format='text'`。
- **频道头像**（需求 3）——前端**只调新接口** `POST /community/channels/avatar-upload`，拿到 url 后填入 `CommunityChannelCreate.avatar_url`。
- **瀑布流卡片渲染**——`top_comments` 默认 2 条；用户点击"查看更多"时**前端二次请求** `/channels/{id}/posts/{pid}/comments?skip=0&limit=20`。
- **审核失败提示**——前端在发布后 5 秒轮询 `GET /channels/{id}/posts/{pid}` 检查 `status`；0=审核中，-1=已驳回，1=通过。
- **错误码**——沿用 HTTP 状态码 + 中文 `detail`（现成约定）。
- **富文本 blocks 协议**（待与前端对齐）：

  ```ts
  type Block =
    | { type: 'paragraph'; text: string }
    | { type: 'heading'; level: 1|2|3; text: string }

    | { type: 'quote'; text: string }
    | { type: 'code'; language?: string; text: string }
    | { type: 'mention'; user_id: number; text?: string };
  ```
