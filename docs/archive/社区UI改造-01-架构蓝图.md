## 0. 阅读须知 & 关键事实校准

1. **以源码为准**(需求基线 §1):凡 spec 与代码不一致处,以代码为准,后续 spec 文档反向更新。
2. **ORM 实体位置校准**:摸底报告说"ORM 实体在 `backend/app/schemas.py`"——已通过 Read 确认:
   - `backend/app/schemas.py:1-15` 顶部 `Base = declarative_base()` 并混入 `TimestampModel`/`BaseModel`,**全文件都是 SQLAlchemy ORM 实体**。
   - `backend/app/models/community.py` 全是 **Pydantic BaseModel**(DTO),与 `models/user.py` 等一致,与 `schemas.py` 不互相引用(符合 `backend/app/CLAUDE.md` "边界铁律")。
   - **结论:架构师后续讨论 ORM 字段、表结构时,均指 `backend/app/schemas.py`**,不再使用 "schemas 里只有 Pydantic" 这类描述。
3. **摸底报告 §5.6 的关键错误需要修正**:
   - 摸底说"`community_channel_post` / `community_channel_comment` / `community_notification` 三张表在 `table.sql` 里**没有**",但 `schemas.py:421-453` 实际**已经定义**了这三个 ORM 类(`CommunityNotification`、`CommunityChannelPost`、`CommunityChannelComment`),只是:
     - `table.sql:416-487` 中**确实**没有对应建表语句(表名漂移);
     - CRUD 与 API **完全没有**引用这三个类,只用老的 `CommunityPost`/`CommunityComment`(`backend/app/crud/crud_community_post.py:31,55,62,80,94,147,205` 全部走 `activity_id` 路径);
     - 即 **ORM 实体在 `schemas.py` 里、但表没建、CRUD/API 也不读**。
   - **架构决策(详见 §4)**:这一现状不强制本阶段修复,优先把"老表 + channel_id 字段"补齐,把"频道动态"语义挂在 `community_post` 上,与代码现状对齐。
4. **权限前置阅读已完成**:已读完 `权限系统架构文档.md` + `权限系统设计原则.md`。本阶段改动只涉及"读取社区列表 / 评论发表"等用户能力,必须遵守:
   - **不新增** `ctx.role == "admin"` 硬拦截用户能力的代码(红线 2);
   - **前端不做** 二次可见性过滤覆盖后端结果(红线 3);
   - **不直接** 把 `User.name/phone` 当 SQL 列 `label/group_by`(红线 4,见摸底 §4 第 6 条历史教训);
   - 任何权限相关改动都需**先列回归清单**(红线 5)。

### 1.2 详情页 `community-post-detail` 的存废建议(强烈建议先定)

#### 三种方案对比

| 方案 | 描述 | 优点 | 缺点 | 评估 |
|------|------|------|------|------|
| **A. 完全废弃** | 删页/路由,所有交互(评论、展开)都收回到瀑布流卡片内 | 信息架构最简,代码体积最小 | 1) 失去分享深链落点 2) 系统通知/邀请接受后的"跳转到动态"找不到落点 3) 第三方扫码/小程序码分享动态无法落地 | **不推荐** |
| **B. 保留作"分享/深链落点"** | 主流交互走瀑布流,详情页仅作"外部入口兜底页":分享链接、系统通知点击、外部扫码等场景打开详情页后,用户可"返回首页" | 1) 主流交互扁平 2) 仍然有可控的深链落点 3) 与"评论原位展开"不冲突 | 详情页不能"喧宾夺主":需弱化评论交互(显示"在首页评论"),但仍要承担渲染任务 | **推荐** |
| **C. 重定向到首页 + 锚点** | 详情页只做 `onLoad` 跳转到 `community/index?focus=post_{id}`,首页瀑布流滚动并展开该卡片 | 1) 代码极简(详情页几乎只剩 redirect) 2) 用户始终在统一 UI 内 | 1) 滚动定位实现复杂(瀑布流两列,锚点可能错列) 2) 用户分享出去的链接会"闪一下"才到位 3) 与小程序"页面栈"心智不一致 | **不推荐(本期)** |

#### 推荐:方案 B — 详情页降级为"分享/深链兜底页"

**新角色定位**:

- **不展示评论区、不展示输入框**(保持轻量,只渲染帖子自身内容 + 顶部"返回社区"按钮)
- 入口仅限 3 类:
  1. 微信分享 / 小程序码扫码打开
  2. 站内信"文章通知"(若有)点击正文
  3. Web 管理端 / 第三方"打开动态"深链(预留)
- 主流交互(本应用内) **全部走首页瀑布流**——评论原位展开、查看全文就地展开,详情页不再触达

**对其他 Agent 的接口**:

- 前端 agent(§7 协作输出):负责把详情页改造成"只读兜底页",删评论 bar,新增"在首页打开"按钮(若无权限时隐藏);同时新增路由 `community/index?focus=post_{id}`(供详情页 fallback 与分享场景跳转首页)
- 后端 agent:保持 `/channels/{id}/posts/{post_id}` GET 接口(兜底页要拉详情),**不删除**;若想降低耦合,可加 query 参数 `view=summary` 让响应不带 `comment_count`/评论列表(默认仍带,前端决定是否用)

## 3. 关键技术决策(对应 3 项需求)

### 3.1 需求 1 — 瀑布流的架构选型

#### 候选方案

| 方案 | 实现 | 优势 | 劣势 |
|------|------|------|------|
| **A. 小程序原生 2 列 `scroll-view` + JS 调度** | 维护两个内部数组(短/长),用 `min-height` 估卡,追加时取当前更短的列 | 1) 无第三方依赖 2) 性能可控 3) 完全自研,可塑性强 | 1) 自己估高/防抖 2) 图片未加载时高度会跳动 |
| **B. 单列虚拟长列表** | 第三方 `recycle-view` / `mescroll` | 1) 实现快 2) 天然支持无限滚动 | 与"两列错落瀑布流"视觉不符(用户明确要 2 列) |
| **C. `masonry` CSS 网格** | CSS `column-count: 2` | 视觉对、代码短 | 1) 卡片顺序按列内填,不是按时间 2) 容易列间高度差大 |
| **D. 第三方瀑布流组件** | `wx-masonry` 等 | 快 | 维护性 / 团队信任成本 |

#### 决策:**A(自研 2 列 `scroll-view` + JS 调度)** — 理由

1. **用户明确要"两列错落瀑布流"**(需求基线 §需求 1),`column-count` 的列内填序在"按时间倒序"诉求下不达标;
2. 小程序对 `scroll-view` + `IntersectionObserver` 的支持稳定,自研可避免被第三方组件"卡包大小"和"维护中断"绑死;
3. 与现有 `miniprogram/utils/api.js` + `wxss` 变量体系对齐,无需引包。

#### 架构骨架(交给前端 agent 细化)

```
┌──────────────────────────────────────────────────────┐
│ scroll-view (scroll-y)                                │
│  └─ view.feed (display:flex; flex-direction:row)       │
│       ├─ view.column.left   (动态:min-height)         │
│       │    └─ post-card × N                            │
│       └─ view.column.right  (动态:min-height)         │
│            └─ post-card × N                            │
└──────────────────────────────────────────────────────┘
调度器 (瀑布流 reducer):
  state = {
    posts: [],           // 后端返回的扁平列表
    leftIds: [],         // 短列候选
    rightIds: [],
    expanded: Set,       // 原位展开全文的 post_id
    commentsOpen: Set,   // 原位展开评论的 post_id
  }
  appendPosts(newPosts):
    for post in newPosts:
      target = leftIds.length <= rightIds.length ? left : right
      target.push(post.id)
  onImageLoad(post_id, height):
    // 重新均衡两列: 短列 + 移动 post_id
  onToggleExpand(post_id):
    expanded ^= {post_id}
    // 重新计算两列 min-height
  onToggleComments(post_id):
    commentsOpen ^= {post_id}
    // 懒加载: 第一次展开时 fetch comments
```

#### 性能护栏(前端 agent 必看)

- **图片懒加载**: 卡片所有 `image` 必须 `lazy-load="{{true}}"`(沿用 spec §11.7)
- **首屏渲染**: 一次拉 20 条,`onReachBottom` 拉下一页
- **避免重复 setData**: 瀑布流分发用"一次 setData 写两列",不要逐条 push
- **图片高度预估**: 富文本内的图片,先按富文本里 `width/height` 算 aspectRatio,等 `onLoad` 后校正
- **评论懒加载**: 评论原位展开后**只拉第 1 页**(默认 10 条),"加载更多"再分页

### 3.2 需求 2 — 富文本存储格式三选一

| 格式 | 后端 `content` 字段值 | 优点 | 缺点 | 跨端(Web 接入) |
|------|------------------------|------|------|------------------|
| **HTML** | `<p>hi</p><img src="...">` | 1) 与 `wxparse` / `parser` 生态贴合 2) 前端解析成熟 | 1) XSS 风险(必须服务端白名单清洗) 2) 微信小程序 `rich-text` 节点白名单坑多 3) 后端正则清洗脆弱 | 中等(Web 端直接 `v-html`) |
| **Markdown** | `**hi** ![](img)` | 1) 纯文本,易搜索/审计 2) 编辑器库丰富(`md-editor`) 3) 后端易做安全转换 | 1) 小程序**没有原生** markdown 渲染,需第三方库 2) 复杂布局(引用、表格)展示受库能力限制 | 简单(Web 端 `marked` / `markdown-it`) |
| **JSON blocks** | `{"text":"hi","blocks":[{"type":"text","content":"hi"},{"type":"image","url":"..."}]}` | 1) 与 spec §2.2 规范完全一致(前瞻兼容) 2) 渲染端可控,易扩展 video/mention 3) 微信 `rich-text` + 第三方 `parser` 都能吃 | 1) 写作式编辑需自研 UI(toolbar + 块插入) 2) 解析器要自写或选 | 中等(Web 端自写 React 渲染器) |

#### 决策:**JSON blocks(强烈推荐)** — 理由

1. **与 spec §2.2 一致**:`{"text": ..., "blocks": [...]}` 已经是 spec 规划的存储格式,改造后 spec 与代码自然对齐,不需要反向改 spec。
2. **可扩展性最强**: 未来要加 `video` / `mention` / `link preview` 块,只需要在 `blocks` schema 里加 type,前后端契约稳定。
3. **与权限/审核不冲突**: JSON 结构便于服务端按块做敏感词审核(每块单独跑策略),不用做整段正则。
4. **微信小程序 `rich-text` 友好**: 服务端把 JSON blocks 编译成"节点数组"喂给 `<rich-text nodes="{{...}}">` 是最稳的渲染方式(见微信官方文档对 `rich-text` 的支持矩阵)。
5. **富文本编辑器选型**:
   - **编辑器(写入端)**: 推荐 **微信原生 `<editor>` 组件** + 自研工具栏(加粗/斜体/列表/图片按钮);
     - 原因:`<editor>` 已经支持图文混排、多图插入、`getContents` 拿 HTML,自研部分只是把 HTML 解析回 JSON blocks(写一个 `html2blocks` 转换器,约 100-200 行 JS)。
   - **渲染端(读出端)**: 走 `blocks → rich-text nodes` 的小工具函数(可与编辑器的反向解析共用同一份"块模型")。

#### 与"老 content 字段"的关系

- 现状:`community_post.content` 是 `Text`(摸底 §2.5)
- 决策:**新增** `content_blocks Text NULL` 字段,保留 `content` 字段(详见 §4)

### 3.3 需求 3 — 频道头像的录入交互(独立页 vs 弹层 vs 半屏 picker)

## 4. 数据模型增量决策(关键)

> 严格遵守 `backend/app/CLAUDE.md` 规则:
> - **新表** → `backend/sql/table.sql`
> - **改字段** → `backend/sql/field.sql`
> - **禁止** 走 `db_migrations.py` 补字段
> - 已 `CREATE TABLE` 的表只能 ALTER
>
> 本节**只给 SQL 草案,写在文档里**,Phase 2 实施时由后端 agent 落到 `field.sql`。

### 4.1 是否给 `community_post` 加 `channel_id` 字段?

**决策:加 `channel_id` 字段,并把 `activity_id` 改为允许 NULL。**

#### 理由

1. **消除 spec 与代码的语义漂移**: 摸底 §5.6 已指出"频道动态"语义挂在 `community_post` 上但没有 `channel_id` 字段,导致后端代码 / 文档 / SQL 三方语义不一致。
2. **以源码为准**: 既然后端实际把"频道动态"存在 `community_post` 里,那补一个 `channel_id` 字段比新建 `community_channel_post` 表(对应 ORM 类在 `schemas.py:432-453` 但无人用)更经济。
3. **不破坏老数据**: 老 activity 级帖子的 `activity_id` 已有值,字段可空、可有;用 `(activity_id IS NULL) XOR (channel_id IS NULL)` 做完整性约束(应用层做,数据库 CHECK 在 MySQL 8 之前不友好)。
4. **回退路径**: 若未来真要切到 `community_channel_post`,只需 dual-write + 数据迁移;但本期不动。

#### 互斥约束建议(应用层)

- "频道动态": `channel_id IS NOT NULL AND activity_id IS NULL`
- "活动动态": `activity_id IS NOT NULL AND channel_id IS NULL`
- 任何情况下两者不能同时有值,或同时为 NULL(不合法)

### 4.2 富文本 JSON blocks 字段

**决策:新增 `content_blocks Text NULL` 字段,保留 `content` 字段(老帖子兼容)。**

- 写入:前端把 `<editor>` 的 HTML 解析成 JSON blocks,存 `content_blocks`;同时把 `text`(纯文本)存 `content`(保留可搜索、可摘要)。
- 读取:前端优先用 `content_blocks` 渲染;为空时 fallback 到 `content`(老数据)。
- 老数据兼容:`content_blocks` 默认 NULL,UI 层用 `content` 渲染(现有体验);新发布必须写 `content_blocks`。

#### 为什么不直接替换 `content`?

1. **可搜索性**: `content` 纯文本,后端未来要加 ES 全文检索时可直接用。
2. **审核**: 微信内容安全接口是按文本/图片 url 单独送审,JSON blocks 反而难用。
3. **回退**: 一旦新格式出问题,可临时回退到 `content` 渲染。

### 4.3 是否加 `comment_preview` 缓存字段?

**决策:本期不加。**

- 理由: 瀑布流卡片首屏只显示 `comment_count`,不渲染评论内容;评论内容只有在用户**主动**"展开评论"时才请求 `/comments` 接口,首屏没有性能瓶颈。
- 等基线出来后(瀑布流实际跑 1-2 周),若发现"展开评论"那一步确实慢,再加 `comment_preview` 缓存字段(结构: `comment_count INT + top_comments JSON`)。
- 现阶段加了反而引入"双写一致性"复杂度。

### 4.4 SQL 草案(写入本文档,Phase 2 由后端 agent 落 `field.sql`)

```sql
-- 4.4.1 community_post 增加 channel_id 字段
ALTER TABLE community_post
  ADD COLUMN channel_id INT NULL COMMENT '频道ID;与 activity_id 互斥,为 NULL 表示非频道动态' AFTER activity_id,
  ADD KEY idx_post_channel_id (channel_id);

-- 4.4.2 community_post.content 改为可空(频道动态可以只有 content_blocks)
ALTER TABLE community_post
  MODIFY COLUMN content TEXT NULL COMMENT '动态纯文本(可空;与 content_blocks 二选一或并存)';

-- 4.4.3 community_post 增加 content_blocks 字段
ALTER TABLE community_post
  ADD COLUMN content_blocks TEXT NULL COMMENT '富文本 JSON blocks: {"text":..,"blocks":[{"type":"text","content":..},{"type":"image","url":..}]}' AFTER content;

-- 4.4.4 community_post.activity_id 改为可空
ALTER TABLE community_post
  MODIFY COLUMN activity_id INT NULL COMMENT '活动ID;与 channel_id 互斥';

-- 4.4.5 community_comment 同理(评论也可能有富文本,本期先不动,放 P2 评估)
-- (留空)
```

> ⚠️ **本节 SQL 仅供评审,本阶段不直接落到 `field.sql`**。Phase 2 启动时由后端 agent 整体评审、执行。

### 4.5 API 契约方向(给后端 agent §7 用)

## 6. 风险点 & 迁移路径

### 6.1 老 `activity_id NOT NULL` 数据怎么办?

| 风险 | 影响 | 缓解方案 |
|------|------|----------|
| 频道动态若 `activity_id` 仍 NOT NULL,新增 `channel_id` 时老频道动态记录可能同时为空 | API 读不到频道动态,瀑布流为空 | `activity_id` 也改为 NULL(见 §4.4.4);同时给所有"频道动态"CRUD 路径加 `activity_id IS NULL AND channel_id IS NOT NULL` 谓词 |
| 历史帖 `content` 是 HTML / 纯文本,渲染会跳变 | 老的"图文混排"突然按"纯文本"渲染,丢图 | 前端渲染策略:`content_blocks` 非空优先,空则用 `content`(老路径不变),并把老 content 标记为"纯文本",不试图解析成图 |
| 迁移期 API 兼容性 | 老 Flutter / 老小程序可能读不到 `channel_id` / `content_blocks` | 字段都是**可选** + **后向兼容**;老客户端走老字段,新客户端优先用新字段 |
| 数据完整性 | "频道动态"和"活动动态"在 `community_post` 同一张表,业务上需要互斥 | 应用层用 Pydantic `field_validator` + 前端表单控制;DB 层不强加 CHECK(MySQL 8 之前不友好) |

### 6.2 上线节奏建议

| 阶段 | 内容 | 范围 | 回滚 |
|------|------|------|------|
| **Phase 2.0 — 后端字段 + API 灰度** | 落 §4.4 SQL,改 `get_posts_by_activity` / `get_channel_posts` 走 `channel_id` 过滤;新旧字段双写 | 内部灰度(白名单租户) | `content_blocks` 删列(ALTER)即可,`channel_id` 同 |
| **Phase 2.1 — 小程序"频道创建页"上线** | 独立页 + 后端 `description` / `avatar_url` 字段实际写入 | 全量小程序 | 前端路由可关,后端字段无破坏性 |
| **Phase 2.2 — 富文本编辑器上线** | `<editor>` + JSON blocks 解析;老帖 fallback | 全量小程序 + 灰度租户 | 前端回退到单列列表 + 块编辑器;后端 `content_blocks` 不读 |
| **Phase 2.3 — 瀑布流 + 评论原位展开上线** | 首页重构;详情页降级 | 全量小程序 | 前端可回退到旧首页(单列 + 跳详情) |
| **Phase 2.4 — 顺手改造(审核页/通知页/tabBar/--green)** | 风格统一;铃铛组件化 | 全量 | 各自独立可回退 |

- **不建议 AB**: 瀑布流 / 富文本是"打开就看到"的强视觉,AB 不易得出结论;建议直接灰度租户。
- **不建议全量前置**: 字段变更先行(Phase 2.0),为前端铺路;前端必须等后端就绪。

### 6.3 富文本格式变更对历史数据的兼容

| 数据类型 | 字段 | 兼容策略 |
|----------|------|----------|
| 老 `content`(纯文本 / 老 HTML) | `community_post.content` | **保留字段**,渲染时当 fallback 路径;**不**重写老数据 |
| 老 `images`(JSON 字符串数组) | `community_post.images` | **保留**;新帖不再用(全走 `content_blocks.image`),老帖继续读 `images` 渲染 |
| 新 `content_blocks` | 新增 | 只对新发布帖写入;老帖保持 NULL |
| 评论 `content` | 不动 | 评论富文本留 P2 |
