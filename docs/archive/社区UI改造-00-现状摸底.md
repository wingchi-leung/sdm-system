### 1.5 当前 wxss 设计风格特征

主要色值与变量集中在 `miniprogram/app.wxss:1-104`,社区页的样式在此基础上做了叠加。设计系统采用"极简奢华"基调,绝大多数色值为黑/灰/白,辅以少量语义色。

#### Design Tokens 健康度

- 有,统一在 `miniprogram/app.wxss:1-104`(`page { ... }` 块)声明 50+ CSS 自定义属性。
- 但社区几个子页面**未完全消费变量**:

## 3. 文档

### 3.1 `docs/specs/社区频道功能-spec.md`

- 576 行规格说明书,分 10 章:1 概述、2 数据库设计、3 后端 API、4 小程序页面结构、5 业务流程、6 权限设计、7 未来扩展预留、8 技术实现计划、9 注意事项、10 通知中心规范(摘要在 `spec.md:1-10`)。
- 关键要点:
  - **数据模型**(spec §2.1,`spec.md:25-122`):列出 5 张表 `community_channel`、`community_channel_member`、`community_channel_post`、`community_channel_comment`、`community_notification`,与落地 `table.sql` 存在表名差异(spec 用 `_channel_post/_comment`,实际复用老的 `community_post/comment`)。
  - **API**(spec §3,`spec.md:141-197`):列了 5 大类(频道管理 / 成员管理 / 动态管理 / 评论管理 / 通知)共 ~30 个端点,实际实现见 §2.1 基本对齐;但 `PUT/DELETE /channels/{id}`、`/posts/{id}` DELETE、`/comments/{id}` DELETE、`pin / official / transfer-admin` 这几个 spec 里写出的接口在 `community.py` 路由表里**没找到**。
  - **页面**(spec §4.2,`spec.md:209-249`):规划 7 个页面 `community/index`、`community/channel-create`、`community/channel-detail`、`community/post-create`、`community/post-detail`、`community/channel-manage` + 复用站内信;实际只实现了 6 个(没有独立的 channel-detail 和 channel-manage 页面,频道信息直接铺在 `community-post-list` 顶部)。
  - **业务流程**(spec §5,`spec.md:251-`):创建/邀请/接受/踢禁言/发布/评论 6 步,落地基本对应。

## 5. 现状问题观察(改进点)

以下每条都附 `file_path:line_number` 证据:

1. **「社区」Tab 视觉与活动 Tab 完全相同,缺乏辨识度。** 三个 Tab 中只有社区 Tab 复用了活动的 `activity-default.png / activity-active.png` 图标,而中间项本应有自己的 channel 图标——`D:\sdm-system 2\sdm-system\miniprogram\app.json:50-52` 与 `D:\sdm-system 2\sdm-system\miniprogram\custom-tab-bar\index.js:10-13` 路径相同;`miniprogram\assets\icons\` 里也没有 `channel.png` 之外的对应图标(目录里只有 `activity-* / mine-* / bell / location / setting / time`)。

2. **`--green` 设计 token 被覆写成黑色,链接颜色与文字同色,失去提示作用。** `D:\sdm-system 2\sdm-system\miniprogram\app.wxss:103` 声明 `--green: #000000;`,导致 `D:\sdm-system 2\sdm-system\miniprogram\pages\community-post-list\community-post-list.wxss:107` 的 `.post-link { color: var(--green); }` 渲染为纯黑,在白底卡片里几乎不可见。建议要么修正 token 值,要么单独定义链接色。

3. **「内容审核」页面风格与社区其他页完全割裂,几乎全是硬编码色值。** `D:\sdm-system 2\sdm-system\miniprogram\pages\community-moderation\community-moderation.wxss:1-85` 全文用 `#fff/#111/#888/#f7f8fa/#eceef2/#12b76a/#e5484d`,**一个 `var(--*)` 都没有用**,而其他社区页(hero/通知/列表)都用 `var(--ink)` / `var(--form-border)` / 阴影令牌;同时该页 4 个 group 内的内容只显示 ID、用户名、纯文本,没有和后端 `CommunityChannelPostResponse.is_official / is_pinned` 等字段联动,管理员无法直观区分官方/置顶。

4. **站内信入口有两套视觉(组件 vs 内联),铃铛参数和位置不可控。** 首页活动页用内联 `<image class="hero-bell">`(`D:\sdm-system 2\sdm-system\miniprogram\pages\index\index.wxml:7-14`),而 mine 页用 `<floating-bell>` 组件(`D:\sdm-system 2\sdm-system\miniprogram\pages\mine\mine.wxml:8`),两个入口视觉/点击区不一致;`D:\sdm-system 2\sdm-system\miniprogram\components\floating-bell\floating-bell.js:8-22` 通过 inline `styleText` 注入定位,使得样式无法被主题/响应式统一管理。

5. **频道创建交互是 `wx.showModal + editable` 原生弹窗,只支持名称,没有描述/头像/简介预览。** `D:\sdm-system 2\sdm-system\miniprogram\pages\community\index\index.js:62-89` 的 `onCreateChannel` 直接用 `wx.showModal({ editable: true, placeholderText: '请输入频道名称' })`,但 spec §4.2.2 (`D:\sdm-system 2\sdm-system\docs\specs\社区频道功能-spec.md:218-220`) 明确要求"频道名称 + 描述 + 频道头像"三要素;后端 `CommunityChannelCreate` (`D:\sdm-system 2\sdm-system\backend\app\models\community.py:135-164`) 也支持 description/avatar_url——目前前端完全没用到,建议改造为独立 `channel-create` 页面。

6. **Spec 与落地的「表名差异」容易让接手的人困惑。** spec (`D:\sdm-system 2\sdm-system\docs\specs\社区频道功能-spec.md:62-101`) 规划 `community_channel_post` / `community_channel_comment` / `community_notification` 三张表,但实际 `D:\sdm-system 2\sdm-system\backend\sql\table.sql:416-487` 落地的只有 `community_channel`、`community_channel_member`、`community_post`、`community_comment`、`community_media_moderation_task`,且 `community_post` 在表结构里没有 `channel_id` 字段(spec §2.1.3 要求有)——`get_posts_by_activity` (`D:\sdm-system 2\sdm-system\backend\app\crud\crud_community_post.py:53-100`) 仍以 `activity_id` 过滤,导致后端代码、文档、SQL 三方语义不一致;接手人需要重新读 CRUD 才能确认「频道动态」到底走哪条路径。
