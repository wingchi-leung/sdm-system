# 配置与备份梳理

> 更新时间：2026-05-09
> 适用范围：旧电脑正式部署、Docker 部署、本地开发联调

## 一、先说结论

当前项目里配置文件较多，但正式使用时建议只记住下面这套分工：

| 场景 | 唯一主入口 | 说明 |
|------|------------|------|
| **旧电脑正式环境（Docker）** | 根目录 `.env` | 给 `docker-compose.yml`、`backend` 容器、`frontend` 容器使用 |
| **后端本地直接运行** | `backend/.env` | 只在 `cd backend && uv run uvicorn ...` 时使用 |
| **小程序接口切换** | `miniprogram/config/index.js` | 控制开发版/体验版/正式版请求哪个 API 域名 |
| **Cloudflare Tunnel** | `tunnel/config.yml` + `tunnel/certs/*.json` | 负责公网入口，不走 `.env` |
| **微信支付证书** | `secrets/wechat_pay/` | Docker 后端容器挂载使用，不应写宿主机绝对路径 |

一句话版：

- **正式环境只维护根目录 `.env`**
- **本地调试 backend 才维护 `backend/.env`**
- **小程序不放密钥，只放域名和环境切换逻辑**

## 二、当前配置文件分工

### 1. 根目录 `.env`

用途：

- `docker-compose.yml` 读取
- Docker 内 `backend` 服务读取
- Docker 内 `frontend` 构建时读取

当前应承担的正式配置：

- MySQL 账号与库名
- JWT 密钥
- 后端安全开关
- API / 静态资源公网地址
- CORS 白名单
- 微信小程序配置
- 微信支付配置
- 本地上传存储配置

注意：

- 这是**旧电脑正式环境的唯一配置源**
- 不要再把正式值维护到 `backend/.env`

### 2. 根目录 `.env.example`

用途：

- 给新机器部署、同事接手、未来迁移时当模板

本次已补齐：

- `STATIC_BASE_URL`
- `DB_ECHO`
- `REQUIRE_HTTPS_FOR_LOGIN`
- 登录限流配置
- `LOCAL_UPLOAD_DIR`
- Docker 场景下的微信支付证书路径

同时已明确：

- 当前 compose **不使用** `TUNNEL_TOKEN`
- Tunnel 依赖的是 `tunnel/config.yml` 和 `tunnel/certs/`

### 3. `backend/.env`

用途：

- 仅供本地直接启动 backend 使用

适用命令：

```powershell
cd backend
uv run uvicorn app.main:app --host 0.0.0.0 --port 8000
```

不适用场景：

- Docker 正式部署
- 旧电脑正式环境

建议：

- 正式环境不要再修改这个文件
- 里面如果还是宿主机绝对路径证书，只保留给本地联调即可

### 4. `backend/.env.example`

用途：

- 说明 backend 本地开发直跑时需要什么变量

和根目录 `.env.example` 的区别：

- 根目录的是 **Docker/正式环境模板**
- backend 目录的是 **本地开发模板**

### 5. `miniprogram/config/index.js`

用途：

- 统一决定小程序当前环境使用哪个 `baseUrl`
- 不存储任何密钥

当前逻辑：

- `release` 固定走生产域名
- 开发者工具可切换本地/远端
- 真机开发版默认走远端 HTTPS

上线前要确认：

- `PROD_HOST`
- `TUNNEL_HOST`
- `tenantCode` 默认值是否符合正式租户策略

### 6. `tunnel/config.yml`

用途：

- Cloudflare Tunnel 容器内使用

当前作用：

- `api.chronono.org -> backend:8000`
- `web.chronono.org -> frontend:3000`

注意：

- 这里是公网入口配置
- 它和根目录 `.env` 是两条不同配置链路
- 域名改了以后，要同时改：
  - 根目录 `.env`
  - `tunnel/config.yml`
  - 小程序 `miniprogram/config/index.js`
  - 微信公众平台合法域名
  - 微信支付回调地址

### 7. `secrets/wechat_pay/`

用途：

- 存放 Docker 场景下的微信支付私钥和微信支付公钥

当前建议：

- 正式环境统一只认这里
- 不再依赖 `C:\Users\xxx\...` 这类宿主机绝对路径

## 三、推荐的正式环境配置原则

### 原则 1：正式环境只有一个主配置入口

旧电脑正式运行时，以根目录 `.env` 为准。

### 原则 2：代码里允许有默认值，但正式值必须显式配置

尤其是：

- `JWT_SECRET`
- `CORS_ORIGINS`
- `API_BASE_URL`
- `STATIC_BASE_URL`
- `WECHAT_PAY_NOTIFY_URL`

### 原则 3：密钥与证书分开管理

- 文本型密钥：放 `.env`
- 文件型证书：放 `secrets/wechat_pay/`

### 原则 4：域名相关值要成组变更

改正式域名时，至少同步检查这 5 处：

1. 根目录 `.env`
2. `tunnel/config.yml`
3. `miniprogram/config/index.js`
4. 微信公众平台合法域名
5. 微信支付回调地址

## 四、建议保留/废弃策略

### 建议保留

- 根目录 `.env.example`
- `backend/.env.example`
- `tunnel/config.yml`
- `miniprogram/config/index.js`
- `secrets/wechat_pay/README.md`

### 建议继续使用，但只限开发

- `backend/.env`

### 建议不要再作为正式配置来源

- 宿主机绝对路径形式的微信支付证书路径
- 文档里出现但 compose 实际不消费的 `TUNNEL_TOKEN`

---

## 五、备份范围定义

旧电脑正式环境至少要备份下面这些内容：

| 类别 | 路径/对象 | 是否必须 |
|------|-----------|----------|
| 数据库 | Docker MySQL 数据 | **必须** |
| 上传文件 | `backend/uploads/` | **必须** |
| 支付证书 | `secrets/wechat_pay/` | **必须** |
| 正式配置 | 根目录 `.env` | **必须** |
| Tunnel 凭据 | `tunnel/config.yml`、`tunnel/certs/` | **建议** |
| 部署编排 | `docker-compose.yml` | **建议** |

不建议备份的内容：

- `node_modules/`
- `frontend/build/`
- `backend/.venv/`
- `htmlcov/`
- `.pytest_cache/`

---

## 六、备份演练的最低标准

不是“文件拷出来了”就算完成，至少要做到下面这套：

1. 执行一次完整备份
2. 在另一个空目录解压
3. 恢复数据库
4. 恢复 `.env`、上传目录、支付证书、Tunnel 配置
5. 执行 `docker compose up -d`
6. 验证：
   - `https://api.../docs` 可访问
   - `https://web...` 可访问
   - 小程序登录/报名/支付查询可用

只要这套没做过，就不能算“备份可靠”。

---

## 七、推荐演练频率

- 每天自动备份一次
- 每周至少手动抽查一次备份文件是否生成
- 每月至少做一次恢复演练
- 每次修改支付证书、域名、MySQL 密码后，立刻补做一次备份

---

## 八、落地建议

当前最推荐的维护方式：

1. 正式环境配置统一只维护根目录 `.env`
2. 本地调试 backend 才使用 `backend/.env`
3. Tunnel 单独维护 `tunnel/config.yml + tunnel/certs`
4. 微信支付证书统一放 `secrets/wechat_pay/`
5. 用脚本自动备份数据库、上传目录、证书和配置

下一步可以直接执行：

- 用 `scripts/backup-production.ps1` 做一次备份
- 再按文档做一次恢复演练
