# 权限系统重构方案（RBAC）

## 1. 数据模型

### 核心表结构

```sql
-- 权限表（系统预定义）
CREATE TABLE permission (
  id INT PRIMARY KEY AUTO_INCREMENT,
  code VARCHAR(64) UNIQUE NOT NULL,  -- 'activity.create', 'activity.edit'
  name VARCHAR(100) NOT NULL,         -- '创建活动', '编辑活动'
  resource VARCHAR(32) NOT NULL,      -- 'activity', 'user', 'report'
  action VARCHAR(32) NOT NULL,        -- 'create', 'edit', 'delete', 'view'
  description TEXT,
  create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
  update_time DATETIME
);

-- 角色表
CREATE TABLE role (
  id INT PRIMARY KEY AUTO_INCREMENT,
  tenant_id INT NOT NULL,             -- 0 表示平台级角色
  name VARCHAR(64) NOT NULL,
  is_system TINYINT NOT NULL DEFAULT 0,  -- 1=系统预设，0=自定义
  description TEXT,
  create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
  update_time DATETIME,
  UNIQUE KEY uk_tenant_role_name (tenant_id, name)
);

-- 角色-权限关联表
CREATE TABLE role_permission (
  id INT PRIMARY KEY AUTO_INCREMENT,
  role_id INT NOT NULL,
  permission_id INT NOT NULL,
  create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
  update_time DATETIME,
  UNIQUE KEY uk_role_permission (role_id, permission_id)
);

-- 用户-角色关联表
CREATE TABLE user_role (
  id INT PRIMARY KEY AUTO_INCREMENT,
  user_id INT NOT NULL,
  role_id INT NOT NULL,
  tenant_id INT NOT NULL,
  scope_type VARCHAR(32),             -- NULL=全局, 'activity_type', 'activity'
  scope_id INT,                       -- 活动类型ID 或 活动ID
  create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
  update_time DATETIME,
  UNIQUE KEY uk_user_role_scope (user_id, role_id, tenant_id, scope_type, scope_id)
);

-- 管理员账号表（仅用于登录认证）
CREATE TABLE admin_user (
  id INT PRIMARY KEY AUTO_INCREMENT,
  tenant_id INT NOT NULL,
  user_id INT NOT NULL,               -- 关联 user 表的 ID
  username VARCHAR(64) NOT NULL,      -- 用户名（手机号）
  password_hash VARCHAR(255) NOT NULL DEFAULT '',
  must_reset_password INT NOT NULL DEFAULT 1,  -- 1=需要改密，0=已改密
  create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
  update_time DATETIME
);
```

## 4. 权限检查逻辑

### 三级权限范围

| scope_type | scope_id | 含义 |
|------------|----------|------|
| NULL | NULL | 全局权限（超级管理员） |
| `activity_type` | 活动类型ID | 该活动类型下的所有活动 |
| `activity` | 活动ID | 单个活动的权限 |

### 代码实现（`crud_rbac.has_permission`）

```python

    user_id: int,

    tenant_id: int,
    resource_id: Optional[int] = None,
    resource_type: Optional[str] = None  # 'activity_type' or 'activity'
) -> bool:
    # 1. 查询用户的所有角色授权
    # 2. 检查是否有全局权限（scope_type is None）→ 直接通过
    # 3. 检查活动类型范围：scope_type='activity_type' 且 resource_type='activity_type' 且 scope_id == resource_id
    # 4. 检查单活动范围：scope_type='activity' 且 resource_type='activity' 且 scope_id == resource_id
```

## 5. 用户使用流程

### 场景：分配一个"参活动类型管理员"

**Step 1：超级管理员登录 Web 后台**

**Step 2：进入"权限与管理员"页面**

**Step 3：分配角色**
- 选择用户（例如：李四，手机号 13800138000）
- 选择角色（例如："参活动管理员"）
- 权限范围选择"活动类型" → 选择"参"
- 确认分配

**Step 4：系统自动创建 admin_user 记录**
- `username` = 用户手机号
- `password_hash` = 123456（bcrypt 加密）
- `must_reset_password` = 1（需要改密）

**Step 5：李四收到通知，用手机号 + 123456 登录**

**Step 6：登录成功后强制跳转到改密页面**
- 设置新密码（至少6位）
- 改密成功后 `must_reset_password` = 0

**Step 7：跳转登录页，用新密码重新登录**
- 以后可用新密码从 Web 或小程序管理员入口登录

### 超级管理员重置用户密码

在"权限与管理员"页面，找到对应用户，点击"重置密码"按钮，可强制将密码重置为 123456（同时 `must_reset_password` 恢复为 1）。

## 6. Web 与小程序管理员体系

### 两套独立的认证体系

| 入口 | 登录方式 | 表 | token role | 说明 |
|------|---------|-----|-----------|------|
| Web 租户管理员 | 用户名 + 密码 | `admin_user` | `"admin"` | 通过 `auth/login` 接口 |
| Web 平台管理员 | 用户名 + 密码 | `platform_admin` | `"platform_admin"` | 通过 `auth/platform-login` 接口 |
| 小程序普通用户 | 微信 openid | `user` | `"user"` | 通过 `auth/wechat-login` 接口 |
| 小程序管理员 | 手机号 + 密码 | `admin_user` | `"admin"` | 通过 `auth/login` 接口 |

### 关键逻辑

- **分配角色时自动创建 admin_user**：当给一个用户分配角色时，系统自动在 `admin_user` 表创建一条记录（用户名=手机号，密码=123456）
- **登录时检查角色**：即使 `admin_user` 认证通过，如果 `user_role` 表没有该用户的记录，也会拒绝登录
- **小程序不受改密流程影响**：小程序管理员登录走的是同一套 `admin_user` 表，密码通用

## 7. API 接口

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/v1/auth/login` | POST | 管理员登录，返回 `must_reset_password` |
| `/api/v1/auth/set-admin-password` | POST | 设置/修改管理员密码（需先登录） |
| `/api/v1/roles/permissions` | GET | 获取所有权限列表 |
| `/api/v1/roles/roles` | GET | 获取所有角色（含权限） |
| `/api/v1/roles/user-roles` | POST | 分配角色（同时自动创建 admin_user） |
| `/api/v1/roles/user-roles/{id}` | DELETE | 移除用户角色 |
| `/api/v1/roles/users/{user_id}/roles` | GET | 获取用户的角色列表 |
| `/api/v1/activity-types` | GET | 获取所有活动类型（用于权限分配范围选择） |

## 8. 迁移说明

### 历史迁移（已完成）

- `admin_user.is_super_admin=1` → `user_role` 表中 `role_id=1`（超级管理员）
- `admin_activity_type_role` → `user_role` 表 `scope_type='activity_type'`

### 字段变更记录（`field.sql`）

```sql
-- admin_user 表新增是否需要改密标志
ALTER TABLE admin_user ADD COLUMN must_reset_password INT NOT NULL DEFAULT 1 COMMENT '1=需要改密，0=已改密' AFTER password_hash;
```

## 9. 优势

- **灵活扩展**：新增权限只需插入数据，不改代码
- **细粒度控制**：可以精确到单个活动或活动类型
- **商业成熟**：RBAC 是行业标准
- **支持活动类型范围**：通过 `scope_type='activity_type'` 实现
- **强制改密机制**：新分配的管理员必须设置自定义密码
- **Web/小程序共用密码**：同一套 `admin_user` 表，两端通用
