# SDM 后端 FastAPI 项目 —— 讲解笔记

## 一、项目一句话介绍

这是一个**单 FastAPI 主应用 + 按业务域拆分的路由模块**的后端系统，核心面向**多租户活动管理、用户与认证、RBAC 权限、活动报名/签到、微信支付、社区内容、通知与上传**这些场景。  
整体不是把服务拆成很多微服务，而是一个结构清晰的**模块化单体**：入口统一、路由分层、CRUD 解耦、业务域清楚，比较适合快速迭代和中小规模系统落地。

技术栈：Python + FastAPI + SQLAlchemy + Pydantic + MySQL/关系型数据库 + 微信生态接口 + 定时任务调度 + 本地/云存储抽象。

---

## 二、整体结构图

可以把后端理解成下面这条链路：

```text
前端 / 小程序 / 管理端
        ↓
FastAPI 主应用 app（统一入口、CORS、中间件、静态资源、生命周期管理）
        ↓
api/v1/router.py（把业务路由按模块挂载）
        ↓
各业务 endpoint（auth / users / activities / payments / community ...）
        ↓
crud 层（封装数据库读写、权限判断、业务聚合）
        ↓
schemas / models（请求响应模型 + ORM 数据模型）
        ↓
数据库 / 微信支付 / 微信审核 / 存储服务 / 定时任务
```

如果面试官追问“你们后端有几个 FastAPI 应用”，可以直接回答：

> 严格来说只有 1 个 FastAPI 主应用，`backend/app/main.py` 负责初始化；下面是 14 个业务路由模块，通过 `APIRouter` 组合起来，不是多个独立 app。

## 三、核心模块速览

| 文件 | 作用 |
|---|---|
| `backend/app/main.py` | FastAPI 主入口，负责生命周期、RBAC 初始化、定时任务、CORS、请求体大小限制、上传静态资源挂载 |
| `backend/app/api/v1/router.py` | 聚合所有 API 路由，是整个后端接口的总入口 |
| `backend/app/api/v1/endpoints/auth.py` | 登录、退出、微信登录、登录态快照、设置密码 |
| `backend/app/api/v1/endpoints/users.py` | 用户注册、个人资料、头像、绑定信息、管理端用户管理、导入导出、封禁解封 |
| `backend/app/api/v1/endpoints/activities.py` | 活动创建、列表、详情、编辑、删除、状态、统计、导出 |
| `backend/app/api/v1/endpoints/participants.py` | 活动报名、报名列表、我的活动、报名审核 |
| `backend/app/api/v1/endpoints/checkins.py` | 签到记录列表与签到动作 |
| `backend/app/api/v1/endpoints/payments.py` | 微信支付下单、支付回调、订单查询、退款、退款回调 |
| `backend/app/api/v1/endpoints/roles.py` | 权限、角色、用户角色分配与回收 |
| `backend/app/api/v1/endpoints/tenants.py` | 租户管理、租户列表、租户统计 |
| `backend/app/api/v1/endpoints/activity_types.py` | 活动类型管理、可用类型查询 |
| `backend/app/api/v1/endpoints/user_activity_types.py` | 用户与活动类型的绑定关系 |
| `backend/app/api/v1/endpoints/community.py` | 社区频道、帖子、评论、公告、日历、审核、邀请、消息、头像上传 |
| `backend/app/api/v1/endpoints/notifications.py` | 订阅消息、任务重试、退款结果入队 |
| `backend/app/api/v1/endpoints/uploads.py` | 海报、头像、社区图片等文件上传 |
| `backend/app/api/v1/endpoints/realname_auth.py` | 实名认证校验 |

> 这里面最值得面试官注意的，不是“路由有多少”，而是它们是按**业务域**拆的：认证、用户、活动、支付、权限、社区、通知，各自边界都比较清楚。

## 四、主应用做了什么

`main.py` 不是简单起个服务，它做了很多生产化的基础设施工作：

1. **生命周期管理**
   - 启动时先校准 RBAC 基础数据
   - 再启动支付订单相关的定时任务
   - 退出时停止任务，避免后台线程残留

2. **中间件**
   - 请求体大小限制：防止大包请求拖垮服务
   - 上传缓存控制：`/uploads/` 静态资源加长缓存，减少重复加载
   - CORS：允许前端和小程序跨域访问

3. **静态资源挂载**
   - 本地存储模式下，直接把 `/uploads` 挂成静态目录
   - 云存储模式下则由外部 URL 提供文件访问

4. **日志与安全**
   - 安装敏感信息过滤器，避免日志里泄露手机号、姓名等信息

这一块你可以总结成一句面试表达：

> 我们把 FastAPI 主入口当作应用运行时的“控制中心”，它不仅负责路由挂载，还统一处理启动初始化、安全限制、静态资源、跨域和后台任务。

## 五、业务链路怎么串起来

### 1. 认证链路

`auth` 模块是整个系统的入口。

- 支持管理员和普通用户登录
- 支持微信小程序登录
- 登录态会返回权限快照
- 还做了登录限流、HTTPS 校验、密码长度限制
- 登录后通过 JWT / Cookie 维持会话

面试官可以听懂的说法：

> 认证层不是单纯做登录，而是把“身份、租户、权限快照”一起打包回来，这样前端进入页面后能快速判断能看到什么、能做什么。

### 2. 用户链路

`users` 模块覆盖了用户全生命周期：

- 注册
- 查询当前用户信息
- 创建用户
- 绑定手机号/身份信息
- 修改头像、个人资料
- 删除资料字段、注销账号
- 后台用户列表、封禁/解封
- 导入模板、Excel 导入

这里比较工程化的点：

- 对历史脏数据做了响应兜底，避免某些字段异常把整页接口打崩
- 管理端和普通端的用户展示策略不同，管理端有更完整的列表和脱敏/非脱敏版本

### 3. 活动链路

`activities` 是系统主业务。

- 管理员创建活动
- 用户或管理员按视角查看活动列表
- 活动可导出报名数据
- 支持未开始活动筛选
- 活动详情、编辑、删除、状态切换、统计、签到信息都在这里

你可以把它概括成：

> 活动模块负责“活动的生命周期管理”，从创建、展示、报名入口，到状态、统计和导出，基本把运营工作闭环了。

### 4. 报名 / 签到链路

`participants` 和 `checkins` 组成活动执行链路：

- 报名时创建参与记录
- 支持查看某个活动的报名人员
- 支持查看我的活动
- 报名后可以签到

这个设计很适合面试时说：

> 报名和签到拆开，是因为它们在业务上是两个阶段：报名是参与意愿，签到是现场履约，状态也需要分开管理。

### 5. 支付链路

`payments` 负责微信支付全流程：

- 创建支付订单
- 拉起支付参数
- 支付回调通知
- 查询订单详情
- 删除待支付订单
- 退款和退款回调

支付模块里能说的工程亮点：

- 会先校验活动是否可报名
- 会校验用户是否被拉黑
- 会处理支付失败后的补偿
- 会尽量关闭本地落单失败时留下的微信远端订单

这类描述非常适合面试官：

> 我们不是“接个支付接口就完了”，而是考虑了幂等、补偿、失败兜底和订单状态收敛，确保支付链路能闭环。

### 6. 权限链路

`roles`、`tenants`、`user_activity_types` 以及 `crud_rbac` 共同构成权限体系。

- 角色和权限可查
- 用户角色可分配、回收
- 支持按租户隔离
- 支持活动类型级别的授权
- 支持平台管理员和租户管理员的分层

这部分你可以这样总结：

> 你们的权限不是单一的“管理员/普通用户”二元模型，而是带租户、带作用域、带资源范围的 RBAC，能做到“谁在什么租户、对什么资源、能做什么”。

### 7. 社区链路

`community` 是相对完整的一块：

- 频道管理
- 发帖 / 评论
- 邀请 / 加入
- 审核 / 内容安全
- 公告
- 日历事件
- 头像上传
- 未读消息和已读处理

它的特点是：

- 不是单一的发帖系统，而是频道型社区
- 内容发布前会结合微信内容安全能力
- 频道和活动之间有结合点，适合活动配套社区

### 8. 通知 / 上传 / 实名

- `notifications` 负责订阅消息、任务重试、退款结果消息入队
- `uploads` 负责海报、头像、社区图片
- `realname_auth` 负责实名校验

这几个模块虽然看起来是配角，但对完整业务闭环很重要。

## 六、几个面试官常问的关键点

### 1. 你们为什么用单体而不是微服务？

可以这样答：

> 这是一个业务边界比较清晰的管理系统，核心诉求是快速迭代和稳定交付，所以采用模块化单体更合适。FastAPI 负责统一入口，按业务域拆路由和 CRUD 层，既保留了结构化边界，也避免了微服务拆分带来的运维和调用复杂度。

### 2. 你们怎么做代码分层？

可以这样答：

> 我们按 `API 路由层 -> CRUD 层 -> ORM 模型 / Pydantic 模型` 分层。路由层只处理请求和响应，CRUD 层专注数据库操作和业务规则，模型层负责统一数据结构，这样可维护性更好，也方便测试。

### 3. 你们的权限模型有什么特点？

可以这样答：

> 我们不是简单的角色判断，而是 RBAC + 租户 + 资源作用域的组合。比如同一个管理员，在不同租户、不同活动类型下面能看到的资源不一样，这样才能支撑多租户和细粒度授权。

### 4. 生产环境上做了哪些保护？

可以这样答：

> 主要做了几类：登录限流、HTTPS 强制、请求体大小限制、日志脱敏、上传缓存控制、后台任务生命周期管理。虽然每一项都不复杂，但组合起来能明显提升稳定性和安全性。

### 5. 支付怎么保证可靠？

可以这样答：

> 支付链路里我们会做业务前置校验、订单状态控制、微信回调校验和失败补偿。核心目标不是“能支付”，而是“支付过程能对账、能恢复、能收敛状态”。

### 6. 社区模块为什么值得讲？

可以这样答：

> 因为它不只是 CRUD，而是把频道、帖子、评论、公告、日历、审核、邀请和消息串成了一个完整互动域，还接了内容安全和微信生态，比较能体现工程复杂度。

“我们后端是一个 FastAPI 模块化单体架构，主应用统一负责生命周期、CORS、静态文件、请求体限制和后台任务，业务上再按认证、用户、活动、报名签到、支付、权限、社区、通知等域拆成多个 `APIRouter`。系统支持多租户和细粒度 RBAC 权限，活动报名链路接了微信支付和支付回调，社区模块还做了频道、帖子、评论、公告、日历和内容安全。整体上我更强调的是工程化能力：分层清晰、权限可控、支付有补偿、日志有脱敏、上传和静态资源也做了运行时优化。”

## 八、如果面试官继续追问，你可以往下展开的点

1. **为什么活动和报名要分模块**
   - 活动是配置和发布，报名是用户行为，状态机不同，职责也不同。

2. **为什么 `main.py` 里要做生命周期初始化**
   - 因为 RBAC 基础权限、后台任务、静态挂载都属于应用级基础设施，不应该散落在业务路由里。

3. **为什么社区模块会这么大**
   - 因为它同时包含内容流、评论、审核、通知、日历和成员管理，本质上是一个独立业务域。

4. **为什么要做上传缓存**
   - 上传文件通常被重复访问，缓存头能减少带宽和延迟，尤其是海报、头像这种静态资源。

5. **你觉得系统还有什么可以继续优化**
   - 路由可以继续细分
   - 关键业务可以补更完整的单测
   - 支付和社区审核可以进一步补监控与审计日志

## 九、术语速查

- **FastAPI 主应用**：`backend/app/main.py` 里真正启动的 `app`
- **APIRouter**：把不同业务接口按模块挂载的路由容器
- **CRUD 层**：封装数据库操作，避免 SQL 和业务逻辑散在路由里
- **RBAC**：基于角色的权限控制，这里还结合了租户和资源范围
- **多租户**：同一套系统服务多个租户，数据按 `tenant_id` 隔离
- **补偿逻辑**：主链路失败后，用额外动作把状态修回来
- **内容安全**：对帖子、评论、图片等做审核或风控
- **静态挂载**：把上传目录直接当成静态资源对外提供

## 十、最后给你的面试建议

如果你要跟面试官讲这个项目，建议按这个顺序：

1. 先说是**模块化单体**
2. 再说**多租户 + RBAC + 活动业务**
3. 然后说**微信支付、社区、通知**
4. 最后补**工程化细节：生命周期、限流、脱敏、缓存、补偿**

这样讲的好处是：

- 结构清楚
- 不会显得只是在背接口
- 能体现你对系统设计和工程落地的理解
