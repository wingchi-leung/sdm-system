# SDM 系统 — 产品规格说明（Spec）

## 一、产品愿景与目标

**愿景**：打造一款 **All in One** 的平台型应用，承载平台的各类学习活动。

**核心目标**：
- 学习活动的全生命周期管理
- 平台人员与权限的清晰管理
- 支持多种登录方式与微信生态
- 预留社区能力与多租户扩展
- 数据私有化、部署灵活、UI 现代、可上架应用商店

### 2.8 管理员分级与活动维度权限

#### 需求简述

- **超级管理员**：拥有系统全部管理权限（活动 CRUD、报名/签到管理、用户与黑名单、其他管理员维护等）。
- **活动管理员**：按**活动类型/大类**划分，仅能管理该类型下的所有活动（含创建、编辑、删除、查看报名与签到）。例如：
  - 「参」是一类活动，可能有多个具体活动实例（如每月几场）；参活动管理员可管理所有属于「参」类的活动的发布、编辑、删除、查看报名与签到，并可**创建**新的参类活动；
  - 「健康锻炼」是另一类，包含多场具体活动；健康锻炼管理员仅能管理健康锻炼大类下的全部活动，并可创建新的健康锻炼活动；
  - 参管理员不能操作健康锻炼类活动，健康锻炼管理员也不能操作参类活动，彼此按类型隔离。

授权粒度是**活动类型**（一类），不是单个活动实例：一名活动管理员可被授权多种类型；一种类型下可有多名活动管理员。

#### 实现思路

| 维度 | 说明 |
|------|------|
| **角色区分** | 在管理员身份上区分两类：超级管理员、活动管理员。推荐在 `admin_user` 表增加字段标识，例如 `is_super_admin`（1=超级管理员，0=活动管理员）。 |
| **活动类型与授权** | 引入**活动类型/大类**（如「参」「健康锻炼」）：每个具体活动实例归属某一类型。新增**管理员-活动类型关联表**（如 `admin_activity_type_role`），字段：`admin_user_id`、`activity_type_id`。表示该管理员被授权管理该**类型**下的所有活动；可为一管理员配置多条记录以绑定多种类型。 |
| **权限校验规则** | 先校验为管理员；再按操作区分：<br>• **全局类操作**（如：管理其他管理员、系统配置）：仅 `is_super_admin=1` 可执行。<br>• **创建活动**：超级管理员可创建任意类型活动；活动管理员只能选择其有权限的**活动类型**并创建该类型下的活动（创建时校验 `activity_type_id` 是否在其授权范围内）。<br>• **与某活动实例相关的操作**（编辑/删除/状态变更、查看该活动报名与签到）：`is_super_admin=1` 或 该活动所属的 `activity_type_id` 在管理员的 `admin_activity_type_role` 中 则可执行。 |
| **列表与可见范围** | 管理端「活动列表」：超级管理员看全部活动；活动管理员仅看到其有权限的**活动类型**下的全部活动实例。报名列表、签到等按「该活动是否属于其有权限的类型」过滤。 |

#### 数据模型（建议）

- **admin_user 表**：增加 `is_super_admin` tinyint，默认 0；1 表示超级管理员。
- **活动类型表**（如 `activity_type`，若当前无「类型」概念则新建）：  
  `id`、`type_name`（如「参」「健康锻炼」）、可选 `code` 等。用于区分活动大类，一个类型下对应多场具体活动。
- **activity 表**：增加 `activity_type_id` 外键关联 `activity_type.id`，表示该活动实例归属哪一大类。
- **admin_activity_type_role 表**（新建）：  
  `id`、`admin_user_id`、`activity_type_id`，唯一约束 `(admin_user_id, activity_type_id)`，外键关联 `admin_user.id` 与 `activity_type.id`。表示该管理员被授权管理该**类型**下的所有活动（含创建、编辑、删除、查看报名与签到）。

若后续需要更细粒度（如仅「查看报名」与「编辑活动」分离），可在该表增加权限位字段；首版可简化为「有记录即拥有该类型的全部管理权限」。

#### 接口与依赖

- 现有 `get_current_admin` 仍返回 `admin_user.id`；建议新增依赖如 `require_super_admin`（仅超级管理员通过）及 `require_activity_type_admin(activity_type_id)` / `require_activity_admin(activity_id)`（后者通过活动的 `activity_type_id` 判断：超级管理员或拥有该类型的活动管理员通过）。创建活动时：若为活动管理员，请求体须带 `activity_type_id` 且须在其授权类型内。
- 管理端：活动列表、报名/签到入口按当前管理员身份与「有权限的活动类型」过滤（只展示这些类型下的活动），避免越权操作。

#### 小结与状态

| 规格项 | 描述 | 当前实现状态 |
|--------|------|--------------|
| 超级管理员 | 拥有全部管理权限 | ⚠️ 部分实现（后端可通过全局 RBAC 角色实现；管理端配置流仍待补） |
| 活动类型（大类） | 活动归属类型，如「参」「健康锻炼」；一类下多场活动实例 | ✅ 已实现（`activity_type` + `activity.activity_type_id` 已存在并使用） |
| 活动管理员 | 按类型授权，可创建/编辑/删除/查看报名与签到（仅限其负责的类型下的活动） | ⚠️ 部分实现（后端权限链路已支持；Web 已补角色分配与 scope 展示第一页，活动类型选择器仍待补） |
| 管理员-活动类型授权 | 为活动管理员绑定可管理的活动类型 | ⚠️ 部分实现（底层 RBAC scope 已支持 `activity_type` 范围；Web 已支持手工录入 scope_id 的角色分配，仍缺完整选择器和管理员专属视图） |

已实现部分：活动类型表与 `activity.activity_type_id`、RBAC 的 `activity_type` scope、创建/编辑/删除/查看报名等接口的范围校验、按权限过滤活动列表。待补部分：管理员授权管理界面、更多权限模板与审计能力。

### 2.9 报名、支付与小程序安全兜底规则

本轮代码审查后，以下规则已从“前端约定/最佳路径”调整为“后端兜底 + 小程序显式传参”：

| 规则项 | 当前实现状态 | 说明 |
|--------|--------------|------|
| **公开活动租户解析** | ✅ 已实现 | 未登录访问活动列表、未开始活动、活动详情、报名信息时，小程序传 `tenant_code`，后端通过统一公开租户上下文解析租户；不再硬编码读取 1 号租户。 |
| **已登录租户状态校验** | ✅ 已实现 | JWT 中携带 `tenant_id`，后端每次解析登录态时都会校验租户存在、启用且未过期；租户禁用后旧 token 不能继续访问。 |
| **RBAC 租户边界** | ✅ 已实现 | 权限查询同时约束用户授权租户与角色归属租户，避免跨租户角色绑定污染权限判断。 |
| **显式租户写入** | ✅ 已实现 | ORM 与建表脚本不再给业务表 `tenant_id` 设置默认 1，新业务写入必须由后端上下文显式注入租户；匿名签到按活动归属租户落库。 |
| **小程序租户上下文** | ✅ 已实现 | 小程序启动、页面进入、分享链接和扫码 scene 可携带 `tenant_code`；租户切换会清理旧登录态，API 登录/公开活动接口统一使用当前租户编码。 |
| **管理员活动可见范围** | ✅ 已实现 | 超级/全局管理员可看全部；活动类型或单活动授权管理员只看授权范围；无 scope 的管理员不再被误判为全量权限。 |
| **活动可报名状态** | ✅ 已实现 | 报名与支付下单均校验活动状态和结束时间；已结束或不可报名活动会被后端拒绝。 |
| **付费活动报名入口** | ✅ 已实现 | 付费活动未满员时必须先支付；满员后允许直接进入候补，避免候补仍被强制支付。 |
| **支付成功补偿** | ✅ 已实现 | 支付回调与订单查询共用成功落库逻辑；用户查询未完成订单时，若微信侧已支付成功，后端会尝试补偿创建参与记录并更新订单状态。 |
| **用户绑定资料完整性** | ✅ 已实现 | 绑定状态按完整资料字段判断；后端拒绝 `other` 性别，并校验同租户证件号不能重复。 |
| **小程序登录跳转** | ✅ 已实现 | 登录后的 `redirect` 只允许跳转到白名单页面，并区分 tab 页跳转方式，避免跳到未注册页或非法路径。 |
| **小程序管理员权限判断** | ✅ 已实现 | 前端不再把缺失 `adminLevel` 的管理员默认当作超级管理员；仅根据后端返回的明确级别或授权活动类型展示能力。 |

仍需继续验证/补齐：生产环境小程序域名、微信支付真机回调、公钥/证书配置、退款/补单/财务对账，以及管理端授权可视化。
