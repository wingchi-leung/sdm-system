# 社区功能 MVP 规格说明

## 一、文档目的

本文档用于定义微信小程序社区功能的最小可行版本（MVP），目标是在**尽量少改动现有系统结构**的前提下，快速上线一个可用版本，用于承载：

- 管理员围绕活动发布文章/通知/学习内容
- 已报名学员在文章下查看内容并发表评论
- 社区内容与活动形成关联，增强活动后的持续互动

本文档聚焦 **MVP 版本**，不追求一步做到完整的 Teams 式社区。

---

## 二、产品定位

### 2.1 功能定位

本期社区不是独立的“广场型社区”，而是一个**活动关联型内容区**：

- 每篇文章必须关联一个活动
- 内容主要由管理员发布
- 已报名学员以“阅读 + 评论互动”为主

### 2.2 为什么这样做

这样设计有三个明显优势：

- 贴合当前系统现状，现有系统已经以“活动”为核心
- 可直接复用现有活动详情页、管理员权限、用户登录体系
- 研发范围可控，适合 1 天内完成 MVP

### 2.3 本期目标

上线一个最简单但完整闭环的社区 MVP：

1. 管理员可以给某个活动发布文章
2. 已报名学员可以在小程序里查看文章
3. 已报名学员可以在文章下发表评论
4. 社区内容自然挂在活动详情页下

---

## 三、范围定义

### 3.1 本期包含

- 活动详情页增加“活动动态/社区内容”入口或区块
- 管理员发布文章
- 文章列表
- 文章详情
- 学员评论
- 评论列表

### 3.2 本期不包含

- 独立社区首页
- 点赞
- 收藏
- 转发统计
- 楼中楼回复
- 评论点赞
- 消息通知
- 内容审核后台
- 敏感词系统
- 老师独立角色体系
- 视频发布
- 打卡/跟练
- 全局搜索

### 3.3 关键取舍

本期“老师发文章”先由**管理员身份代替**。  
后续如有需要，再把“老师”从管理员体系中拆出来，形成独立的内容发布角色。

---

## 四、角色与权限

### 4.1 角色定义

- 管理员：可发布文章、查看文章、查看评论
- 已报名学员：可查看文章、发表评论
- 已登录未报名用户：不可查看社区内容
- 未登录访客：不可查看社区内容

### 4.2 权限规则

#### 管理员

- 可在有权限的活动下发布文章
- 可查看自己租户下的文章列表和详情
- MVP 阶段默认可查看该文章的全部评论

#### 已报名学员

- 仅当用户已报名当前活动后，才可查看该活动关联的文章
- 登录后可发表评论

#### 已登录未报名用户

- 不可查看文章列表、文章详情、评论
- 不可发表评论

#### 未登录用户

- 不可查看文章列表、文章详情、评论
- 不可发表评论

### 4.3 管理员权限范围

延续当前活动权限逻辑：

- 超级管理员可给任意活动发文章
- 活动类型管理员仅可给自己有权限的活动类型下的活动发文章

即：**文章发布权限继承活动权限，不新增独立权限体系。**

---

## 五、核心业务规则

### 5.1 文章规则

- 每篇文章必须绑定一个活动 `activity_id`
- 每篇文章必须属于一个租户 `tenant_id`
- 每篇文章只能由管理员创建
- 文章包含标题、正文、可选封面图
- 正文首期使用纯文本，不做富文本编辑器
- 文章默认仅对该活动的已报名用户可见

### 5.2 评论规则

- 评论必须绑定文章 `post_id`
- 评论人必须是当前租户下、当前活动已报名的登录用户
- 评论内容仅支持纯文本
- 每条评论只支持一级评论，不支持回复评论

### 5.3 展示规则

- 仅管理员和已报名用户可在活动详情页看到该活动下的文章列表
- 文章按发布时间倒序排列
- 评论按发布时间正序排列

### 5.4 可见性校验规则

- 社区内容的可见性按“是否已报名当前活动”判断
- 仅满足以下任一条件时，允许访问社区内容：
  - 当前用户是有权限的管理员
  - 当前用户是该活动已报名学员
- 未登录用户不可访问
- 已登录但未报名当前活动的用户不可访问
- 后端接口必须做服务端校验，不能只依赖前端隐藏入口

### 5.5 删除与审核规则

MVP 阶段先简化：

- 前台不提供评论删除
- 前台不提供文章删除
- 后台若需要删数据，可先由数据库或管理接口手动处理

## 六、用户流程

### 6.1 管理员发文流程

1. 管理员进入某活动详情页
2. 点击“发布动态”或“发布文章”
3. 填写标题、正文、可选封面图
4. 提交后生成文章
5. 返回活动详情页可看到新文章

### 6.2 学员阅读流程

1. 学员已报名当前活动
2. 学员进入活动详情页
3. 在“活动动态”区看到文章列表
4. 点击进入文章详情
5. 阅读正文与评论

### 6.3 学员评论流程

1. 学员登录
2. 学员已报名当前活动
3. 进入文章详情页
4. 输入评论内容
5. 提交成功后刷新评论列表

### 6.4 未登录用户流程

1. 用户进入活动详情页
2. 不展示社区入口
3. 若尝试直接访问文章详情页，系统提示需先登录并报名活动

### 6.5 已登录未报名用户流程

1. 用户进入活动详情页
2. 不展示社区入口，或展示“报名后可查看活动动态”
3. 若尝试直接访问文章详情页，后端返回无权限

### 6.6 管理员阅读流程

1. 管理员进入活动详情页
2. 在“活动动态”区看到文章列表
3. 点击进入文章详情
4. 阅读正文与评论

---

## 七、页面方案

### 7.1 活动详情页改造

在现有活动详情页下增加一个新模块：

- 模块名称建议：`活动动态`
- 展示内容：
  - 最近 3 篇文章
  - 若无文章则显示“暂无动态”
- 仅管理员和已报名用户可见
- 管理员可见按钮：
  - `发布动态`
- 用户可见入口：
  - 点击文章卡片进入详情
  - 点击“查看更多”进入文章列表页
- 未报名用户提示：
  - `报名后可查看活动动态`

### 7.2 新增页面建议

#### 文章列表页

- 页面名建议：`community-post-list`
- 展示某个活动下的全部文章
- 按发布时间倒序

#### 文章详情页

- 页面名建议：`community-post-detail`
- 展示标题、作者、发布时间、正文、评论列表
- 底部固定评论输入框

#### 发布文章页

- 页面名建议：`community-post-create`
- 仅管理员可进入
- 表单字段：
  - 标题
  - 正文
  - 封面图（可选）

### 7.3 交互原则

- 保持与当前小程序风格一致
- 不单独新增 tab
- 社区入口挂在活动详情页内，避免打散现有信息架构

## 八、数据模型建议

基于当前后端结构，建议新增两张表。

### 8.1 文章表 `community_post`

字段建议：

- `id`
- `tenant_id`
- `activity_id`
- `author_user_id`
- `title`
- `content`
- `cover_url`
- `status`
- `create_time`
- `update_time`

字段说明：

- `tenant_id`：租户隔离
- `activity_id`：关联活动
- `author_user_id`：发布人
- `title`：文章标题
- `content`：文章正文，首期纯文本
- `cover_url`：封面图地址，可为空
- `status`：文章状态，MVP 可先约定 `1=正常`

### 8.2 评论表 `community_comment`

字段建议：

- `id`
- `tenant_id`
- `activity_id`
- `post_id`
- `user_id`
- `content`
- `status`
- `create_time`
- `update_time`

字段说明：

- `activity_id`：冗余存储，便于按活动查询
- `post_id`：关联文章
- `user_id`：评论用户
- `content`：评论内容
- `status`：评论状态，MVP 可先约定 `1=正常`

### 8.3 为什么不做更复杂模型

本期先不增加：

- 回复父评论 `parent_comment_id`
- 点赞数
- 阅读数
- 审核状态流
- 置顶标记

这些能力后续都可平滑扩展。

## 九、接口建议

### 9.1 文章接口

#### 获取活动下文章列表

- `GET /community/posts?activity_id={id}`

用途：

- 活动详情页动态预览
- 文章列表页

访问条件：

- 管理员可访问
- 已报名当前活动的用户可访问
- 未登录或未报名用户不可访问

#### 获取文章详情

- `GET /community/posts/{post_id}`

访问条件：

- 管理员可访问
- 已报名当前活动的用户可访问
- 未登录或未报名用户不可访问

#### 创建文章

- `POST /community/posts`

请求体建议：

```json
{
  "activity_id": 1,
  "title": "课前阅读材料",
  "content": "请大家先阅读以下内容……",
  "cover_url": "https://..."
}
```

权限：

- 仅管理员

### 9.2 评论接口

#### 获取文章评论列表

- `GET /community/posts/{post_id}/comments`

访问条件：

- 管理员可访问
- 已报名当前活动的用户可访问
- 未登录或未报名用户不可访问

#### 发表评论

- `POST /community/posts/{post_id}/comments`

请求体建议：

```json
{
  "content": "老师这篇文章很有帮助"
}
```

权限：

- 仅已报名当前活动的登录用户

### 9.3 图片上传接口

可选方案：

- MVP 先复用现有上传能力思路，单独新增社区封面上传接口
- 或先不做上传，首版只支持纯文本文章

建议：

- 若追求 1 天落地，**可先支持无图文章**
- 若时间允许，再补单图封面上传

## 十、前后端改造点

### 10.1 后端改造

需要新增：

- ORM 表模型
- Pydantic 请求/响应模型
- CRUD
- 路由接口
- 权限校验
- 测试用例

后端新增校验重点：

- 校验当前用户是否为管理员
- 或校验当前用户是否已报名当前文章所属活动
- 所有社区读取接口都必须加上“报名态校验”

推荐目录：

- `backend/app/schemas.py`：新增数据库模型
- `backend/app/models/`：新增文章与评论的请求响应模型
- `backend/app/crud/`：新增 `crud_community_post.py`、`crud_community_comment.py`
- `backend/app/api/v1/endpoints/`：新增 `community.py`
- `backend/tests/`：新增社区接口测试

### 10.2 小程序改造

需要新增：

- API 封装
- 活动详情页社区区块
- 文章列表页
- 文章详情页
- 发布文章页

小程序侧新增控制重点：

- 未报名用户不展示社区入口
- 活动详情页根据报名状态决定是否显示“活动动态”
- 强刷或直接跳转文章页时，收到 403 要给出友好提示

推荐目录：

- `miniprogram/utils/api.js`
- `miniprogram/pages/activity-detail/`
- `miniprogram/pages/community-post-list/`
- `miniprogram/pages/community-post-detail/`
- `miniprogram/pages/community-post-create/`

## 十一、验收标准

满足以下条件即可视为 MVP 完成：

1. 管理员可以在某活动下成功发布文章
2. 仅管理员和已报名学员可以看到文章入口或预览列表
3. 已报名学员可以进入文章详情页查看正文
4. 已报名学员可以发表评论
5. 评论发布后可立即在页面看到
6. 不同租户之间文章与评论数据互相隔离
7. 无权限用户不能调用管理员发文接口
8. 未登录或未报名用户不能访问社区相关读取接口

---

## 十二、开发工作量评估

### 12.1 一天内可实现版本

如果目标严格限定为以下内容，一天内有较高概率完成：

- 管理员发文章
- 活动详情页展示文章列表
- 文章详情
- 学员评论
- 基础错误提示

### 12.2 预计耗时

- 后端接口与模型：3 到 4 小时
- 小程序页面与联调：3 到 5 小时
- 测试与修复：2 到 3 小时

合计预估：

- `8 到 12 小时`

### 12.3 一天内不建议加入的内容

以下内容极易导致超时：

- 老师独立角色
- 富文本编辑
- 多图上传
- 评论回复
- 点赞
- 审核流
- 通知提醒
- 独立社区首页

## 十三、后续演进方向

如果 MVP 验证通过，第二阶段可继续演进：

### P1

- 老师独立角色
- 评论删除
- 文章删除
- 封面图上传

### P2

- 回复评论
- 点赞
- 文章置顶
- 独立社区列表页

### P3

- 视频内容
- 学员打卡
- 消息通知
- 内容审核

---

## 十四、最终结论

本期最优方案不是做一个完整社区，而是做一个**挂在活动下的社区内容区**。

最简 MVP 的定义应当是：

- 管理员发文章
- 已报名学员看文章
- 已报名学员评论
- 全部围绕活动展开

这个方案与当前系统架构最匹配，且在控制范围的前提下，**具备 1 天内完成开发落地的可行性**。
