## 二、安全性问题

### 2.1 高危问题

#### 🔴 SEC-001: JWT Secret 默认值不安全
**文件**: `backend/app/core/config.py:15`
```python
JWT_SECRET: str = "change-me-in-production"
```
**风险**: 生产环境未配置环境变量时使用默认密钥，攻击者可伪造任意用户 Token。

**修复方案**:
1. 移除默认值，强制要求配置
2. 应用启动时检查是否为默认值，拒绝启动

---

#### 🔴 SEC-002: CORS 配置过于宽松
**文件**: `backend/app/main.py:19-25`
```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],  # 允许所有来源
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```
**风险**: 允许任意域名跨域访问，配合 `allow_credentials=True` 可能导致 CSRF 攻击窃取用户凭证。

**修复方案**:
1. 添加环境变量 `CORS_ORIGINS` 配置允许的域名列表
2. 生产环境禁止使用 `*` 配置

---

#### 🔴 SEC-003: 登录限流使用内存存储
**文件**: `backend/app/api/v1/endpoints/auth.py:22`
```python
_login_attempts: dict[str, list[float]] = defaultdict(list)
```
**风险**:
- 多实例部署时限流完全失效
- 服务重启后限流记录丢失
- 无内存上限，可能被恶意 IP 耗尽内存导致 OOM

**修复方案**:
1. 引入 Redis 实现分布式限流
2. 添加内存上限和过期清理机制

---

#### 🔴 SEC-004: 微信 access_token 使用内存缓存
**文件**: `backend/app/api/v1/endpoints/auth.py:25`
```python
_wechat_access_token_cache: dict[str, dict] = {}
```
**风险**: 多实例部署时每个实例单独请求微信 API，可能触发微信频率限制导致服务不可用。

**修复方案**:
1. 引入 Redis 缓存 access_token
2. 设置合理的过期时间（微信 token 有效期 2 小时）

---

#### 🔴 SEC-005: SQL 日志在生产环境开启
**文件**: `backend/app/database.py:26`
```python
engine = create_engine(
    SQLALCHEMY_DATABASE_URL,
    echo=True,  # 打印所有 SQL
)
```
**风险**: 打印所有 SQL 语句，泄露数据库结构信息，影响性能。

**修复方案**:
1. 通过环境变量控制 `echo` 开关
2. 生产环境默认关闭

## 三、高性能与高并发问题

### 3.1 严重问题

#### 🔴 PERF-001: 签到存在竞态条件 (Race Condition)
**文件**: `backend/app/crud/crud_checkin.py:67-94`

**问题代码**:
```python
def create_checkin(db: Session, checkin: CheckInCreate, tenant_id: int):
    # 检查是否已签到
    existing_checkin = check_already_checkin(db, ...)
    if existing_checkin:
        raise HTTPException(...)

    # 创建签到 - 并发问题！
    db_checkin = CheckInRecord(**checkin_data)
    db.add(db_checkin)
    db.commit()
```

**场景**: 用户快速点击签到按钮，两个请求同时通过"是否已签到"检查，导致重复签到。

**修复方案**:
1. 数据库层：添加唯一约束 `UNIQUE(activity_id, identity_number, tenant_id)`
2. 应用层：使用 `INSERT IGNORE` 或捕获唯一约束异常
3. 或使用 Redis 分布式锁

## 四、事务一致性问题

### 4.1 严重问题

#### 🔴 TX-001: 删除活动未使用事务保护
**文件**: `backend/app/crud/crud_activity.py:164-188`

**问题代码**:
```python
def delete_activity(db: Session, activity_id: int, tenant_id: int):
    db.query(ActivityParticipant).filter(...).delete()  # 删除参与人
    db.query(CheckInRecord).filter(...).delete()        # 删除签到
    db.delete(db_activity)                              # 删除活动
    db.commit()  # 如果这里失败，前面的删除已执行！
```

**风险**: 如果删除活动失败，参与人和签到记录已被删除，数据不一致。

**修复方案**:
```python
def delete_activity(db: Session, activity_id: int, tenant_id: int):
    try:
        db.query(ActivityParticipant).filter(...).delete()
        db.query(CheckInRecord).filter(...).delete()
        db.delete(db_activity)
        db.commit()
    except Exception as e:
        db.rollback()
        raise
```

**修复方案**: 整个操作使用 `try-except-rollback` 模式。
