## 二、核心问题

### 1. 高危：支付回调绕过正常报名逻辑，导致满员控制失效且已支付用户信息丢失

**修复状态：已修复**

**位置**
- `backend/app/api/v1/endpoints/payments.py:200-226`
- `backend/app/crud/crud_participant.py:103-138`

**现象**
- 支付成功回调里直接 `new ActivityParticipant(...)` 写入数据库。
- 没有复用 `crud_participant.create_participant()` 中已有的名额判断、候补逻辑、字段组装和一致性处理。
- 回调里只写入了 `participant_name`、`phone`、`payment_status`、`paid_amount` 等少数字段。
- `why_join`、`channel`、`expectation`、`activity_understanding`、`has_questions`、`identity_type`、`sex`、`age`、`occupation`、`email`、`industry` 等报名信息全部丢失。

**影响**
- 活动已满时，免费报名会进入候补，但付费报名回调仍会直接写成正式报名，业务规则被绕过。
- 付费用户报名资料与免支付用户资料结构不一致，后续名单、统计、审核、签到都可能出错。
- 这属于“支付成功但业务落库不完整”的核心一致性问题。

**建议**
- 支付订单中完整保存报名快照，而不是只保存姓名和手机号。
- 回调成功后统一走一条“支付后落库报名”的服务方法，复用名额、候补、字段映射和事务控制。
- 不要在回调函数中手写 `ActivityParticipant(...)`。

**复核结果**
- 当前下单接口已把完整报名数据写入 `payment_order.participant_snapshot`。
- 支付回调已改为解析快照后调用 `crud_participant.create_participant()`，正式复用名额与候补逻辑。
- 已新增支付回调测试，覆盖“活动满员后付费报名进入候补且问卷字段完整落库”。

### 2. 高危：支付回调未校验金额、商户和付款人关键信息，信任边界过宽

**修复状态：已修复**

**位置**
- `backend/app/api/v1/endpoints/payments.py:170-240`

**现象**
- 当前回调解密后，只取了 `out_trade_no`、`transaction_id`、`trade_state`。
- 没有把回调中的 `appid`、`mchid`、`amount.total`、`payer.openid` 与本地订单进行比对。

**影响**
- 即使回调报文来自微信，商户侧仍应校验“这笔回调是不是这张订单的回调”。
- 缺少金额与付款人核对，会让订单状态更新过于依赖外部报文结构，后续排查财务异常会很困难。
- 多商户、多应用、金额变更、人工补单等场景下风险更大。

**建议**
- 至少校验以下字段：
  - `amount.total == order.actual_fee`
  - `payer.openid == order.openid`
  - `appid == settings.WECHAT_APPID`
  - `mchid == settings.WECHAT_PAY_MCH_ID`
- 校验失败时记录审计日志并拒绝更新订单状态。

**复核结果**
- 当前回调已校验 `appid`、`mchid`、`amount.total`、`payer.openid`。
- 已新增金额不匹配测试，校验失败时不会更新订单成功状态，也不会创建报名记录。

### 3. 高危：证件号可为空，但支付回调固定写空字符串，容易引发重复判定失真与落库冲突

**修复状态：已修复**

**位置**
- `backend/app/api/v1/endpoints/payments.py:209-219`
- `backend/app/crud/crud_participant.py:190-198`
- `backend/app/schemas.py:104-111`

**现象**
- `ActivityParticipant` 唯一约束是 `(activity_id, identity_number, tenant_id)`。
- 免费报名与支付下单前的重复校验也只看 `identity_number`。
- 支付回调创建参与者时把 `identity_number` 固定写成 `""`。

**影响**
- 当活动允许不填证件号时，重复报名检查几乎失效。
- 同一活动下第二个“无证件号的付费报名”在回调落库时，极有可能因为唯一约束命中空字符串而失败。
- 结果会变成“用户微信侧已支付，但系统内订单仍未正确完成”，属于真实资金与业务状态不一致风险。

**建议**
- 明确“无证件号”场景的唯一性策略，不能继续仅靠 `identity_number`。
- 若业务允许无证件号报名，应改为基于更稳定的业务键判重，例如：
  - 已登录用户优先用 `user_id + activity_id`
  - 未登录场景再考虑手机号/证件号组合
- 回调时必须写回下单时的报名唯一标识，而不是固定空字符串。

**复核结果**
- 当前支付回调不再写死空字符串，而是直接回放下单快照中的 `identity_number`。
- 业务判重已优先使用 `user_id + activity_id`，仅在缺少用户标识时才回退到证件号判重。
- 本次复核补充了数据库层 `uk_participant_user_unique(activity_id, user_id, tenant_id)` 唯一约束，避免并发下仅靠应用层判重。
- 已新增无证件号支付回调测试，确认可以正常落库且后续重复下单会被拒绝。

### 5. 中危：创建支付订单时的重复报名校验过于单薄，无法覆盖真实业务路径

## 三、支付专项结论

当前支付功能的主要风险不是支付 SDK 层，而是业务落库层：

1. 下单时保存的信息太少。
2. 回调时没有复用正常报名逻辑。
3. 回调时没有把微信报文和本地订单做充分核对。
4. 证件号为空时，当前唯一性设计无法支撑稳定报名。

如果现在直接上线真实收款，最容易出现的线上问题是：

1. 用户支付成功，但系统未生成完整报名信息。
2. 活动已满后，付费用户仍被写成正式报名。
3. 第二个无证件号的付费用户支付后，回调落库失败。

## 二、核心问题

### 1. 高危：`wx.requestPayment` 成功后直接提示“支付成功”，没有等待后端订单真正完成

**修复状态：已修复**

**位置**
- `miniprogram/pages/register/register.js:305-325`
- `miniprogram/utils/api.js:501-514`

**现象**
- 小程序在 `wx.requestPayment.success` 后立刻提示“支付成功”并返回上一页。
- 现有 `queryPaymentOrder()` 已封装，但支付完成后没有调用。
- 也没有轮询后端订单状态、没有等待支付回调落库成功。

**影响**
- 如果用户支付完成，但后端回调处理失败，前端仍然会提示成功。
- 用户会以为已经报名成功，实际后台可能没有生成参与记录。
- 这是典型的“前端成功、后端失败”错觉，支付类业务里非常危险。

**建议**
- 支付调起成功后，进入“支付结果确认中”状态。
- 轮询 `/payments/order/{order_no}`，直到订单状态明确为成功或失败。
- 只有在后端确认成功后再提示“报名成功/支付成功”。

**复核结果**
- 当前 `doPaymentRegister()` 在 `wx.requestPayment.success` 后会调用 `confirmPaymentResult(orderNo)` 轮询订单状态。
- 只有后端订单状态明确为成功后，前端才提示“报名成功”或“支付成功，已进入候补”。

### 2. 高危：付费报名只上传少量字段，导致问卷与用户资料在支付链路中整体丢失

**修复状态：已修复**

**位置**
- `miniprogram/pages/register/register.js:218-251`
- `miniprogram/pages/register/register.js:283-288`
- `miniprogram/utils/api.js:482-499`

**现象**
- `buildParticipantData()` 构造了完整报名数据。
- 但支付分支调用 `api.createPaymentOrder()` 时，最终只有 `activity_id`、`participant_name`、`phone`、`identity_number`、`actual_fee` 被真正发送。
- 问卷字段和用户画像字段没有进入支付订单创建请求。

**影响**
- 免费报名和付费报名的数据模型不一致。
- 用户认真填写的参与原因、了解渠道、学习期望等内容，在支付报名路径中不会被后端保存。
- 后台看到的付费报名记录会明显缺字段，影响筛选、审核、统计与后续运营。

**建议**
- 付费报名请求应提交完整报名快照。
- 前端和后端统一一套“报名资料模型”，支付只是其中一个状态，而不是另一条字段更少的支线。

**复核结果**
- 当前支付分支直接复用 `buildParticipantData()`，并连同 `actual_fee` 一起提交到 `createPaymentOrder()`。
- `api.createPaymentOrder()` 也已改为透传完整有效字段，不再只保留少数字段。

### 3. 高危：前端把“是否满员”当成支付前判断依据，但它不是最终一致性保障

**修复状态：已修复**

**位置**
- `miniprogram/pages/register/register.js:353-364`
- `miniprogram/pages/register/register.js:90-101`

**现象**
- 页面进入时会读取一次 `enrollment-info`。
- 如果当时未满员，就会走支付流程。
- 但从加载页面到实际支付完成之间，活动名额可能已经变化。

**影响**
- 前端基于旧数据发起支付，无法保证最终一定还能占到正式名额。
- 这类校验只能作为提示，不能作为最终业务约束。
- 一旦后端也没有兜底，就会出现“前端显示可支付，实际已超卖”的问题。

**建议**
- 前端文案上把它视为“当前名额信息”，不要暗示这是最终锁定结果。
- 支付后应以服务端最终返回的报名状态为准，例如“报名成功”或“已进入候补”。

**复核结果**
- 当前前端支付完成后会以 `/payments/order/{order_no}` 返回的 `participant_enroll_status` 作为最终结果展示依据。
- 即使页面初始显示未满员，只要后端在支付回调时判定进入候补，前端最终也会展示“支付成功，已进入候补”。

### 4. 中危：报名页没有前置登录兜底，错误反馈主要依赖接口返回

**修复状态：已修复**

**位置**

**现象**

**影响**

**建议**
- 页面加载时就校验登录态。
- 未登录用户直接跳登录页，并带回跳参数。

**复核结果**

### 5. 中危：支付完成后缺少失败恢复路径，用户无法知道该重新支付还是联系客服

**修复状态：已修复**

**位置**

**现象**
- 当前 catch 里只做了简单的错误文案处理。
- 没有区分“取消支付”“微信侧成功但后端确认失败”“订单已创建但未支付完成”等情况。

**影响**
- 用户不知道当前订单是否还能继续支付、是否需要重试、是否已经扣款。
- 客服排查时缺少前端可见的订单号和明确状态提示。

**建议**
- 在支付流程中保留 `order_no`。
- 失败时根据订单状态给出明确下一步提示：
  - 未支付：可重试
  - 已支付待确认：请稍候刷新
  - 已失败：联系客服并提供订单号

**复核结果**
- 当前代码已保留 `orderNo`，并对取消支付、结果确认超时、订单关闭、支付失败做了差异化提示。
- 当前错误提示已带上订单号，用户在“确认超时/支付失败/订单关闭”等场景下可直接把订单号提供给客服排查。

---

## 三、支付专项结论

当前小程序支付链路的主要问题是“过早宣布成功”和“资料只传了一半”。

如果配合当前后端实现一起看，最容易出现的用户侧问题是：

1. 用户看见“支付成功”，但后台没有完整报名记录。
2. 用户填写的问卷内容在付费报名后消失。
3. 用户支付时页面显示还有名额，但最终名单状态并不可靠。

---

## 四、建议修复顺序

### P0：先修

1. 支付成功后改为查询订单状态，不再直接 toast 成功。
2. `createPaymentOrder` 改为上传完整报名数据。
3. 支付失败页或错误提示中展示订单号与下一步动作。

### P1：随后修

1. 报名页增加登录前置校验。
2. 支付流程增加“确认中”状态与超时处理。
3. 页面文案改为以服务端最终结果为准，避免误导用户。
