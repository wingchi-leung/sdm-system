# 权限系统重构方案（RBAC）

## 0. 架构概览

### 三层分离

```
┌─────────────────────────────────────────────┐
│  认证层 (Authentication)                      │
│  "你是谁" — user_credential 表统一管理凭证     │
├─────────────────────────────────────────────┤
│  身份层 (Identity)                            │
│  "你有什么身份" — user 表是唯一主体            │
├─────────────────────────────────────────────┤
│  授权层 (Authorization)                       │
│  "你能干什么" — RBAC，与登录完全解耦           │
└─────────────────────────────────────────────┘
```

### 核心原则

- **登录只验身份**：认证通过 = 拿到 user_id，不检查角色
- **权限在接口层检查**：没有角色的用户能登录，但访问管理接口时返回 403
- **凭证统一管理**：所有登录方式（密码、微信、手机号）统一存储在 `user_credential` 表
- **身份统一**：`user` 表是唯一的"人"，不再有 `admin_user` / `platform_admin` 等派生身份表
- **JWT 无角色**：token 只含 `{sub, tid, exp}`，身份由 `user_role` 表实时决定

## 1. 数据模型

### 核心表结构

```sql
-- 用户表（唯一身份主体）
CREATE TABLE user (
  id INT PRIMARY KEY AUTO_INCREMENT,
  tenant_id INT NOT NULL DEFAULT 1,     -- 主租户ID，0=平台级用户
  name VARCHAR(255),
  phone VARCHAR(255),
  email VARCHAR(255),
  sex VARCHAR(10),
  age INT,
  occupation VARCHAR(100),
  industry VARCHAR(100),
  isblock TINYINT DEFAULT 0,            -- 0-正常 1-拉黑
  block_reason VARCHAR(255),
  identity_number VARCHAR(255),
  identity_type VARCHAR(20),
  create_time DATETIME,
  update_time DATETIME
);

-- 用户凭证表（统一登录凭证）
CREATE TABLE user_credential (
  id INT PRIMARY KEY AUTO_INCREMENT,
  user_id INT NOT NULL,                 -- 关联 user.id
  tenant_id INT NOT NULL,
  credential_type VARCHAR(32) NOT NULL, -- 'password', 'wechat', 'phone_code'
  identifier VARCHAR(255) NOT NULL,     -- 用户名/openid/手机号
  credential_hash VARCHAR(255),         -- bcrypt hash（password 类型）
  must_reset_password TINYINT DEFAULT 0,-- 1=需要改密（仅 password 类型）
  status TINYINT DEFAULT 1,             -- 1-正常 0-禁用
  UNIQUE KEY uk_credential_lookup (tenant_id, credential_type, identifier)
);

-- 用户-租户关联表
CREATE TABLE user_tenant (
  id INT PRIMARY KEY AUTO_INCREMENT,
  user_id INT NOT NULL,
  tenant_id INT NOT NULL,
  status TINYINT DEFAULT 1,             -- 1-正常 0-禁用
  joined_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  UNIQUE KEY uk_user_tenant (user_id, tenant_id)
);

-- 权限表（系统预定义）
CREATE TABLE permission (
  id INT PRIMARY KEY AUTO_INCREMENT,
  code VARCHAR(64) UNIQUE NOT NULL,     -- 'activity.create', 'activity.edit'
  name VARCHAR(100) NOT NULL,
  resource VARCHAR(32) NOT NULL,        -- 'activity', 'user', 'report'
  action VARCHAR(32) NOT NULL,          -- 'create', 'edit', 'delete', 'view'
  description TEXT
);

-- 角色表
CREATE TABLE role (
  id INT PRIMARY KEY AUTO_INCREMENT,
  tenant_id INT NOT NULL,               -- 0 = 系统预设角色
  name VARCHAR(64) NOT NULL,
  is_system TINYINT NOT NULL DEFAULT 0,
  description TEXT,
  UNIQUE KEY uk_tenant_role_name (tenant_id, name)
);

-- 角色-权限关联表
CREATE TABLE role_permission (
  id INT PRIMARY KEY AUTO_INCREMENT,
  role_id INT NOT NULL,
  permission_id INT NOT NULL,
  UNIQUE KEY uk_role_permission (role_id, permission_id)
);

-- 用户-角色关联表
CREATE TABLE user_role (
  id INT PRIMARY KEY AUTO_INCREMENT,
  user_id INT NOT NULL,
  role_id INT NOT NULL,
  tenant_id INT NOT NULL,
  scope_type VARCHAR(32),               -- NULL=全局, 'activity_type', 'activity'
  scope_id INT,                         -- 活动类型ID 或 活动ID
  UNIQUE KEY uk_user_role_scope (user_id, role_id, tenant_id, scope_type, scope_id)
);
```

### 历史表（删表前需完成迁移，见第 8 节）

- `admin_user` — 凭证已迁移到 `user_credential`
- `platform_admin` — 需迁移到 `user` + `user_role`

## 5. 认证体系

### JWT 结构

```json
{ "sub": "user_id", "tid": "tenant_id", "exp": "..." }
```

- `tid` = 0 表示平台级用户
- 不含 `role` 字段，身份由 `user_role` 表实时决定

### 凭证类型

| credential_type | identifier | credential_hash | 说明 |
|----------------|------------|-----------------|------|
| `password` | 用户名/手机号 | bcrypt hash | 密码登录 |
| `wechat` | wx_openid | NULL | 微信授权登录 |
| `phone_code` | 手机号 | NULL | 微信手机号授权登录 |

### 后端鉴权依赖（`deps.py`）

```python
class AuthContext:
    user_id: int
    tenant_id: int | None      # None/0 = 平台级
    tenant_code: str | None

get_current_user()          # 任意已登录用户，不检查角色
get_current_admin()         # 要求在该租户下有至少一个角色
get_current_platform_admin()# 要求有平台管理员角色 (role_id=3)
get_current_admin_or_platform() # 租户管理员或平台管理员
require_permission(code)    # RBAC 权限检查
```

## 6. API 端点

### 认证接口

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/v1/auth/login` | POST | 统一密码登录（管理员/用户/平台管理员） |
| `/api/v1/auth/wechat` | POST | 统一微信认证（openid/phone 两种模式） |
| `/api/v1/auth/set-password` | POST | 修改密码（需先登录） |
| `/api/v1/auth/me` | GET | 获取当前用户信息 |

### 登录流程

#### POST /auth/login — 统一密码登录

请求：
```json
{ "identifier": "13800138000", "password": "xxx", "tenant_code": "default" }
```

- `tenant_code` = "platform" → 平台管理员认证 (tenant_id=0)
- `tenant_code` 有值 → 租户内认证

响应（统一结构）：
```json
{
  "access_token": "...",
  "user": { "id": 1, "name": "张三", "phone": "138..." },
  "tenant": { "id": 1, "name": "默认租户", "code": "default" },
  "auth": {
    "is_admin": true,
    "is_platform_admin": false,
    "is_super_admin": true,
    "permissions": ["activity.create", "activity.edit", ...],
    "activity_types": [{"id": 1, "name": "参", "code": "can"}],
    "must_reset_password": false
  }
}
```

#### POST /auth/wechat — 统一微信认证

请求：
```json
{ "code": "xxx", "tenant_code": "default", "mode": "phone", "login_code": "yyy" }
```

- `mode` = "openid"（默认）：code 是 wx.login 的 code
- `mode` = "phone"：code 是 getPhoneNumber 的 code，login_code 是 wx.login 的 code

响应（统一结构 + 额外字段）：
```json
{
  "access_token": "...",
  "user": { ... },
  "tenant": { ... },
  "auth": { ... },
  "is_first_login": false,
  "require_bind_info": false,
  "phone": "13800138000",
  "wechat_payment_ready": true,
  "wechat_payment_hint": null
}
```

### RBAC 接口

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/v1/roles/permissions` | GET | 获取所有权限列表 |

## 7. 用户使用流程

### 场景：分配一个"参活动类型管理员"

**Step 1：超级管理员登录 Web 后台**

**Step 2：进入"权限与管理员"页面**

**Step 3：分配角色**
- 选择用户（例如：李四，手机号 13800138000）
- 选择角色（例如："参活动管理员"）
- 权限范围选择"活动类型" → 选择"参"
- 确认分配

**Step 4：系统自动创建登录凭证**
- `user_credential` 表新增一行：type='password', identifier=手机号
- `credential_hash` = 123456（bcrypt 加密）
- `must_reset_password` = 1（需要改密）

**Step 5：李四收到通知，用手机号 + 123456 登录**

**Step 6：登录成功后强制跳转到改密页面**
- 设置新密码（至少6位）
- 改密成功后 `must_reset_password` = 0

**Step 7：跳转登录页，用新密码重新登录**
- 以后可用新密码从 Web 或小程序管理员入口登录

### 超级管理员重置用户密码

在"权限与管理员"页面，找到对应用户，点击"重置密码"按钮，可强制将密码重置为 123456（同时 `must_reset_password` 恢复为 1）。

## 8. 删表前迁移要点

代码层已完全不依赖 `admin_user` 和 `platform_admin` 表。删表前需确认以下数据迁移已执行：

### admin_user → user_credential

```sql
-- 确认所有 admin_user 的密码凭证已迁移
SELECT au.id, au.username, au.tenant_id
FROM admin_user au
LEFT JOIN user_credential uc
  ON uc.user_id = au.user_id
  AND uc.tenant_id = au.tenant_id
  AND uc.credential_type = 'password'
WHERE uc.id IS NULL AND au.password_hash != '';
-- 期望结果：0 行。如有遗漏，补执行 field.sql 中的 Phase 1 迁移语句
```
 

### user_tenant 填充

```sql
-- 确认所有用户都有 user_tenant 记录
SELECT u.id, u.tenant_id
FROM user u
LEFT JOIN user_tenant ut ON ut.user_id = u.id AND ut.tenant_id = u.tenant_id
WHERE u.tenant_id > 0 AND ut.id IS NULL;
-- 期望结果：0 行。如有遗漏，补执行：
-- INSERT INTO user_tenant (user_id, tenant_id) SELECT id, tenant_id FROM user WHERE tenant_id > 0 ON DUPLICATE KEY UPDATE status = 1;
```

### 确认无误后删表

```sql
DROP TABLE IF EXISTS admin_user;
DROP TABLE IF EXISTS platform_admin;
```

## 9. 设计优势

- **认证与授权分离**：登录只验身份，权限在接口层检查
- **单一身份模型**：`user` 表是唯一的"人"，不再有派生身份表
- **统一凭证管理**：密码、微信、手机号统一存储在 `user_credential`
- **统一登录入口**：2 个端点覆盖所有登录场景
- **灵活扩展**：新增权限只需插入数据，新增凭证类型只需加一行
- **细粒度控制**：权限可精确到单个活动或活动类型
- **JWT 无状态**：token 不含角色，身份实时查询，角色变更即时生效
