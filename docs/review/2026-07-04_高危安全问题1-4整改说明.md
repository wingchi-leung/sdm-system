## 二、对应问题总览

本次处理的 4 个问题分别是：

1. 退款幂等依赖应用层查询，缺少数据库唯一约束，存在并发重复退款风险
2. 非成功支付回调未做业务字段绑定校验，可能错误下调本地订单状态
3. 超级管理员初始化脚本使用仓库内硬编码密码，存在高权限凭据泄露风险
4. 角色分配会给目标用户设置可预测默认密码，且该密码可直接登录

## 三、逐项整改说明

### 1. 退款幂等并发问题

#### 原问题

退款接口原来只是在应用层通过：

- `tenant_id`
- `payment_order_id`
- `idempotency_key`

先查一次，再决定是否创建退款记录。

这会导致两个并发请求同时通过“未查到”的判断，然后各自插入退款流水，形成并发穿透。

#### 已落地修复

已经在后端做了两层修复：

1. ORM 层新增唯一约束  
   `payment_refund(tenant_id, payment_order_id, idempotency_key)`

2. 接口层增加并发兜底  
   如果并发插入触发唯一键冲突，接口会回查已有退款记录，并把那条记录作为幂等结果返回，而不是让请求直接炸掉

#### 代码位置

- `backend/app/schemas.py`
- `backend/app/api/v1/endpoints/payments.py`
- `backend/sql/table.sql`

#### 你还需要做的事

这条修复要真正生效，必须把数据库变更同步到真实环境。

你需要确认：

1. 生产库已经执行了最新 `backend/sql/table.sql` 中的补充 `ALTER TABLE`
2. `payment_refund` 表里已经存在唯一索引 `uk_refund_order_idem`

建议你在数据库里执行类似检查：

```sql
SHOW INDEX FROM payment_refund;
```

确认返回里包含：

```text
uk_refund_order_idem
```

如果没有，说明代码虽然修了，但数据库层还没真正生效。

### 2. 非成功支付回调缺少业务绑定校验

#### 原问题

支付成功回调路径已有业务字段绑定校验，但失败分支没有同样的校验。

这意味着一个来源合法、但业务字段不匹配的失败回调，可能把本地订单错误标记为失败。

#### 已落地修复

现在无论支付回调是成功还是失败，都会先校验：

- `appid`
- `mchid`
- `amount.total`
- `payer.openid`

只有校验通过，才允许修改本地订单状态。

#### 代码位置

- `backend/app/api/v1/endpoints/payments.py`

#### 你还需要做的事

这项修复本身不需要额外手工改代码，但你需要确认部署环境里的微信支付配置是正确且一致的：

需要确认这些环境变量在真实环境中是正确值：

- `WECHAT_APPID`
- `WECHAT_PAY_MCH_ID`
- `WECHAT_PAY_API_V3_KEY`
- `WECHAT_PAY_NOTIFY_URL`

因为现在失败回调也会做强绑定校验，如果这些值配置错了，回调会被拒绝。

建议上线前做一次：

1. 测试环境真实支付成功回调验证
2. 测试环境退款/关闭/失败回调验证

### 3. 超级管理员初始化脚本硬编码密码

#### 原问题

`backend/create_admin.py` 里原来直接写死了超级管理员密码。

风险点不是“这个脚本能不能用”，而是：

1. 仓库里长期保留高权限默认密码
2. 任何拿到仓库的人都知道这个密码
3. 一旦脚本在真实环境被重复执行，就可能把管理员密码重置回已知值

#### 已落地修复

现在脚本已经不再允许使用仓库内硬编码密码。

改成了：

- 必须通过环境变量 `BOOTSTRAP_ADMIN_PASSWORD` 显式提供
- 如果没提供，脚本会直接退出并报错

#### 代码位置

- `backend/create_admin.py`
- `backend/app/core/config.py`

#### 现在应该怎么用

在 Windows PowerShell 下：

```powershell
$env:BOOTSTRAP_ADMIN_PASSWORD='这里换成强密码'
cd backend
python create_admin.py
```

如果你要临时执行一次，以上写法就够了，只在当前 PowerShell 会话生效。

如果你是放在部署脚本或服务启动环境里，需要把它配置到正式环境变量中，而不是写回仓库。

#### 密码建议

建议至少满足：

1. 长度不低于 20 位
2. 包含大小写字母、数字、特殊字符
3. 不与其他系统共用
4. 不写入仓库，不写入截图，不发到群里

#### 你还需要做的事

你需要完成这 4 件事：

1. 在测试环境和生产环境补充 `BOOTSTRAP_ADMIN_PASSWORD`
2. 确认旧的硬编码密码已经不再被任何脚本、文档、部署流程引用
3. 如果这个脚本曾在真实环境跑过，需要人工确认管理员密码是否需要重新轮换
4. 如果已有运维 SOP，请把“执行 `create_admin.py` 前必须设置 `BOOTSTRAP_ADMIN_PASSWORD`”补进去

#### 建议的执行方式

如果你们的部署方式是手工 PowerShell：

```powershell
$env:BOOTSTRAP_ADMIN_PASSWORD='一个新的强密码'
python backend/create_admin.py
```

如果你们的部署方式是 `.env` 注入或容器注入：

- 把 `BOOTSTRAP_ADMIN_PASSWORD` 配到部署环境
- 不要把它写回 `backend/.env.example`
- 更不要写回真实仓库中的 `.env`

### 4. 角色分配下发可预测默认密码

#### 原问题

原来给用户分配角色时，如果用户有手机号，就会自动创建或重置密码为固定默认值，例如 `123456`，并且这个密码可直接用于登录。

这会造成：

1. 被授权用户账号出现可预测密码
2. 只要知道目标手机号，就可能尝试接管账号
3. `must_reset_password` 只是提示，不是真正的登录门禁

#### 已落地修复

这次修了两层：

1. 角色分配时，不再写入可预测默认密码  
   现在只有在“该用户完全没有密码凭证”时，才会生成一个随机占位密码

2. 密码登录阶段强制拦截 `must_reset_password=1`  
   即使存在临时密码凭证，也不能直接用密码登录，必须先走重置流程

#### 代码位置

- `backend/app/api/v1/endpoints/roles.py`
- `backend/app/api/v1/endpoints/auth.py`

#### 现在的行为变化

现在用户被授予角色后：

- 已有正常密码的账号：保持原密码，不会被覆盖
- 没有密码凭证的账号：系统会创建随机占位密码，但这个账号不能直接用密码登录
- 只有完成后续密码设置/重置后，才允许走密码登录

#### 你还需要做的事

这一条需要你从业务和运维两侧确认：

1. 现在管理员给用户分配角色后，用户第一次应该通过什么路径完成“设置密码”
2. 你们是否要把“首次登录请先通过微信/手机号登录，再设置密码”写进操作说明
3. 如果后台页面或小程序页面里有“管理员授权后默认密码是 123456”之类的旧提示，必须删掉
4. 如果客服或运营以前按“默认密码 123456”指导用户，现在也要同步更新话术

建议至少补一条人工流程说明：

- 用户被赋予管理角色后
- 如需使用密码登录
- 先通过已绑定的微信/手机号完成身份进入
- 再在系统内主动设置密码
