# 功能说明：Flutter 前端 vs 后端

## 一、Flutter 前端实现了啥

前端（`event_app`）是**活动报名 C 端 Demo**，同一套代码可跑在 **Web、iOS、Android、iPad**。

### 已实现功能

| 功能 | 说明 |
|------|------|
| **活动列表** | 调用 `GET /api/v1/activities` 拉取活动，展示名称、开始时间、状态（未开始/进行中/已结束），支持下拉刷新、加载失败重试 |
| **活动详情** | 点击某条活动进入详情页，展示状态、开始/结束时间、标签 |
| **报名** | 未结束的活动显示「我要报名」；点击后进入报名页，填写**姓名、手机号、证件号（选填）**，提交到 `POST /api/v1/participants/`；报名成功有提示，失败会显示后端返回的错误信息 |
| **多端运行** | 支持 Chrome（Web）、iOS 模拟器、iPhone 真机、Android；API 地址在 `lib/services/api_service.dart` 里按环境配置（localhost / 局域网 IP / 10.0.2.2） |

### 前端未实现（后续可加）

- 用户登录/账号体系（当前报名无需登录）
- 签到入口（后端有 `POST /checkins`，前端未做页面）
- 管理后台（发布活动、管理人员等）
- 黑名单、多团队、名额限制等（依赖后端先支持）

---

## 二、后端改了啥

后端（`backend`）的**业务接口和表结构**基本是原有设计，没有大改；**改动主要是工程与依赖**。

### 业务接口（原有，未改）

| 模块 | 接口 | 说明 |
|------|------|------|
| 活动 | `GET /api/v1/activities` | 活动列表（分页、按状态筛选） |
| | `POST /api/v1/activities` | 创建活动（含参与人） |
| | `PUT /api/v1/activities/{id}/status` | 更新活动状态 |
| 报名 | `POST /api/v1/participants/` | 报名（activity_id + 姓名 + 手机 + 证件号可选） |
| | `GET /api/v1/participants/{activity_id}/` | 某活动的报名列表 |
| 签到 | `POST /api/v1/checkins/` | 签到（校验已报名、防重复） |
| 用户 | `GET/POST /api/v1/users` 等 | 用户 CRUD（报名不依赖登录，可选用） |

### 后端实际改动（我们做的）

| 改动 | 说明 |
|------|------|
| **依赖与工程** | 增加 `pyproject.toml`、`requirements.txt`，用 **uv** 或 pip 管理依赖；增加 `.gitignore`、`run.sh` |
| **MySQL 8 认证** | 增加 **`cryptography`** 依赖，解决 `caching_sha2_password` 报错，确保 PyMySQL 能连 MySQL 8 |
| **文档与启动** | `README.md` 写清用 **uv** 的步骤（`uv venv`、`uv sync`、`uv run uvicorn ...`），以及不用 uv 时的 pip 备用方式 |

### 后端未改（仍是原状）

- 活动/报名/签到/用户的**接口定义和业务逻辑**没改
- **没有**鉴权（无 JWT/登录）
- **没有**黑名单、多团队、数据隔离、名额上限等（可按评估文档后续加）

---

## 三、前后端对应关系（当前）

- 前端**只用到**：`GET /activities`、`POST /participants/`
- 后端的 `checkins`、`users`、活动创建/状态更新等接口，前端 Demo 未接，留给管理端或后续迭代用。
