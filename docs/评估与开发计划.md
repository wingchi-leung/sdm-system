# 活动报名 App 需求评估与开发计划

## 一、需求摘要

| 需求项 | 说明 |
|--------|------|
| 终端 | iOS、Android、iPad（Flutter） |
| 第一版核心 | 活动报名流程：发布活动 → 用户报名 |
| 报名方式 | 报名人**无需在系统登录或维护账号**，仅填写报名信息即可 |
| 控制能力 | **黑名单控制**（限制部分人报名） |
| 管理后台 | 发布活动、管理人员、支持**多团队接入**且**数据隔离** |

---

## 二、后端现状与需求对照

### 2.1 已有能力（与需求基本匹配）

- **活动 CRUD**：`/api/v1/activities` 创建、列表、状态更新已有。
- **报名人独立于用户**：`activity_participants` 表 `user_id` 可空，支持「不登录仅填信息」报名。
- **报名接口**：`POST /participants` 按活动 + 姓名/证件号/手机号报名，且防重复（同一活动+证件号）。
- **签到**：`POST /checkins` 校验已报名后再签到，防重复签到。
- **技术栈**：FastAPI + MySQL + SQLAlchemy，结构清晰，便于扩展。

### 2.2 缺口与差异

| 能力 | 现状 | 需要补足 |
|------|------|----------|
| 黑名单 | 无 | 黑名单表/逻辑：按证件号或手机号拦截报名 |
| 管理后台 | 仅 API，无鉴权 | 管理员登录、权限、后台 API 鉴权 |
| 多团队/数据隔离 | 无 | 团队(租户)表 + 活动/报名等表加 team_id，查询与写入均按团队过滤 |
| 活动创建与报名解耦 | 创建活动时必须带 `participants` 列表 | 支持「仅创建活动、后续再报名」的流程（participants 可选或空） |

### 2.3 后端已知问题（需先修复）

1. **activities 接口**
   - `get_activity_checkins` 使用了未定义的 `crud.get_activity_checkins`，且该函数在 crud 层未实现。
   - `get_activity_statistics` 应改为调用 `crud_participant.get_activity_statistics`，并修正 `activities` 里对 `crud` 的引用。
2. **crud_activity**
   - `update_activity_status` 调用了不存在的 `get_activity(db, activity_id)`，需在 crud_activity 中实现 `get_activity`。
3. **crud_participant**
   - `get_activity_statistics` 中使用了 `CheckInRecord` 和 `datetime`，未导入，运行会报错。
4. **活动创建**
   - 若希望「先发布活动、后开放报名」，需支持创建活动时不带或带空 `participants`（如 `ActivityCreate.participants` 设为可选并默认 `[]`）。

---

## 三、可行性结论

- **可行。** 现有后端已覆盖「活动 + 无登录报名 + 签到」主流程，用 Flutter 做 iOS/Android/iPad 也合适；缺口（黑名单、多团队隔离、管理端鉴权）均可通过增量开发完成。
- **建议**：先修上述后端 bug 并补全活动/签到统计接口，再按阶段加黑名单、多团队与鉴权，最后接 Flutter 端与管理后台。

---

## 四、开发计划（分阶段）

### 阶段 0：后端修 bug 与接口补全（约 0.5～1 天）

- 在 `crud_activity` 中实现 `get_activity(db, activity_id)`。
- 在 `crud_checkin` 中实现 `get_activity_checkins(db, activity_id, skip, limit)`，并在 `activities` 路由中改为调用 `crud_checkin` / `crud_participant`，修正 `crud` 引用。
- 在 `crud_participant` 中为 `get_activity_statistics` 补全 `CheckInRecord`、`datetime` 的导入。
- 视产品需求调整 `ActivityCreate`：`participants` 改为可选，默认 `[]`，以支持「先发活动、后报名」。

### 阶段 1：黑名单与报名拦截（约 1～2 天）

- **数据**：新增黑名单表（如按 `identity_number` 或 `phone`，可加 `team_id` 为后续多团队预留）。
- **逻辑**：报名时（`POST /participants`）先查黑名单，命中则直接拒绝并返回明确提示。
- **接口**：管理端「黑名单管理」：增删查（可放在 admin 或独立 prefix，后续与鉴权一起加）。

### 阶段 2：多团队与数据隔离（约 2～3 天）

- **数据**：新增团队表（如 `team`）；`activity`、`activity_participants`、`checkin_records`、黑名单等表增加 `team_id`（或 `tenant_id`），并建索引。
- **逻辑**：所有活动/报名/签到/黑名单的查询与写入都带 `team_id`（从当前登录管理员或 token 中取）；确保各团队只能看到、操作本团队数据。
- **接口**：若多团队对应多管理员，需在登录/鉴权时绑定 `team_id`；团队列表、切换等按实际管理方式设计。

### 阶段 3：管理端鉴权与基础后台 API（约 2～3 天）

- **认证**：JWT 或 Session；管理员表（如 `admin_user`，可带 `team_id`、角色）。
- **权限**：管理端接口统一加依赖项校验 token、并注入当前 `team_id`（及可选角色），与阶段 2 的数据隔离一致。
- **接口**：活动发布/编辑/下线、报名列表、签到列表、黑名单管理、活动统计等，均走已鉴权且按团队隔离的 API。

### 阶段 4：Flutter App（C 端报名 + 可选签到）（约 2～3 周）

- **C 端**：活动列表（可区分未开始/进行中/已结束）→ 活动详情 → 报名表单（姓名、证件号、手机号等）；无需登录；若被黑名单拦截，提示「无法报名」。
- **可选**：签到入口（扫码或输入活动+证件号）与后端 `POST /checkins` 对接。
- **适配**：Flutter 一套代码适配 iOS、Android、iPad（响应式布局）；注意 iPad 横竖屏与多列展示。

### 阶段 5：管理后台（Web 或 Flutter）（约 2～3 周）

- **功能**：登录 → 活动发布/编辑、报名列表、黑名单管理、签到与统计看板；若多团队，则登录后仅见本团队数据。
- **形态**：Web 管理后台（React/Vue 等）或 Flutter Web/Desktop，与阶段 3 的 API 对接即可。

---

## 五、建议实施顺序

1. **先做阶段 0**，保证现有接口可运行、无缺失引用。
2. **再做阶段 1（黑名单）**，业务价值高、改动集中。
3. **阶段 2（多团队）与阶段 3（鉴权）** 可并行设计，建议先定「团队与管理员」模型再落库与接口。
4. **Flutter App（阶段 4）** 可与阶段 2/3 并行启动，先用「无鉴权、单团队」的现有 API 做 C 端报名与列表，待鉴权与多团队就绪后再接新 API 与策略。
5. **管理后台（阶段 5）** 在阶段 3 完成后推进，效率最高。

---

## 六、风险与注意点

- **数据隔离**：多团队上线前务必对所有相关表与 API 做一次「带 team_id 过滤」的排查，避免越权。
- **黑名单维度**：若未来要按「活动维度」黑名单，可再扩展表结构；首版按全局（或按团队）黑名单即可。
- **报名人数/名额**：若需「名额上限」，在活动表加 `max_participants`，报名时比较当前报名数即可，与现有逻辑兼容。
