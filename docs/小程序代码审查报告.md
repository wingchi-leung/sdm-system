## 二、安全性问题

### 2.1 高危问题

#### 🔴 SEC-001: API baseUrl 硬编码，使用 HTTP 非安全连接
**文件**: `miniprogram/utils/api.js:5`
```javascript
const baseUrl = 'http://10.4.19.188:8000/api/v1';
```
**风险**:
- HTTP 明文传输，密码和 Token 可被中间人攻击窃取
- 内网 IP 硬编码，无法灵活切换环境
- 真机部署需要在小程序后台配置 request 合法域名（仅支持 HTTPS）

**修复方案**:
1. 使用环境变量或配置文件管理 baseUrl
2. 生产环境必须使用 HTTPS
3. 配置多环境支持（开发/测试/生产）

---

#### 🔴 SEC-002: AppID 暴露在代码中
**文件**: `miniprogram/project.config.json:37`
```json
"appid": "wx4157e7ad044df60a"
```
**风险**: AppID 与后端 `.env` 中的敏感配置一致，可能被用于恶意请求。

**建议**: 虽然小程序 AppID 是公开信息，但应确保微信后台配置了服务器域名白名单。

---

#### 🔴 SEC-003: urlCheck 关闭导致安全检查失效
**文件**: `miniprogram/project.config.json:10`
```json
"urlCheck": false
```
**风险**: 开发者工具不校验合法域名，可能导致：
- 开发时使用不安全的域名
- 生产环境因域名未配置而失败
- 绕过微信的安全域名检查

**修复方案**: 开发完成后开启 `urlCheck: true`，确保所有域名已在小程序后台配置。

---

#### 🔴 SEC-004: Token 存储在 Storage 中无加密
**文件**: `miniprogram/utils/auth.js:165`
```javascript
wx.setStorageSync(KEY_TOKEN, accessToken);
```
**风险**:
- Storage 可被物理访问设备的人读取
- 越狱/Root 设备上可能被恶意应用读取

**建议**:
- Token 有效期不宜过长（建议与后端 JWT_EXPIRE 保持一致）
- 敏感操作应二次验证
- 考虑使用微信的安全存储 API

---

### 2.2 中危问题

#### 🟡 SEC-005: 登录页管理员入口隐藏方式不安全
**文件**: `miniprogram/pages/login/login.js:25-51`
```javascript
// 快速点击5次进入管理员模式
if (this.gateTapCount === 5) {
  this.setData({ isAdminMode: true });
}
```
**风险**:
- 隐藏入口可被猜测或意外触发
- 无任何身份验证即可访问管理员登录界面

**建议**: 虽然最终需要账号密码验证，但建议：
- 在管理端提供显式的管理员登录入口
- 或使用更安全的入口验证方式

---

#### 🟡 SEC-006: 报名页面缺少权限验证
**文件**: `miniprogram/pages/register/register.js`

**问题**: 报名接口调用时未验证用户是否已登录。

**风险**: 后端会因 Token 缺失返回 401，但前端未提前拦截，用户体验差。

**建议**: 在页面 onLoad 时检查登录状态，未登录则跳转登录页。

---

#### 🟡 SEC-007: 活动列表页缺少权限验证
**文件**: `miniprogram/pages/activity-list/activity-list.js`

**问题**: 仅检查 `isAdmin()` 但未阻止非管理员访问，API 调用可能失败。

## 三、性能问题

### 3.1 严重问题

#### 🔴 PERF-001: 活动列表缺少分页加载
**文件**: `miniprogram/pages/activity-list/activity-list.js:20-31`
```javascript
const result = await api.getActivities({});
this.setData({ activities: result.items || [] });
```
**问题**: 一次性加载所有活动，数据量大时会导致：
- 页面加载缓慢
- 内存占用过高
- 列表渲染卡顿

**修复方案**:
1. 实现分页加载（skip/limit）
2. 添加下拉刷新和上拉加载更多
3. 虚拟列表优化长列表渲染

---

#### 🔴 PERF-002: 首页活动列表全量加载
**文件**: `miniprogram/utils/api.js:52-57`
```javascript
function getEnrollableActivities() {
  return getActivities({}).then((data) => {
    const items = (data.items || []).filter(...);
    return { items, total: items.length };
  });
}
```
**问题**: 先加载全部数据再前端过滤，浪费带宽。

**修复方案**: 后端提供专门的接口，或使用 status 参数过滤。

#### 🟡 PERF-003: 活动详情页 URL 传递完整活动对象

## 四、代码质量问题

### 4.1 中等问题

#### 🟡 CODE-001: 重复的错误处理代码
**问题**: 每个页面都有相似的错误处理逻辑：
```javascript
.catch((err) => {
  const msg = err && err.message ? err.message : String(err);
  this.setData({ error: msg });
});
```

**建议**: 封装统一的错误处理函数：
```javascript
// utils/error.js
function handleError(err, page) {
  const msg = err?.message || String(err);
  page.setData({ error: msg, submitting: false });
  wx.showToast({ title: msg, icon: 'none' });
}
```

---

#### 🟡 CODE-002: 时间格式化函数重复
**问题**: 多个页面有相似的时间格式化函数：
- `pages/index/index.js:93-101`
- `pages/activity-detail/activity-detail.js:79-88`
- `pages/edit-activity/edit-activity.js:62-73`

**建议**: 提取到 `utils/format.js` 统一管理。

---

#### 🟡 CODE-003: ApiError 类定义不规范
**文件**: `miniprogram/utils/api.js:216-223`
```javascript
function ApiError(code, message) {
  this.statusCode = code;
  // ...
}
ApiError.prototype.toString = function () { ... };
```

**建议**: 使用 ES6 class 语法：
```javascript
class ApiError extends Error {
  constructor(statusCode, message) {
    super(message);
    this.statusCode = statusCode;
    this.name = 'ApiError';
  }
}
```

---

#### 🟡 CODE-004: 缺少代码注释和文档
**问题**: 部分函数缺少注释，业务逻辑不清晰。

**建议**: 添加 JSDoc 注释，说明参数和返回值。

---

#### 🟡 CODE-005: 魔法数字未定义常量
**文件**: 多处使用数字常量：
```javascript
// 状态码
activity.status === 1  // 1-未开始，2-进行中，3-已结束

// 延迟时间
setTimeout(() => ..., 1500);
```

**建议**: 定义常量：
```javascript
const ACTIVITY_STATUS = {
  NOT_STARTED: 1,
  ONGOING: 2,
  ENDED: 3
};

const TOAST_DURATION = 1500;
```
