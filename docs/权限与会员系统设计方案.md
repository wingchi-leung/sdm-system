# 权限与会员系统设计方案

> 版本：v1.2
> 日期：2026-03-23
> 状态：设计中

---

## 一、需求概述

### 1.1 现有系统

**管理员系统（已实现，不做修改）：**
- `admin_user` 表：管理员信息
- `is_super_admin`：超级管理员标记
- `admin_activity_type_role`：管理员按活动类型授权

**用户系统（已实现）：**
- `user` 表：普通用户信息
- `wx_openid`：微信 openid
- 无角色、无会员概念

### 1.2 新增需求

**会员系统（需新增）：**
- 普通用户注册后默认为"普通会员"
- 会员类型由**租户管理员授权**，非付费
- 不同会员类型可访问不同类型的活动
- 支持设置会员有效期

## 二、会员系统设计

### 2.1 会员类型定义

| 会员代码 | 会员名称 | 说明 | 可访问活动类型 |
|----------|----------|------|----------------|
| `normal` | 普通会员 | 新用户默认 | 公开活动 |
| `member_a` | A类会员 | 由租户管理员授权 | 公开 + 类型A |
| `member_b` | B类会员 | 由租户管理员授权 | 公开 + 类型B |
| `vip` | VIP会员 | 由租户管理员授权 | 全部活动 |

**注意：会员类型非付费，由租户管理员手动授权**

### 2.2 新用户默认配置

```
用户注册流程：
1. 用户通过微信授权登录
2. 系统自动创建用户记录
3. 默认会员类型：normal（普通会员）
4. 可立即访问"公开活动"类型的活动
```

### 2.3 会员授权流程

```
租户管理员操作：
1. 进入用户管理页面
2. 选择用户
3. 设置会员类型
4. 可选：设置有效期
5. 保存后用户立即获得对应的访问权限
```

## 五、权限检查实现

### 5.1 会员访问权限检查

```python
def can_access_activity_type(user: User, activity_type_id: int) -> bool:
    """检查用户是否可访问某类型活动"""
    # 获取用户有效的会员类型
    member_type = get_effective_member_type(user)
    
    # 获取该会员类型可访问的活动类型
    allowed_ids = get_allowed_activity_type_ids(member_type.id)
    
    return activity_type_id in allowed_ids

def get_effective_member_type(user: User) -> MemberType:
    """获取用户有效的会员类型（处理过期）"""
    if user.member_expire_at and user.member_expire_at < datetime.now():
        # 已过期，返回普通会员
        return get_default_member_type()
    return user.member_type or get_default_member_type()

def get_activities_for_user(user: User, filters=None) -> List[Activity]:
    """获取用户可见的活动列表"""
    # 获取可访问的活动类型
    member_type = get_effective_member_type(user)
    allowed_type_ids = get_allowed_activity_type_ids(member_type.id)
    
    # 查询活动
    query = db.query(Activity).filter(
        Activity.activity_type_id.in_(allowed_type_ids),
        Activity.tenant_id == user.tenant_id,
        Activity.status != 'deleted'
    )
    
    return query.all()
```

## 八、注意事项

### 8.1 不改动现有管理员系统

```
现有系统保持不变：
- admin_user 表
- admin_activity_type_role 表
- 管理员权限检查逻辑（deps.py）
```

### 8.2 会员系统独立于管理员系统

```
会员系统只影响：
- 普通用户能看到哪些活动

会员系统不影响：
- 管理员的操作权限
- 活动的管理功能
```

### 8.3 租户隔离

- 会员类型按租户隔离
- 用户会员设置按租户隔离
- 活动类型按租户隔离

## 十二、注意事项

### 12.1 安全考虑

1. **前端权限检查仅作 UI 控制**，所有权限验证必须后端执行
2. **会员过期检查**：每次请求都要验证会员是否有效
3. **租户隔离**：确保用户只能访问自己租户的数据

### 12.2 性能考虑

1. **权限缓存**：用户权限可缓存 5 分钟，减少数据库查询
2. **会员类型缓存**：会员类型配置变化频率低，可缓存

### 12.3 扩展性考虑

1. **数据权限预留**：数据库设计预留 scope 字段，后续可扩展
2. **自定义会员类型**：租户可创建自己的会员类型
3. **会员升级支付**：预留支付流程接入点
