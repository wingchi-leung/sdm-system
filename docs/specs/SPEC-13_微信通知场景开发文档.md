# 微信通知场景开发文档

> 适用范围：报名审核、报名确认、Zoom 会议号通知
> 目标：把“长期人员立即通过”和“新伙伴首次报名”两类微信通知，拆成可落地、可审核、可维护的订阅消息方案
> 现状说明：当前项目已具备微信订阅消息底座、模板配置、任务入队、失败重试能力；本文件聚焦于新业务场景设计

---

## 1. 需求背景

当前业务希望通过微信订阅消息覆盖两类用户：

1. 长期人员：审核通过后，立即发送一条消息，内容包含 Zoom 会议号。
2. 新伙伴：首次报名后，先发送“已收到报名”的确认消息；随后再根据审核结果发送第二条消息：
   - 审核通过：发送 Zoom 会议号
   - 审核不通过：发送报名不成功通知

这个需求的核心难点不在“能不能发”，而在“微信订阅消息的模板、授权、触发点”三者如何拆分。

---

## 2. 微信机制说明

### 2.1 不是逐条人工审核

微信订阅消息的关键限制是：

1. 模板需要先在微信公众平台申请并通过审核。
2. 用户需要先对对应模板完成订阅授权。
3. 真实发送时，微信只检查模板、授权、字段格式和接口状态，不是每条消息再走一次人工审核。

### 2.2 自定义能力边界

可以自定义的部分：

1. 模板字段里的具体文案值。
2. 跳转页面。
3. 场景配置中的标题、说明、启用状态。

不能随意自定义的部分：

1. 模板字段类型和字段数量，必须符合微信模板定义。
2. 模板本身不能脱离微信审核直接使用。
3. 用户没有对某个模板授权时，不能强行发送。

### 2.3 两条消息能不能发

可以，但建议这样理解：

1. 两条消息最好拆成两个模板场景。
2. 如果要给同一用户发两条，就需要用户对这两个模板都完成订阅授权。
3. 如果某个场景只会触发一次，那就把它设计成“一次性通知模板”。

---

## 3. 本次业务建议方案

### 3.1 推荐拆分

建议至少拆成 2 个模板，最好是 3 个模板：

#### 方案 A：最稳妥，3 个模板

1. `registration_received`
   - 新伙伴首次报名确认
   - 文案：已收到报名，正在审核中

2. `review_approved_zoom`
   - 审核通过通知
   - 文案：审核通过 + Zoom 会议号

3. `review_rejected`
   - 审核不通过通知
   - 文案：报名未通过 + 友好原因

#### 方案 B：模板更少，2 个模板

1. `registration_received`
   - 新伙伴首次报名确认

2. `review_result`
   - 审核结果通知
   - 审核通过时填 Zoom 会议号
   - 审核不通过时填失败原因

> 推荐优先选择方案 A。原因是“确认报名”和“审核结果”本来就是不同时间点、不同语义的消息，拆开后更清晰，也更利于微信模板审核。

---

## 4. 场景流程设计

### 4.1 长期人员流程

1. 管理员审核通过。
2. 后端写入审核通过状态。
3. 后端发送 `review_approved_zoom`。
4. 用户收到包含 Zoom 会议号的通知。

特点：

1. 只发一条。
2. 内容直接、目标明确。
3. 不需要先发“报名已收到”。

### 4.2 新伙伴流程

1. 用户首次报名提交成功。
2. 后端发送 `registration_received`。
3. 管理员后续审核。
4. 审核通过时发送 Zoom 通知；审核不通过时发送失败通知。

特点：

1. 第一条是过程通知，第二条是结果通知。
2. 用户体验更稳定，不会一直等待没有反馈。
3. 结果消息一定要和审核状态绑定，避免重复或错发。

---

## 5. 推荐模板字段

### 5.1 `registration_received`

建议字段：

1. 报名名称
2. 报名时间
3. 审核状态
4. 提示文案

示例文案：

- `已收到你的报名，我们会尽快完成审核。`

### 5.2 `review_approved_zoom`

建议字段：

1. 活动名称
2. 审核结果
3. Zoom 会议号
4. 加入时间
5. 跳转提示

示例文案：

- `审核通过，请使用 Zoom 会议号加入会议。`

### 5.3 `review_rejected`

建议字段：

1. 活动名称
2. 审核结果
3. 不通过原因
4. 联系方式或后续提示

示例文案：

- `很抱歉，本次报名未通过。`

## 6. 后端实现建议

### 6.1 场景定义

建议在通知中心新增以下场景常量：

1. `registration_received`
2. `review_approved_zoom`
3. `review_rejected`

如果采用 2 模板方案，则可只保留：

1. `registration_received`
2. `review_result`

### 6.2 触发点

建议把触发点固定在这三个业务节点：

1. 报名创建成功时
2. 审核通过时
3. 审核拒绝时

当前项目里已经有报名成功通知和退款通知的任务入队机制，可以复用同样的入队、幂等、重试逻辑。

### 6.3 需要改的地方

1. 在 `notification_center` 增加新场景默认配置。
2. 在 `notifications/config` 中暴露新场景模板 ID。
3. 在报名创建、审核通过、审核拒绝的业务分支中入队消息任务。
4. 给每个场景补测试用例，确保：
   - 不重复入队
   - 模板未配置时不发
   - 用户未授权时不发
   - 审核结果变更后只触发一次

### 6.4 现有代码对照

当前代码里已经有这些可复用能力：

1. 订阅消息发送服务：`backend/app/services/wechat_subscribe.py`
2. 场景配置和模板渲染：`backend/app/services/notification_center.py`
3. 通知配置接口：`backend/app/api/v1/endpoints/notifications.py`
4. 报名成功授权弹窗：`miniprogram/pages/register/register.js`

## 7. 小程序实现建议

### 7.1 授权时机

建议在“报名提交”这个动作上，一次性请求需要的模板授权。

例如：

1. 新伙伴报名时，请求 `registration_received` 和 `review_result`。
2. 如果长期人员只走审核通过通知，则请求 `review_approved_zoom`。

### 7.2 前端原则

1. 不要在页面加载时强制弹授权。
2. 不要为了多发消息在多个页面重复弹。
3. 在用户有明确动作时发起授权，微信审核和用户接受率都会更稳。

### 7.3 用户拒绝时的兜底

如果用户拒绝订阅：

1. 前端给出轻提示，不阻塞报名。
2. 后端记录授权结果，方便排查。
3. 后续消息不再尝试强发。

---

## 8. 审核风险点

以下内容容易导致模板审核被打回：

1. 文案过于营销化。
2. 模板场景和实际业务不一致。
3. 字段内容过长，像聊天记录，不像通知。
4. 模板中塞入太多无关信息。
5. 失败通知里直接输出过于生硬或敏感的拒绝原因。

建议做法：

1. 把 Zoom 会议号放到独立字段。
2. 把“已收到报名”和“审核结果”拆开。
3. 失败原因尽量做成友好文案，不要直接暴露内部审核逻辑。

---

## 9. 打回原因和展示

### 9.1 模板审核被打回

一般会在微信公众平台后台看到审核意见或驳回原因。

### 9.2 发送失败

发送时如果失败，接口会返回微信错误码和错误信息，后端应记录：

1. `errcode`
2. `errmsg`
3. 场景
4. 模板 ID
5. 用户 ID / openid

### 9.3 用户侧展示

通常不建议把微信后台审核细节直接展示给最终用户。
用户侧只需要看到业务提示，例如：

1. 已收到报名
2. 审核通过
3. 审核未通过

---

## 10. 推荐实施顺序

1. 先确定采用 2 模板还是 3 模板方案。
2. 去微信公众平台准备模板并提交审核。
3. 在后端补充新场景配置和入队逻辑。
4. 在小程序报名页补充授权请求。
5. 在审核通过 / 审核拒绝节点补充发送。
6. 补测试与联调。

---

## 11. 验收标准

1. 长期人员审核通过后能收到一条包含 Zoom 会议号的通知。
2. 新伙伴报名后能先收到“已收到报名”。
3. 新伙伴审核通过后能收到 Zoom 通知。
4. 新伙伴审核不通过后能收到失败通知。
5. 用户未订阅时不会误发消息。
6. 同一业务事件不会重复推送。

---

## 12. 与现有系统的关系

当前项目已有的订阅消息底座可以直接复用：

1. 任务去重
2. 失败重试
3. 场景配置
4. 模板渲染
5. 小程序授权上报

所以这次工作重点不是重造通知系统，而是把“审核通过”和“首次报名确认”两个业务场景补进现有底座。
