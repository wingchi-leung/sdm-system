# 社区频道日历 规格说明书

> 本文档用于定义「社区频道日历」功能的产品范围、交互形态、数据边界与实施约束。  
> 依赖背景：社区频道、频道公告、频道动态已存在；本功能定位为社区内的独立时间组织能力，不替代活动模块。

---

## 一、背景与定位

### 1.1 用户需求

用户希望在不同社区里看到不同的日历和活动安排，例如：

- 这个社区下周有哪些活动
- 某一天是否有讲座、聚会或报名截止
- 管理员是否在本社区发布了时间表、提醒或周期安排

### 1.2 产品定位

社区频道日历是一个 **频道内独立的时间管理模块**，用于承载该社区自己的时间信息。

它的核心价值不是替代现有活动模块，而是把社区里“什么时候发生什么”聚合到一起，让成员可以更直观地掌握安排。

### 1.3 设计原则

- **频道独立**：每个社区有自己的日历，数据按频道隔离。
- **轻量可用**：先做查看和基础管理，不做复杂排班系统。
- **活动可关联**：允许把现有活动挂到日历事件上，但不强依赖活动。
- **权限清晰**：默认只有频道成员可见，管理员可维护。
- **静默失败**：所有异步操作必须有 try-catch 和用户友好提示。

## 二、目标与非目标

### 2.1 本期目标

1. 用户进入某个社区后，可以看到该社区自己的日历。
2. 用户可以按月查看日历，按日期查看当天事件。
3. 管理员可以创建、编辑、删除日历事件。
4. 日历事件可以关联现有活动，作为活动前后时间安排的承载位。
5. 社区内的时间安排可以形成清晰的列表与日历双视图。

### 2.2 本期不做

- 多人协同编辑
- 复杂重复规则
- 自动冲突检测
- 跨社区共享日历
- 日历订阅到系统外部日历
- 复杂提醒中心
- 类似 Google Calendar 的完整排班能力
- 用户自定义拖拽排班

---

## 三、产品形态

### 3.1 信息架构

建议把日历放在社区内部页面中，和动态、公告并列为社区的核心内容能力之一。

```
社区首页
└─ 进入某个社区
   ├─ 动态
   ├─ 公告
   └─ 日历
        ├─ 月视图
        ├─ 日期事件列表
        └─ 事件详情页
```

### 3.2 页面入口建议

- 社区首页增加「日历」入口
- 进入频道后，在顶部或功能区展示「日历」标签/按钮
- 日历页内提供「月视图 / 列表视图」切换

### 3.3 用户视角

| 场景 | 用户动作 | 结果 |
|------|----------|------|
| 查看本社区安排 | 进入社区后点击「日历」 | 看到月视图和近期事件 |
| 查看某天事件 | 点击日历中的某一天 | 看到当天事件列表 |
| 查看事件详情 | 点击事件卡片 | 进入事件详情页 |
| 管理员新增事件 | 点击「新建事件」 | 打开创建页并提交 |
| 管理员编辑事件 | 在事件详情页进入编辑 | 修改时间、标题、说明等 |

## 四、角色与权限

### 4.1 角色定义

| 角色 | 权限 |
|------|------|
| 频道管理员 | 创建、编辑、删除日历事件；可将事件关联活动；可置顶重要事件 |
| 频道成员 | 查看日历、查看事件详情；如后续开放 RSVP，可确认参加 |
| 非成员 | 不可查看该社区日历内容 |

### 4.2 基础规则

- 日历事件必须归属一个频道。
- 日历内容默认仅该频道成员可见。
- 事件可以关联一个活动，但事件本身不是活动。
- 若事件关联了活动，跳转活动详情前仍需走原有活动权限校验。

---

## 五、核心业务规则

### 5.1 事件类型

建议首期支持以下类型：

- 活动
- 会议
- 提醒
- 截止时间
- 周期安排

### 5.2 事件字段

建议一个日历事件至少包含：

- 标题
- 开始时间
- 结束时间
- 所属频道
- 发布人
- 事件类型
- 说明
- 是否关联活动
- 地点
- 封面图

### 5.3 展示规则

- 月视图展示事件分布。
- 当天有多个事件时，按开始时间排序。
- 未来 7 天内的事件可在列表区优先展示。
- 已过期事件默认保留在历史列表中，不自动删除。

### 5.4 删除规则

- 管理员可删除自己频道内的事件。
- 已删除事件不再在日历中展示。
- 若事件关联活动，删除事件不影响活动本身。

## 六、页面设计

### 6.1 页面清单

#### 6.1.1 community-calendar

社区日历主页，承担以下能力：

- 月历展示
- 近期事件列表
- 日期筛选
- 新建事件入口

#### 6.1.2 community-calendar-create

新建日历事件页，包含：

- 标题
- 开始/结束时间
- 事件类型
- 说明
- 地点
- 封面图
- 关联活动

#### 6.1.3 community-calendar-detail

事件详情页，包含：

- 标题与时间
- 所属社区
- 事件说明
- 关联活动入口
- 管理员编辑/删除入口

#### 6.1.4 community-calendar-edit

事件编辑页，可复用创建页表单结构。

### 6.2 页面交互建议

- 月视图中，带事件的日期用高亮点标识。
- 点击某天后，下方显示当天事件列表。
- 点击事件卡片进入详情页。
- 管理员在详情页可进入编辑或删除。

## 七、数据库设计建议

### 7.1 新表：`community_channel_calendar_event`

建议使用独立表，不复用公告或帖子表。

```sql
CREATE TABLE IF NOT EXISTS `community_channel_calendar_event` (
  `id` int NOT NULL AUTO_INCREMENT,
  `tenant_id` int NOT NULL COMMENT '租户ID',
  `channel_id` int NOT NULL COMMENT '所属频道ID',
  `activity_id` int DEFAULT NULL COMMENT '关联活动ID',
  `author_user_id` int NOT NULL COMMENT '发布人用户ID',
  `title` varchar(120) NOT NULL COMMENT '事件标题',
  `event_type` varchar(32) NOT NULL DEFAULT 'activity' COMMENT '事件类型：activity/meeting/reminder/deadline/routine',
  `content` text DEFAULT NULL COMMENT '事件说明',
  `location` varchar(200) DEFAULT NULL COMMENT '地点',
  `cover_url` varchar(500) DEFAULT NULL COMMENT '封面图',
  `start_time` datetime NOT NULL COMMENT '开始时间',
  `end_time` datetime DEFAULT NULL COMMENT '结束时间',
  `status` tinyint NOT NULL DEFAULT 1 COMMENT '状态：1-正常 0-已删除',
  `create_time` datetime DEFAULT CURRENT_TIMESTAMP,
  `update_time` datetime DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  KEY `idx_calendar_channel_id` (`channel_id`),
  KEY `idx_calendar_tenant_id` (`tenant_id`),
  KEY `idx_calendar_start_time` (`start_time`),
  KEY `idx_calendar_activity_id` (`activity_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='社区频道日历事件表';
```

### 7.2 设计决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 独立表 vs 复用公告/帖子表 | 独立表 | 日历有时间轴语义，和帖子、公告的内容模型不同 |
| 是否强绑定活动 | 不强绑定 | 日历要能承载纯社区安排，也要能挂活动 |
| 是否支持软删除 | 首期不支持 | 先保持简单，删除即不展示 |
| 是否支持重复规则 | 不支持 | 控制复杂度，后续可扩展 |
| 是否支持 RSVP | 预留，不作为本期核心 | 避免把日历做成完整报名系统 |

## 八、后端 API 设计

### 8.1 API 列表

| 接口 | 方法 | 说明 | 权限 |
|------|------|------|------|
| `/api/v1/community/channels/{id}/calendar/events` | GET | 获取频道日历事件列表 | 频道成员 |
| `/api/v1/community/channels/{id}/calendar/events` | POST | 创建日历事件 | 频道管理员 |
| `/api/v1/community/channels/{id}/calendar/events/{event_id}` | GET | 获取事件详情 | 频道成员 |
| `/api/v1/community/channels/{id}/calendar/events/{event_id}` | PUT | 编辑事件 | 频道管理员 |
| `/api/v1/community/channels/{id}/calendar/events/{event_id}` | DELETE | 删除事件 | 频道管理员 |
| `/api/v1/community/channels/{id}/calendar/month-summary` | GET | 获取某月事件聚合数据 | 频道成员 |

### 8.2 请求建议

#### 创建事件

```json
{
  "title": "周六线下分享会",
  "event_type": "activity",
  "content": "提前 10 分钟到场签到",
  "location": "北京朝阳区",
  "cover_url": "/uploads/...",
  "start_time": "2026-06-21T10:00:00+08:00",
  "end_time": "2026-06-21T12:00:00+08:00",
  "activity_id": 123
}
```

### 8.3 权限与安全

- 非成员访问列表和详情接口时应拒绝。
- 管理员身份以频道成员表中的角色为准，不要混用全局 RBAC。
- 删除事件时必须校验频道归属，避免跨频道误删。
- 若事件关联活动，活动 ID 也必须属于当前租户。

## 九、小程序实现建议

### 9.1 入口建议

建议在以下位置提供入口：

- 社区首页卡片区
- 频道详情页顶部功能区
- 动态页右上角的「更多」区域内

### 9.2 交互建议

- 月视图以白底日历格为主，事件用小圆点或短条标记。
- 近期事件列表以卡片形式展示，突出时间、标题和地点。
- 事件详情页保留返回路径，不打断用户浏览社区。

### 9.3 空状态文案

建议文案：

- `这个社区还没有日历事件`
- `管理员创建后，成员就能在这里看到安排`

## 十一、验收标准

满足以下条件即可视为第一版完成：

1. 进入某个社区后可以看到该社区日历。
2. 同一个用户在不同社区看到的日历内容不同。
3. 用户可以查看某一天有哪些事件。
4. 管理员可以创建事件并立即在日历中看到。
5. 管理员可以编辑和删除事件。
6. 事件可关联现有活动。
7. 非成员不能查看该社区日历。
